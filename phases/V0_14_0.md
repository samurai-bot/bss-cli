# v0.14.0 — Provider-Adapter Foundation + Email (Resend)

> **The release that establishes the per-domain adapter doctrine, ships the shared webhook substrate, and proves the whole pattern end-to-end on the lowest-stakes integration: email.** v0.14 is deliberately the smallest of three provider-integration releases (v0.14 email → v0.15 KYC + eSIM seam → v0.16 payment). It introduces no new containers, lays the `integrations` schema, ships `packages/bss-webhooks/`, and swaps the `LoggingEmailAdapter` story for a real Resend integration with webhook-driven delivery/bounce/complaint logging. The `bss onboard` Typer wizard ships in skeleton form with `--domain email` only; KYC and Payment domains are wired in their own releases. By the end of v0.14, the operator can clone the repo, run `bss onboard`, paste a Resend API key, and watch a real magic-link login email arrive in their inbox while a webhook fires and lands in `integrations.webhook_event`.

The architecture verdict, locked in by v0.14 and inherited by v0.15/v0.16: **per-domain adapter Protocols, no integration-broker container, no `bss-providers` mega-package.** The four domains have fundamentally different shapes (sync charge vs fire-and-forget email vs async eSIM order vs signed-attestation receipt) and the existing `EmailAdapter` Protocol in `packages/bss-portal-auth/bss_portal_auth/email.py:32-39` is the gold-standard pattern to replicate. The one shared package — `packages/bss-webhooks/` — owns only the genuinely cross-cutting concerns (signature verification, idempotency keys, redaction, persistence). It is *not* a unified API for the adapters themselves.

## Goal

Three deliverables, scoped to be reviewable in a single ~4–5 day window:

1. **`integrations` schema + `bss-webhooks` package + audit doctrine.** Foundation that v0.15 and v0.16 build on. Two tables (`external_call`, `webhook_event`), one ~150-LoC package, an `external_ref` payload-envelope convention on `audit.domain_event` so forensic joins work without a schema change to the existing event log.
2. **Email — Resend.** Extend the existing `EmailAdapter` Protocol with `ResendEmailAdapter` (3k/mo free tier; webhooks for delivery/bounce/complaint). Rename `BSS_PORTAL_EMAIL_ADAPTER` → `BSS_PORTAL_EMAIL_PROVIDER` for naming consistency across the three domains; old name keeps working with a deprecation warning until v0.16.
3. **`bss onboard` wizard, email-scoped.** Skeleton of the Typer command at `cli/bss_cli/commands/onboard.py` with email as the only configurable domain in v0.14. v0.15 extends it with `--domain kyc`; v0.16 extends it with `--domain payment`. Re-runnable; writes `.env` atomically; probes Resend with a real test send before saving.

When v0.14 ships, the architecture pattern is proven on the lowest-stakes integration. v0.15 and v0.16 then become "do the same thing again, with these new adapters" rather than "design the architecture from scratch under payment-scope pressure."

## Deliverables

### Track 1 — `integrations` schema + `bss-webhooks` package + audit doctrine

#### 1.1 New schema `integrations` (Alembic `0015_v0140_integrations_schema.py`)

```sql
CREATE SCHEMA integrations;

CREATE TABLE integrations.external_call (
    id              bigserial PRIMARY KEY,
    provider        text NOT NULL,                 -- 'resend' | (v0.15+: 'didit') | (v0.16+: 'stripe')
    operation       text NOT NULL,                 -- 'send_login' | 'send_step_up' | …
    aggregate_type  text,                          -- 'identity' | 'session' | NULL
    aggregate_id    text,                          -- IDT-* | SES-* | NULL for fire-and-forget
    success         boolean NOT NULL,
    latency_ms      integer NOT NULL,
    provider_call_id text,                         -- Resend msg id, Stripe pi_*, etc.
    error_code      text,
    error_message   text,
    redacted_payload jsonb,                        -- request/response envelope, redacted
    occurred_at     timestamptz NOT NULL DEFAULT now(),
    tenant_id       text NOT NULL DEFAULT 'DEFAULT'
);
CREATE INDEX ix_ec_aggregate     ON integrations.external_call(aggregate_type, aggregate_id);
CREATE INDEX ix_ec_provider_time ON integrations.external_call(provider, occurred_at);

CREATE TABLE integrations.webhook_event (
    provider         text NOT NULL,
    event_id         text NOT NULL,
    event_type       text NOT NULL,                -- 'email.delivered'
    body             jsonb NOT NULL,
    signature_valid  boolean NOT NULL,
    received_at      timestamptz NOT NULL DEFAULT now(),
    processed_at     timestamptz,
    process_outcome  text,                         -- 'reconciled' | 'noop' | 'errored'
    process_error    text,
    PRIMARY KEY (provider, event_id)
);
CREATE INDEX ix_we_received_unprocessed
    ON integrations.webhook_event(received_at) WHERE processed_at IS NULL;
```

Two tables, deliberately. **No `provider_config` table** — config is `.env` only. DB-backed multi-tenant config is post-v0.16.

The schema is built at v0.14 size (only `resend` flows through it initially) but the `provider` column is open enum so v0.15 adds `'didit'` and v0.16 adds `'stripe'` without migration.

#### 1.2 `packages/bss-webhooks/` (new ~150 LoC package)

```
packages/bss-webhooks/
├── pyproject.toml
└── bss_webhooks/
    ├── __init__.py
    ├── signatures.py    # verify_signature(secret, body, header, scheme, max_skew_seconds)
    ├── store.py         # WebhookEventStore + ExternalCallStore
    ├── idempotency.py   # idempotency_key(aggregate_id, retry_count)
    └── redaction.py     # redact_provider_payload(provider, body) -> dict
```

`verify_signature()` is built with three schemes from day one even though only one — `"svix"` (Resend's format) — has a real consumer in v0.14:

- `"svix"` — Resend (used in v0.14)
- `"stripe"` — `Stripe-Signature` HMAC-SHA-256 with timestamp tolerance (used in v0.16)
- `"didit_hmac"` — Didit's `X-Signature-V2` (used in v0.15)

Constant-time compare via `hmac.compare_digest`, replay-window check (default 300s). Building all three schemes upfront is cheap (~30 LoC each) and prevents a "signature verification refactor" mid-v0.16 when payment is the wrong time to be touching shared crypto code. Tests cover all three schemes against tampered-body and replay-window fixtures even though only one has a live consumer.

`WebhookEventStore.persist(provider, event_id, body, signature_valid)` is idempotent on `(provider, event_id)` PK.

`redaction.py` filters per-provider. v0.14 ships only `redact_resend(body)` (strips email PII from logged response bodies); `redact_stripe` and `redact_didit` are added in their respective releases.

#### 1.3 Audit doctrine — `external_ref` envelope on `audit.domain_event.payload`

No schema change to `audit.domain_event`. Existing payloads gain an optional `external_ref` envelope:

```json
{
  "kind": "magic_link",
  "external_ref": {
    "provider": "resend",
    "operation": "send_login",
    "id": "msg_abc123",
    "idempotency_key": "IDT-0042-r1"
  }
}
```

Forensic join: `audit.domain_event.payload->'external_ref'->>'id'` ↔ `integrations.external_call.provider_call_id`.

The doctrine entry lands in v0.14 even though Resend is the only consumer, so v0.15/v0.16 inherit it without re-litigation.

### Track 2 — Email (Resend)

Lowest-risk consumer of the Track 1 substrate. Email failures are recoverable (re-OTP); shake out the adapter pattern + webhook + onboarding shape end-to-end here.

#### 2.1 Extend existing `EmailAdapter` Protocol

`packages/bss-portal-auth/bss_portal_auth/email.py` — add `ResendEmailAdapter` mirroring `LoggingEmailAdapter`'s shape (lines 42–118). Resend's Python SDK is sync; preserve the existing sync call pattern at the three call sites (`send_login`, `send_step_up`, `send_email_change_verification`). Each call records to `integrations.external_call` via `bss_webhooks.ExternalCallStore`, including latency, success, and Resend's returned `msg_*` id.

New env vars:

```bash
BSS_PORTAL_EMAIL_PROVIDER=resend          # was BSS_PORTAL_EMAIL_ADAPTER (renamed; old name deprecated until v0.16)
BSS_PORTAL_EMAIL_RESEND_API_KEY=re_...
BSS_PORTAL_EMAIL_RESEND_WEBHOOK_SECRET=whsec_...
BSS_PORTAL_EMAIL_FROM="BSS-CLI <noreply@example.com>"
```

`select_adapter()` at `email.py:177-194` extends to handle `"resend"`; raises on missing key or sender. Old name `BSS_PORTAL_EMAIL_ADAPTER` continues to work with a `warnings.warn` until v0.16. Naming convention `BSS_<DOMAIN>_PROVIDER` + `BSS_<DOMAIN>_<NAME>_<KEY>` is established here; v0.15 and v0.16 follow it.

#### 2.2 Resend webhook receiver

New route `POST /webhooks/resend` in `portals/self-serve/bss_self_serve/routes/webhooks.py`. Verifies Svix-style signature via `bss_webhooks.signatures.verify_signature(scheme="svix", …)`. Persists into `integrations.webhook_event`. For `email.delivered` / `email.bounced` / `email.complained`, emits a domain event into `audit.domain_event` (`portal_auth.email.delivered`, `…bounced`, `…complained`).

v0.14 records bounces but doesn't act on them (suppression list is post-v0.16).

Webhook routes are exempt from `BSSApiTokenMiddleware` (like `/health`). Auth is provider signature only. Add `/webhooks/*` to exempt-paths in `packages/bss-middleware`.

#### 2.3 PR exit: Resend live in staging

Smoke test (single manual test loop): blank `.env` → `bss onboard --domain email` → restart portal → trigger a magic-link login → email arrives in real inbox → webhook fires → `integrations.webhook_event` row + `audit.domain_event` row both visible via `bss external-calls --provider resend`.

## Doctrine and constraints

1. **Per-domain adapter Protocols.** No `bss-providers` mega-package. No integration-broker container. Each domain's adapter lives where its consumer lives.
2. **`BSS_<DOMAIN>_PROVIDER` env naming.** Then secrets under `BSS_<DOMAIN>_<NAME>_<KEY>`. Adding a future provider doesn't rename existing vars.
3. **Fail-fast at startup** on unknown provider, missing key, or env-mode mismatch. Never silently downgrade.
4. **Webhooks land on the receiving domain's service.** No webhook-broker container. Each webhook authenticates via provider signature, exempt from `BSSApiTokenMiddleware`.
5. **`integrations.external_call` + `integrations.webhook_event`** are the forensic substrate. `audit.domain_event` keeps its event-log role; no schema change there.
6. **Secrets in `.env`, never `settings.toml`** (preserves v0.13 doctrine).
7. **`bss onboard` writes `.env` atomically; rerunnable per-domain.** First-run UX without becoming a runtime config service.
8. **DB-backed multi-tenant `tenant_config` is post-v0.16.** v0.14–v0.16 single-tenant `.env` config is the deliberate scope.
9. **Build the substrate at full strength on day one.** Three signature schemes, the `external_ref` envelope doctrine, the `bss onboard` skeleton — all ship in v0.14 even though only Resend consumes them. Avoids a mid-v0.16 refactor of shared crypto under payment-scope pressure.

### Anti-patterns added to CLAUDE.md

- **(v0.14+) Don't add a unified `Provider.execute()` API.** The four domains have genuinely different shapes; a unified API forces lowest-common-denominator and erases information consumers need.
- **(v0.14+) Don't put webhook receivers behind `BSSApiTokenMiddleware`.** Webhooks authenticate via provider signature. Exempt path in middleware; greppable: `rg 'WEBHOOK_EXEMPT_PATHS' packages/bss-middleware/` must list `/webhooks/*`.
- **(v0.14+) Don't log raw provider responses.** Use `bss_webhooks.redaction.redact_provider_payload()`. Resend responses contain customer email; future Stripe/Didit responses contain more sensitive payloads. Greppable: `rg 'log\.(info|debug).*\b(stripe|didit|resend)\.(response|body)' --type py` must stay empty.
- **(v0.14+) Don't store provider config in DB.** v0.14–v0.16 are `.env` only. DB-backed `tenant_config` is post-v0.16 and arrives with multi-tenancy.
- **(v0.14+) Don't silently fall back to a mock when prod creds are unset.** Fail-fast at startup with a specific missing-var message. `select_*` functions must raise `RuntimeError`, not return `MockAdapter`.
- **(v0.14+) Don't accept user-controllable provider name on a route.** Provider is resolved from env at startup, attached to `app.state`. A POST body field that selects a provider is an arbitrary-tool primitive.

## Test strategy

- **Unit**: `EmailAdapter` Protocol + `ResendEmailAdapter` (with a faked `resend.Emails.send`); `select_adapter` for unknown-name, missing-key.
- **Integration**: `bss-webhooks` signature verification across all three schemes (`svix`, `stripe`, `didit_hmac`) with tampered-body and replay-window fixtures. All three tested even though v0.14 only consumes svix — the v0.15/v0.16 path is paved upfront.
- **Resend sandbox test** (gated on env, skipped in CI by default; runnable nightly on a developer machine): `BSS_PORTAL_EMAIL_PROVIDER=resend` + sandbox key → magic-link login → email delivered → webhook fires.
- **Hero scenarios**: existing v0.13 heroes pass three runs with `BSS_PORTAL_EMAIL_PROVIDER=logging` (mock mode). Production-mode dual-run is exercised in v0.16's hero suite once all three providers exist.
- **Doctrine guards**: greppable rules in `make doctrine-check`:
  - No `Provider.execute(` — should not exist.
  - `WEBHOOK_EXEMPT_PATHS` listed in `bss-middleware`.
  - No raw provider response logging.
  - No `os.environ.get("BSS_*_PROVIDER")` outside `select_*` files.

**Manual testing budget for v0.14: one round.** Resend mailbox smoke test only. KYC and Payment add their own rounds in v0.15 and v0.16.

## The trap

**The trap of "just enough":** the temptation to ship a Resend-only `bss-webhooks` (single signature scheme, no `external_ref` doctrine, no `bss external-calls` verb) because "we'll add the rest in v0.15". Don't. Building the substrate at full strength on day one is the whole point of splitting the release. v0.16 must not be the first time anyone touches HMAC code under payment-scope pressure. The 30 lines of `verify_signature(scheme="stripe")` + the 30 lines of `verify_signature(scheme="didit_hmac")` + their test fixtures are part of v0.14's done-line, even though their only purpose is to *not exist* as v0.15/v0.16 work.

**The second trap:** treating Resend like the simulator. The `LoggingEmailAdapter` writes to a file and never fails. `ResendEmailAdapter` is a real network call to a real third-party API with a real rate limit and a real key that can leak. Every call records to `integrations.external_call`, every webhook verifies its signature, every error path returns a recognizable error code. If you find yourself thinking "it's just email, this is over-engineering," remember v0.16 will inherit every shortcut you take here.

**The third trap:** letting the `bss onboard` wizard become a maintenance burden by trying to handle every conceivable `.env` edit case. The skeleton is small on purpose: read existing `.env`, prompt only for the email block, write back atomically, exit. The wizard does not parse arbitrary YAML, support comments-on-comments, or handle multi-line values. v0.15 and v0.16 add their domains by appending similar small functions, not by extending a generic config-editing engine.

Don't call this v0.15 — v0.15 is KYC, and KYC has its own portal-side adapter, its own JWS verification, its own free-tier-cap concerns. Keep them separate.
