# v0.2.0 — Observability: OpenTelemetry + `bss trace`

> **The first screenshot release after v0.1.** Everything the stack did in v0.1 was invisible the moment it left the terminal — this version makes it legible to a stranger reading a LinkedIn post. Auto-instrument the 9 services, propagate W3C trace context through HTTP / MQ / SQL, and turn the `bss trace` stub into an ASCII swimlane that shows a customer signup fanning out across every service it touched.

## Goal

v0.1 proved the BSS works end-to-end. v0.2 makes it **provable to someone who isn't sitting in the terminal with you**. Two tightly-coupled deliverables:

1. **OpenTelemetry auto-instrumentation across the 9 services**, with W3C `traceparent` propagation through httpx (sync HTTP calls via `bss-clients`), aio-pika (async events), and asyncpg (via SQLAlchemy), exporting OTLP to a Jaeger backend that can be deployed either as a bundled container or installed once on a BYOI host like `tech-vm`.
2. **`bss trace <trace_id>` populated.** The command is a stub today (`cli/bss_cli/commands/trace.py` prints "not implemented" and exits 2). v0.2 replaces it with a Jaeger API client and an ASCII swimlane renderer that shows every span from the CLI's own root span down to leaf SQL calls, with parent-child indentation, service columns, and wall-clock breakdown.

The swimlane is the release's demo artifact. It's what lands in a slide, a README hero image, a LinkedIn post. Its visual quality is the point of v0.2 — invest in iteration until it looks right, not just functional. The rest of the deliverables (bootstrap package, backend deploy story, hero scenario) are the machinery that makes the swimlane possible.

## Deliverables

### 1. `packages/bss-telemetry/` — shared OTel bootstrap

New workspace package, mirroring `packages/bss-clock/` and `packages/bss-events/` in shape.

```
packages/bss-telemetry/
├── pyproject.toml
├── bss_telemetry/
│   ├── __init__.py
│   ├── bootstrap.py         # configure_telemetry(service_name) — services call once at startup
│   ├── instruments.py       # named tracers for the 3 manual spans
│   ├── propagation.py       # W3C traceparent helpers for aio-pika message headers
│   ├── config.py            # pydantic-settings for BSS_OTEL_* env vars, _REPO_ROOT pattern
│   └── semconv.py           # BSS-CLI-specific attribute keys (bss.customer_id, bss.subscription_id, etc.)
└── tests/
    ├── test_bootstrap.py
    ├── test_propagation.py
    └── test_semconv.py
```

`configure_telemetry(service_name: str)` sets up a `TracerProvider`, an `OTLPSpanExporter` (HTTP/protobuf to `$BSS_OTEL_EXPORTER_OTLP_ENDPOINT`), a `BatchSpanProcessor`, and installs the auto-instrumentors for `FastAPI`, `HTTPX`, `AsyncPG`, and `AioPika`. Services call it once during lifespan startup, passing `service_name="crm"` etc. The function is idempotent (safe to call twice in tests) and tolerates the endpoint being unreachable (drops spans, logs once, does not crash startup — observability must not gate the business path).

New env vars (append to `.env.example`):

```
BSS_OTEL_ENABLED=true
BSS_OTEL_EXPORTER_OTLP_ENDPOINT=http://tech-vm:4318
BSS_OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
BSS_OTEL_SERVICE_NAME_PREFIX=bss
BSS_OTEL_SAMPLING_RATIO=1.0
```

`BSS_OTEL_ENABLED=false` short-circuits `configure_telemetry` to a no-op. Used in unit tests where the SDK's batch processor leaks a background thread across pytest cases.

### 2. Service wiring — 9 identical one-line edits

Each service's lifespan gains one call. For `services/crm/app/main.py`:

```python
from bss_telemetry import configure_telemetry

@asynccontextmanager
async def lifespan(app: FastAPI):
    configure_telemetry(service_name="crm")          # <-- one line
    # ... existing startup
    yield
    # ... existing teardown
```

No other changes in routers, services, policies, or repositories. Auto-instrumentation hooks:

- **FastAPI** — every endpoint becomes a server span with `http.method`, `http.route`, `http.status_code`, `http.request.header.x-bss-actor`, `http.request.header.x-bss-channel`
- **httpx** — every outbound call from `bss-clients` becomes a client span; `traceparent` is injected automatically
- **asyncpg** — every SQL query becomes a client span child of the HTTP span; `db.statement` is captured (truncated to 500 chars)
- **aio-pika** — basic_publish injects `traceparent` into the AMQP message headers; basic_consume extracts it

#### 2a. Manual spans — exactly 3

Auto-instrumentation gives plumbing. It does NOT give business semantics. Three hand-picked manual spans turn the swimlane into a readable story:

| Service | Span name | Code site |
|---|---|---|
| COM | `com.order.decompose` | `services/com/app/services/order.py` — wraps the SOM decomposition call so the "I created 1 CFS + 2 RFS" moment is visible |
| SOM | `som.service.wait_for_provisioning` | `services/som/app/services/service.py` — wraps the "all tasks completed?" check so the provisioning-wait tail is a named span, not a silent gap |
| Subscription | `subscription.purchase_vas` | `services/subscription/app/services/subscription.py` — wraps the VAS charge+apply transaction so the unblock moment is bold on the swimlane |

Each uses `bss_telemetry.instruments.tracer("bss-<svc>")`. Each sets business attributes via the `bss_telemetry.semconv` constants (`BSS_CUSTOMER_ID`, `BSS_SUBSCRIPTION_ID`, `BSS_ORDER_ID` — all prefixed-ID strings, never PII). **No additional manual spans.** The grep guard in §6 enforces this.

### 3. `packages/bss-clients/` — outbound context propagation

`bss-clients` already uses httpx and already has a `ContextVars`-based header propagation layer (`X-BSS-Actor`, `X-BSS-Channel`, `X-Request-ID`). Adding OTel is deps-only:

- `opentelemetry-api`, `opentelemetry-instrumentation-httpx` as dependencies
- No code changes in `bss_clients/` — the auto-instrumentor globally wraps every `httpx.AsyncClient` once `configure_telemetry` has run in the process

### 4. `packages/bss-events/` — aio-pika header propagation

aio-pika auto-instrumentation injects `traceparent` on publish. On consume, the handler needs to activate the extracted span context so child spans attach correctly. Wrap this once in `bss-events`:

```python
# packages/bss-events/bss_events/consumer.py

from bss_telemetry.propagation import use_amqp_span

async def _dispatch(handler, message: AbstractIncomingMessage):
    async with use_amqp_span(message):
        return await handler(message)
```

Consumers (SOM, Rating, Subscription) don't need to know about OTel. They register handlers via the existing `consumer.bind(routing_key, handler)` API; the propagation wrap is internal.

### 5. Trace backend — Jaeger, two deployment paths

**Decision: Jaeger all-in-one.** Not Tempo. Rationale lands in `DECISIONS.md`:

- Single container, no object storage (Tempo requires S3/MinIO/filesystem — adds moving parts)
- Built-in UI at `:16686`, no Grafana stack required
- Native OTLP/HTTP (4318) and OTLP/gRPC (4317) ingress via `COLLECTOR_OTLP_ENABLED=true`
- ~200 MB RAM vs Tempo + MinIO + Grafana at ~600 MB
- `bss trace` reads via Jaeger's stable HTTP JSON API (`/api/traces/<id>`)

#### 5a. BYOI path (primary — matches how v0.1 actually runs)

Jaeger installs once on a host reachable from the services (`tech-vm` in the canonical setup, same machine that already hosts Postgres and RabbitMQ). **Preferred: add to tech-vm's existing `docker-compose.yml`** (the same one managing postgres and rabbitmq) so Jaeger is declaratively managed alongside the other infra:

```yaml
# on tech-vm — append to the existing docker-compose.yml
  jaeger:
    image: jaegertracing/all-in-one:1.65.0
    container_name: jaeger
    restart: unless-stopped
    environment:
      COLLECTOR_OTLP_ENABLED: "true"
      SPAN_STORAGE_TYPE: "memory"
    ports:
      - "4317:4317"   # OTLP/gRPC
      - "4318:4318"   # OTLP/HTTP
      - "16686:16686" # Jaeger UI
```

Then:
```bash
ssh tech-vm 'cd <compose-dir> && docker compose pull jaeger && docker compose up -d jaeger'
```

**Alternative: standalone `docker run`** if tech-vm doesn't use compose for its infra:

```bash
docker run -d --name jaeger --restart unless-stopped \
  -e COLLECTOR_OTLP_ENABLED=true \
  -e SPAN_STORAGE_TYPE=memory \
  -p 4317:4317 \
  -p 4318:4318 \
  -p 16686:16686 \
  jaegertracing/all-in-one:1.65.0
```

Ports:
- `4317` — OTLP/gRPC ingress
- `4318` — OTLP/HTTP ingress (services use this; simpler to debug with curl)
- `16686` — Jaeger UI

On the services host, every service's `.env` gets:

```
BSS_OTEL_EXPORTER_OTLP_ENDPOINT=http://tech-vm:4318
```

Trade-offs: memory storage is lost on Jaeger restart. Fine for demo scope. For a long-running dev box, swap `SPAN_STORAGE_TYPE=memory` for `badger` with a mounted volume — documented in RUNBOOK.md as a "if you want persistence" note.

Full walkthrough in a new RUNBOOK.md section: **"Installing Jaeger on a BYOI host"**. Includes: the docker run command, firewall/Tailscale notes (ports 4318 and 16686 need to be reachable from the services box), a `curl http://tech-vm:16686/api/services` smoke test, and a troubleshooting checklist for "services started but no spans appearing".

#### 5b. Bundled-infra path (all-in-one quickstart)

`docker-compose.infra.yml` gains a `jaeger` service alongside postgres, rabbitmq, metabase:

```yaml
jaeger:
  image: jaegertracing/all-in-one:1.65.0
  environment:
    COLLECTOR_OTLP_ENABLED: "true"
    SPAN_STORAGE_TYPE: "memory"
  ports:
    - "4317:4317"
    - "4318:4318"
    - "16686:16686"
```

`.env.example` defaults `BSS_OTEL_EXPORTER_OTLP_ENDPOINT=http://jaeger:4318` for the bundled case (services resolve `jaeger` via the compose network). The BYOI section of README shows how to swap it for `http://tech-vm:4318`.

Both paths coexist. The operator picks one and sets the endpoint accordingly.

### 6. `audit.domain_event.trace_id` — additive migration

New column `trace_id TEXT NULL` on `audit.domain_event`. Populated from the current OTel context when the audit row is written (inside the same DB transaction as the domain write — same pattern as the existing `actor` / `channel` columns). Enables `bss trace for-order ORD-014` to look up the trace_id without the user copy-pasting from a log.

Alembic migration: additive, nullable column, no backfill. Old rows stay NULL. `bss trace for-*` commands fall back gracefully (`"no trace_id recorded for this order — was it created before v0.2?"`).

### 7. `bss trace` — replace the stub

Current file: `cli/bss_cli/commands/trace.py`, ~10 lines of `print("not implemented") + exit(2)`. Replace with:

```
bss trace <trace_id>                      # full swimlane
bss trace <trace_id> --service crm        # filter to one service column
bss trace <trace_id> --width 120          # override terminal width
bss trace <trace_id> --show-sql           # include SQL spans (default: collapsed for readability)
bss trace <trace_id> --json               # emit raw JSON for piping
bss trace for-order <order_id>            # resolve via audit.domain_event.trace_id
bss trace for-subscription <sub_id>       # same, for a subscription
bss trace for-ask                         # most recent `bss ask` invocation's trace_id
```

Implementation files:

- `cli/bss_cli/jaeger.py` — httpx wrapper for `/api/traces/<id>` and `/api/services`
- `cli/bss_cli/commands/trace.py` — Typer commands replacing the stub
- `cli/bss_cli/renderers/trace.py` — swimlane renderer (§8)
- `orchestrator/bss_orchestrator/tools/ops.py` — `trace.get`, `trace.for_order`, `trace.for_subscription` tool definitions so the LLM can call them during troubleshooting

Rendering rules:

- One row per span, indented by parent-child depth (2 spaces per level)
- Service column is left-padded to 12 chars; business-meaningful spans (the 3 manual ones) get a trailing `*`
- Bar uses `━` for duration, `┃` for start/end markers, scaled to `terminal_width - labels_width`
- Error spans render in red with `ERR` suffix
- SQL spans collapse by default (expanded with `--show-sql`) — otherwise a trace is 80% SQL noise and the shape is lost
- Header: trace_id (short), total wall-clock, span count, service count, error count
- Footer: count of hidden spans + tip to expand

Snapshot test: a deterministic fixture trace (JSON, committed to `tests/integration/v0_2_0/fixtures/signup_trace.json`) feeds the renderer and the output is diff-tested against a golden file at `tests/integration/v0_2_0/snapshots/signup_swimlane.txt`. Iterating on the renderer = regenerating the golden file intentionally.

## Doctrine and constraints

1. **Auto-instrumentation over manual spans.** Routers, services, policies, repositories stay untouched. The 3 manual spans in §2a are the only hand-written spans in the whole codebase. A grep guard in CI enforces this.
2. **No breaking changes to v0.1 APIs.** OTel propagates via `traceparent` / `tracestate` headers — additive. The `audit.domain_event.trace_id` column is nullable; old rows remain valid.
3. **<4 GB RAM budget still holds.** Services + OTel SDK is ~1.5 GB unchanged. Jaeger (bundled path) adds ~200 MB; bundled footprint becomes ~3.05 GB. BYOI path leaves the services-side footprint unchanged.
4. **<30s cold start still holds.** OTel SDK init is ~100 ms per service; parallel startup absorbs it.
5. **<50 ms p99 internal API still holds.** BatchSpanProcessor exports off the hot path. Re-measure at ship time; if p99 moved, document the new number in DECISIONS.md and decide whether it's acceptable.
6. **Campaign OS untouched.** OTel auto-instruments asyncpg globally — verify no code path in `packages/bss-telemetry/` or any service's telemetry wiring reads from `campaignos.*`. Grep guard enforces.
7. **PII discipline in span attributes.** OTel attributes are an easy vector for PII leakage (attach `email` "for debugging" and suddenly customer emails are in the trace backend). `bss_telemetry.semconv` defines the allowed attribute keys — all prefixed IDs, never raw email / NRIC / card / name. CI grep guard fails on `.set_attribute(.*email`, `.*card`, `.*nric` in any span-creating code path.
8. **Sampling ratio is env-driven.** 100% in dev (`BSS_OTEL_SAMPLING_RATIO=1.0`), tunable down for prod. Code never hardcodes a ratio.
9. **The CLI is a root span.** `bss ask` / `bss order create` / scenario-runner actions originate traces. `configure_telemetry(service_name="cli")` lives in `cli/bss_cli/main.py`'s Typer callback, before command dispatch. Without this, every trace would start at the first BSS service and mask the CLI's own latency.
10. **Observability must not gate the business path.** If Jaeger is unreachable, services log one warning on startup and proceed. `configure_telemetry` never raises into the service lifespan.

## Test strategy

1. **Unit — `packages/bss-telemetry/tests/`.** `configure_telemetry` returns a `TracerProvider` with the expected processors and exporter. `use_amqp_span` round-trips a traceparent through aio-pika message headers. `semconv` constants match the documented keys.
2. **Per-service telemetry wiring — `services/*/tests/test_telemetry_wiring.py`.** One file per service. Asserts that after the test fixture calls `configure_telemetry(BSS_OTEL_ENABLED=false)` (no-op mode for unit tests), an httpx request through the service still works, and that in enabled mode, spans are emitted with the expected attributes. Spans captured via the OTel `InMemorySpanExporter` test utility; no Jaeger required for unit tests.
3. **Cross-service integration — `tests/integration/v0_2_0/test_trace_propagation.py`.** Spin up the full compose stack (either profile). Send a signup request via the CLI. Poll Jaeger's HTTP API (`/api/traces/<id>`) for 5 seconds. Assert the returned trace has the expected service list and span chain. Uses `httpx` against Jaeger — no respx needed because this test wants the real wire.
4. **Snapshot — `tests/integration/v0_2_0/test_swimlane_snapshot.py`.** Deterministic fixture trace JSON → renderer → golden file diff.
5. **Hero scenario — `scenarios/trace_customer_signup_swimlane.yaml`.** End-to-end, tagged `hero, observability`, promoted into `make scenarios-hero`, must pass three runs in a row before the tag.

## Out of scope

- **Metrics.** Traces only in v0.2. OTel metrics + Prometheus scraping is a separate decision deferred past v0.6. Do not scope-creep — the SDK supports both but the storage backends, sampling models, and semantic conventions are different concerns.
- **Logs-to-OTel.** `structlog` JSON stays. Correlating logs to traces via the `trace_id` field in log lines already works (Phase 9 forward-compat).
- **Tempo / object-storage-backed trace storage.** Jaeger memory storage is sufficient for demo and BYOI.
- **Tail sampling.** Head sampling (`sampling_ratio`) only.
- **Cross-organization trace propagation.** Only internal BSS services + CLI + orchestrator. No trace context across external systems (payment processors, KYC vendors) — those are channel-layer concerns.
- **Trace-driven alerting.** Metrics territory. Not v0.2.
- **Replacing `audit.domain_event` with spans.** Audit is an append-only business log; traces are ops telemetry. Different retention, different consumers, different questions answered. Keep both.
- **Production-hardening the Jaeger deploy.** BYOI docs explicitly call out "memory storage, expect span loss on restart — demo-grade". Production persistence is a Tier-2 AWS deploy decision documented for the post-v0.6 window.
- **Breaking changes to `bss-clients` or service APIs.** Pure additive instrumentation. If a tempting "while we're touching everything" refactor shows up, it gets its own issue and a later phase.

## The trap

**Do not write a `tracer.start_as_current_span` call outside the 3 sites in §2a.** The grep guard will catch it, but more importantly — the point of v0.2 is that auto-instrumentation is enough. If you feel the urge to add a manual span to "explain what this service method does", the answer is a better method name, not a span. Manual spans that exist because auto feels sparse turn the swimlane into a forest where the signal gets lost.

**Do not pick Tempo over Jaeger because Tempo "looks more modern".** Tempo needs object storage. That storage needs a bucket. The bucket needs credentials. The credentials need rotation. None of that is the point. v0.2 ships a screenshot; Jaeger all-in-one is one container, one image, one port to remember.

**Do not skip asyncpg auto-instrumentation.** Without SQL spans, the swimlane is flat — you see "POST /productOrder took 680ms" with no visible reason. SQL spans are what make the swimlane legible and the screenshot worth pasting.

**Do not back-fill `trace_id` into existing `audit.domain_event` rows.** The column is nullable. Old rows stay NULL. `bss trace for-*` falls back with a helpful message. No migration heroics.

**Do not bundle metrics into v0.2 "because it's the same SDK".** Metrics have different sampling, different exporters, different storage questions, different semantic conventions. v0.2 is traces. v0.X+ can be metrics; not here.

**Do not forget the CLI root span.** If `bss trace` produces swimlanes that start at the first BSS service, you've lost 20-50 ms of CLI-local time that the screenshot needs to be honest about. `configure_telemetry("cli")` goes in the Typer callback in `cli/bss_cli/main.py`, before any command runs.

**Do not let observability crash the business path.** If Jaeger is unreachable, services must warn once and proceed. An exception from `configure_telemetry` that tears down service startup is a worse bug than the one it was meant to reveal.

**And do not ship v0.2 until three hero-scenario runs pass in a row.** v0.1 shipped on that discipline. v0.2 keeps it.
