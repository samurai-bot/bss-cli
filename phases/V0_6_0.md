# v0.6.0 — Docs sweep + renderer polish + tech debt

> **The "presentable before going more public" release.** No new features. Three loosely-coupled cleanup tracks bundled into one minor version because none alone is big enough to deserve its own: rewrite the docs around the post-v0.5 reality, iterate on the ASCII renderers that shipped flatter than their spec mockups suggested, and reconcile the DECISIONS.md-flagged drift that accumulated across v0.1 through v0.5. When v0.6 is done, someone finding the repo cold for the first time should be able to form an accurate mental model of the project in under fifteen minutes.

## Goal

Three tracks, one release:

1. **Docs sweep.** The post-v0.5 system has two portals, a trace backend, an auth token surface, and a meaningfully different "what does this do" story than the v0.1 README implies. Rewrite the docs to reflect what's actually shipped. Add two new top-level docs (CONTRIBUTING.md, ROADMAP.md) for outside contributors. Commit real screenshots of the portals and the `bss trace` swimlane into the repo.

2. **Renderer polish.** The v0.1 ship-gate review identified five hero ASCII renderers that shipped functional but flatter than the spec mockups: `bss subscription show`, `bss customer show` (the 360), `bss order show` (the decomposition tree), `bss catalog list` (the plan comparison), and the eSIM activation card. v0.6 iterates on each until it matches the spec's intent. Snapshot tests update intentionally. No new renderers, no new data, no new commands.

3. **Tech debt sweep.** Walk every DECISIONS.md entry that was written with a "track for later" note and either resolve it or explicitly re-defer with a new DECISIONS entry. Reconcile TOOL_SURFACE.md drift from v0.2-v0.5 (new `trace.*` tools, possibly `customer.find_by_msisdn`, possibly others). Re-measure the three ship-criteria budgets (RAM, cold start, p99 latency) — they've almost certainly moved with OTel + middleware + two portals, and SHIP_CRITERIA.md should state the v0.6 numbers honestly. Catalog service has a stale directory layout (pre-template era); reconcile it. Evaluate the per-service Dockerfile `sed` workaround against current `uv` workspace support. Audit silently-skipped or pre-v0.1 known-failing tests and either green them or retire them with reason.

Exit state: v0.6 ships when all six hero scenarios (v0.1 × 3, v0.2, v0.4, v0.5) pass three runs in a row against the re-measured stack, all new docs are committed, all renderer snapshots are updated with intentional diffs, and DECISIONS.md has an entry for every drift item that was either fixed or re-deferred.

#### 2.6 Snapshot test discipline

Every renderer edit regenerates the corresponding snapshot intentionally. The PR diff is the review — reviewers read the golden-file diff to confirm the visual change is what was intended.

New test helper in `cli/bss_cli/tests/conftest.py`: `assert_snapshot(name, actual)` — reads the golden file, diffs, on mismatch prints both versions side-by-side. On `UPDATE_SNAPSHOTS=1`, writes `actual` to the golden file instead.

Workflow documented in RUNBOOK.md §"Regenerating renderer snapshots":
```
UPDATE_SNAPSHOTS=1 uv run pytest cli/bss_cli/tests/test_renderers.py
git add cli/bss_cli/tests/snapshots/
# review the diff; commit with the renderer change in the same PR
```

#### 3.1 Catalog service structure reconciliation

Per v0.6-entry reconnaissance, the catalog service uses an older layout (`bss_catalog/app.py` + `routes/` + `schemas/` + `repository.py`) while every other service uses the post-Phase-3 template (`app/main.py` + `app/api/` + `app/services/` + `app/policies/` + `app/repositories/` + `app/auth_context.py`).

Deliverable: migrate catalog to the standard structure. Constraints:

- **Structural-only.** No endpoint signatures change. No policy surface changes. No schema changes.
- Existing `services/catalog/tests/` must pass without modification.
- Contract tests with other services (v0.1 hero scenarios) must pass without modification.
- One commit, reviewable as a pure reorg.

If the migration proves non-trivial (catalog turns out to have shape that doesn't fit the template — e.g., it's stateless and has no policies, so `app/policies/` stays empty), document the rationale in DECISIONS.md and either proceed with empty-directory-ceremony or grant catalog a documented exception. The decision lands in DECISIONS; the outcome is binary: reconciled or formally exempted.

#### 3.2 Per-service Dockerfile `sed` workaround — re-evaluate

DECISIONS.md 2026-04-11 documents: each service's Dockerfile runs `sed -i 's|workspace = true|path = "../../packages/..."|' pyproject.toml` before `uv pip install`, because `uv` workspace resolution doesn't work inside a Docker build context that only contains the service subtree. Tracked as Phase 11 backlog.

v0.6 evaluates:

- Is the current `uv` version's `sync --package <service>` workspace handling improved? (Claude Code runs `uv --version` and checks changelogs.)
- Does a shared template Dockerfile copying the workspace root `pyproject.toml` + `uv.lock` + all of `packages/` + the target service directory work cleanly now? If yes, migrate all services + both portals to the shared template.
- If no, update the DECISIONS.md entry with the v0.6-era status and keep the workaround.

Binary outcome: migrate, or formally re-defer with a new DECISIONS entry documenting why and what has to change for the next attempt.

#### 3.4 `datetime.now()` grep guard in CI

DECISIONS.md says "No business-logic module calls `datetime.utcnow()` — `bss_clock.now()` is the only path. Enforced by grep guard in CI (post-v0.1)." The grep guard was not added in v0.1.

v0.6 adds it. One-line addition to CI config and `Makefile`:

```makefile
doctrine-check:
	@rg 'datetime\.(now|utcnow)\(\)' services/ portals/ orchestrator/ cli/bss_cli/ \
	   --glob '!**/config.py' --glob '!**/tests/**' && \
	   (echo "FAIL: use bss_clock.now() instead of datetime.now/utcnow" && exit 1) || \
	   echo "OK: no datetime.now/utcnow in business logic"
```

Add to CI workflow as a required check. Run locally as part of `make verify`.

If the grep currently finds violations, fix them as part of v0.6 (they're bugs that should never have made it past v0.1 and only didn't bite because clock-freezing wasn't thoroughly tested).

#### 3.5 Ship criteria re-measurement

SHIP_CRITERIA.md currently states v0.1 numbers: <4 GB RAM, <30s cold start, <50ms p99 internal API. The post-v0.5 system has:

- +Jaeger (bundled path): ~200 MB
- +OTel SDK in every service: ~5-10 MB each + batch processor thread
- +Middleware: ~50 µs per request
- +Portals × 2: ~150 MB each (bundled path)

v0.6 re-measures and updates SHIP_CRITERIA.md. Steps:

1. Fresh compose up (bundled path, all-in-one including Jaeger, both portals, all 9 services, postgres, rabbitmq, metabase)
2. `docker stats --no-stream` after 5 minutes idle → RAM number
3. `docker compose down && time docker compose up -d --wait` → cold start number
4. Run `bss scenario run scenarios/customer_signup_and_exhaust.yaml` 10 times; read `bss trace` swimlanes; compute p99 of internal HTTP span durations → p99 number

**Document the v0.6 numbers even if they're worse than v0.1.** If the stack now takes 4.5 GB bundled (300 MB over v0.1 ceiling), that's the number SHIP_CRITERIA.md states, and either the ceiling is formally raised (new DECISIONS entry) or items are trimmed until the original ceiling holds. The point is honesty.

BYOI numbers separately: services + portals only (no postgres/rabbitmq/metabase/jaeger) — should stay well under 2 GB.

## Doctrine and constraints

1. **No new features.** Zero. If a renderer polish suggests "we could add `--sort-by-balance`", defer to v0.7+. The v0.6 budget is audit, reconcile, polish.
2. **Every change is reversible.** No destructive deletions without DECISIONS entries. No structural refactors that can't be cleanly rolled back.
3. **Every v0.1 through v0.5 hero scenario still passes unchanged.** Six hero scenarios total. Three runs in a row each. Any YAML edit is a red flag that v0.6 leaked into behavior.
4. **Every existing test suite is still green.** `make test` before v0.6 work = green. After v0.6 work = green. Every delta is intentional.
5. **Snapshot test changes are reviewed as diffs.** Golden-file updates are part of the PR, not auto-regenerated in CI. If a reviewer can't read the before/after and say "yes, that's the polish we wanted", the PR doesn't merge.
6. **Campaign OS untouched.** Grep guards carry forward from v0.1-v0.5.
7. **Docs changes reflect current reality, not aspiration.** ROADMAP.md can list future plans, but README / ARCHITECTURE / CLAUDE describe what exists at the v0.6 tag.
8. **Ship-criteria honesty.** Update the numbers to whatever the v0.6 stack measures. If worse, that's the honest state. If the original ceiling holds, great. Do not fudge.
9. **Screenshots are committed to the repo.** Not external links. Not generated at build time. Real bytes at `docs/screenshots/`, real filenames in README.
10. **v0.6 is the minor version before v1.0 territory becomes a real question.** Optimize the repo for a stranger discovering it cold, not for the people who have been in every session.

## Test strategy

v0.6 is audit + reconcile + polish. Test strategy is inversion-shaped: prove nothing broke.

1. **Full suite green before work begins.** `make test` baseline capture.
2. **Full suite green after every track.** After docs sweep (should be trivially green since no code changed), after renderer polish (snapshots updated intentionally), after each tech-debt item.
3. **All six hero scenarios pass three runs in a row.** `make scenarios-hero` loop:
   - `customer_signup_and_exhaust` (v0.1)
   - `new_activation_with_provisioning_retry` (v0.1)
   - `llm_troubleshoot_blocked_subscription` (v0.1)
   - `trace_customer_signup_swimlane` (v0.2)
   - `portal_self_serve_signup` (v0.4)
   - `portal_csr_blocked_diagnosis` (v0.5)
4. **Renderer snapshot tests pass with the intentional golden-file updates.** The diff between old and new golden files is the review artifact.
5. **Grep-guard suite runs clean.** `datetime.now()` (new in v0.6), OTel business-logic imports (v0.2), middleware presence (v0.3), portal-writes-through-agent (v0.4/v0.5), campaignos untouched (v0.1+).
6. **Ship-criteria measurements reproducible.** The exact commands and numbers land in SHIP_CRITERIA.md; a reviewer can re-run and confirm.

- **Refactors for aesthetic reasons.** Catalog reconciliation is targeted because it's documented drift. Other services stay as they are even if their layout could be "nicer".
- **Performance optimization beyond honest re-measurement.** If p99 has regressed to 70 ms, document it; don't spend v0.6 shaving it back to 50 ms.
- **Expanding scenario coverage.** No new hero scenarios.
- **Converting known-failing tests into feature work.** If greening a test requires new capability, retire the test and log the capability as a post-v0.6 roadmap item.
- **CLAUDE.md doctrine rewrites.** Drift-fix edits only; the contract stays the contract.
- **Dependency upgrades.** Pin changes are fine if needed for the Dockerfile workspace evaluation (§3.2); nothing else.
- **CI/CD platform changes.** If CI gains a doctrine-check step, that's additive. No switching GitHub Actions → something else.
- **Multi-language / accessibility passes on the portals.** Still desktop-only English.
- **Vendoring or unvendoring.** `htmx.min.js` stays vendored; no CDN migration.

## The trap

**Do not let "polish" become "rewrite".** The renderer polish track is the highest-risk spot for scope creep — there's always a cleaner way to structure any renderer, and the spec mockup always has three more features you could implement. Iterate on what's there; add nothing.

**Do not skip re-measurement because "the numbers probably didn't move much".** OTel alone adds a non-trivial fraction per service. Two portals add memory. Middleware adds per-request µs. The SHIP_CRITERIA numbers have almost certainly moved; the honest thing is to say by how much. If you defer the measurement, the next version inherits stale numbers and nobody knows the real state.

**Do not green a known-failing test by commenting out its assertions.** If the test fails, either the code is wrong (fix it) or the test is wrong (delete it with a DECISIONS entry). "Make the test pass by weakening it" is the worst of both worlds — the red flag goes away and the underlying gap stays.

**Do not rewrite CLAUDE.md in v0.6.** Drift-fix edits only. A rewrite is a Phase 0 amendment, which is not v0.6's job. If some doctrine feels wrong, flag it in a DECISIONS entry for the next phase to consider.

**Do not add new screenshots for features that don't exist yet.** ROADMAP.md is where future plans go. Screenshots in README.md and `docs/screenshots/` show what's real.

**Do not merge a renderer polish commit without a human-readable diff of the golden files.** The PR review is the sanity check — if a reviewer sees "60 lines of ASCII changed" and can't tell what's different, the PR is not reviewed.

**Do not migrate the Dockerfile template without running the full test suite in the new image.** The sed workaround was chosen in Phase 4 for a reason. If `uv sync --package <service>` works in 2026 where it didn't in Phase 4, prove it: build every service with the new template, bring up compose, run hero scenarios. If any of those fails, re-defer and write the DECISIONS entry.

**Do not ship v0.6 unless all six hero scenarios are green three runs in a row.** v0.1 held that line at three scenarios. v0.2 held it at four. v0.4 added five, v0.5 added six. v0.6 keeps it. The hero scenario gate is the only thing that catches cross-cutting regressions across a reconcile-heavy release.

**And do not let v0.6 turn into v1.0.** This is a polish release, not a rebranding release. The jump to 1.0 is a separate conversation — it implies API stability guarantees, formal support windows, and a wider audience. v0.6 is about being presentable. Tag `v0.6.0`, not `v1.0.0`.
