# v0.15.0 — KYC (Didit, Channel-Layer) + eSIM Adapter Seam

> **The release that puts a real third-party identity verifier in front of signup, without ever letting raw PII cross the BSS boundary, and locks in the eSIM-provider seam for v0.16+.** v0.15 inherits the v0.14 substrate (`integrations` schema, `bss-webhooks` package, `external_ref` audit envelope, `bss onboard` skeleton) and uses it for two new integrations. KYC is the headline: Didit's hosted flow handles document capture + liveness; the portal forwards a signed JWS attestation to BSS; the BSS-side policy actually verifies the JWS now (it was a stub through v0.14). The eSIM track is a thin Protocol seam over `provisioning-sim` so v0.16+ real-provider work (1GLOBAL Connect, eSIM Access — both NDA-gated) is a config swap, not a rewrite. No real eSIM provider integration ships in v0.15; that's deferred until an NDA-gated reseller agreement is in place.

The architecture pattern is inherited verbatim from v0.14. **KYC integration lives in the portal, not in any BSS service** — per CLAUDE.md "Scope boundaries: eKYC", BSS only receives signed attestations. Didit does liveness + doc capture at the portal; portal forwards the verified attestation to BSS via the existing `customer.attest_kyc` tool. The novelty in v0.15 is on the BSS side: signature verification stops being a stub and becomes a real JWS check against Didit's JWKS, with explicit allowlist for the prebaked attestation flow that scenarios depend on.

## Goal

Three deliverables, scoped to be reviewable in a single ~4–5 day window:

1. **KYC — Didit (channel-layer, lives in portal).** New `KycVerificationAdapter` Protocol with `PrebakedKycAdapter` + `DiditKycAdapter`. Portal signup flow swaps the inline prebaked-signature template for the adapter call. Didit's hosted flow runs in the customer's browser; the callback handler verifies Didit's JWS against its JWKS, hashes the document number inside the adapter, and forwards the normalized attestation to BSS.
2. **BSS-side signature verification upgrade.** `services/crm/app/policies/kyc.py:check_attestation_signature` (lines 22–31) is currently a stub that accepts any payload with a `signature` field. v0.15 upgrades it to provider-aware JWS verification with an explicit `BSS_KYC_ALLOW_PREBAKED` flag for the prebaked path. This is a doctrine change: the BSS does care that the signature is real, even though the verification *flow* lives at the portal — otherwise a compromised portal could mint attestations.
3. **eSIM SM-DP+ adapter seam.** New `EsimProviderAdapter` Protocol with `SimEsimProvider` (only impl) wrapping the existing `provisioning-sim` worker. Stub classes for `OneGlobalEsimProvider` and `EsimAccessEsimProvider` raise `NotImplementedError` with a v0.16+ pointer. `BSS_ESIM_PROVIDER=sim` is the only valid value in v0.15.

When v0.15 ships, an operator can run `bss onboard --domain kyc`, paste a Didit sandbox workflow id, restart the portal, and complete a real KYC flow against Didit's sandbox — with the JWS verified end-to-end by the BSS, the document-number hash stored only on the BSS side, and the cap counter visible via `bss external-calls --provider didit --month-to-date`.

## Deliverables

### Track 1 — KYC adapter (portal-side)

#### 1.1 New `KycVerificationAdapter` Protocol

```
portals/self-serve/bss_self_serve/kyc/
├── __init__.py
├── adapter.py        # KycVerificationAdapter Protocol + KycSession + KycAttestation dataclasses
├── prebaked.py       # PrebakedKycAdapter — preserves current behavior
├── didit.py          # DiditKycAdapter — real
└── select.py         # select_kyc_adapter()
```

Protocol shape:

```python
class KycVerificationAdapter(Protocol):
    async def initiate(self, *, email: str, return_url: str) -> KycSession: ...
    async def fetch_attestation(self, *, session_id: str) -> KycAttestation: ...
```

`PrebakedKycAdapter` returns the current `KYC_PREBAKED_SIGNATURE_TEMPLATE` flow synchronously (matches today's behavior at `signup.py:350` exactly). It exists so dev/scenario flows don't need a Didit sandbox key, and so the v0.12 14-day soak corpus continues to work.

`DiditKycAdapter`:
- `initiate()` POSTs Didit `/sessions` with the configured workflow id, returns `{session_id, redirect_url}`. Records to `integrations.external_call` with operation `initiate`, aggregate `(identity, IDT-...)`.
- `fetch_attestation()` GETs Didit `/sessions/{id}/decision`, verifies the JWS against Didit's JWKS (cached, rotated on `kid` mismatch), returns a normalized `KycAttestation` with `{provider:"didit", provider_reference, document_type, document_number_hash, document_country, date_of_birth, signature_payload}`. **Document number is hashed inside the adapter before returning** — raw PII never crosses the BSS boundary.

Env vars:

```bash
BSS_PORTAL_KYC_PROVIDER=didit             # prebaked|didit
BSS_PORTAL_KYC_DIDIT_API_KEY=...
BSS_PORTAL_KYC_DIDIT_WEBHOOK_SECRET=...
BSS_PORTAL_KYC_DIDIT_WORKFLOW_ID=wf_...
BSS_PORTAL_KYC_DIDIT_JWKS_URL=https://verification.didit.me/.well-known/jwks.json
```

Naming follows the v0.14-locked convention.

#### 1.2 Didit free-tier cap behavior — hard-block

`DiditKycAdapter` tracks monthly call count via `integrations.external_call` query (`WHERE provider='didit' AND occurred_at >= date_trunc('month', now())`). On approach (≥450/500), emits a `kyc.cap_warning` event for ops; on cap, `initiate()` raises `KycCapExhausted`; portal renders a templated "KYC temporarily unavailable, please try again later" page; emits a high-priority `kyc.cap_exhausted` event.

**No silent fallback to prebaked, ever** — silent trust degradation is a doctrine violation. The cap behavior is deliberate: cheaper to lose a few signups for a month than to ship customers with unverified identity claims.

#### 1.3 Portal signup flow rework

`portals/self-serve/bss_self_serve/routes/signup.py` (around line 350) replaces the inline `KYC_PREBAKED_SIGNATURE_TEMPLATE` call with `request.app.state.kyc_adapter.initiate(...)`. New callback route `/signup/step/kyc/callback` resumes after the customer returns from Didit's hosted flow; calls `fetch_attestation`; passes the result to `customer.attest_kyc` via `bss-clients` (existing path).

New webhook `POST /webhooks/didit` covers the "customer closed the tab after liveness check" case — secondary path that reconciles by `session_id`. Verifies Didit HMAC signature via `bss_webhooks.signatures.verify_signature(scheme="didit_hmac", …)` (the scheme already exists from v0.14, just unused until now). Persists into `integrations.webhook_event`.

### Track 2 — BSS-side JWS verification upgrade

#### 2.1 `services/crm/app/policies/kyc.py:check_attestation_signature`

Currently (v0.14 and earlier) a stub that accepts any payload with a `signature` field. v0.15 upgrades to provider-aware:

- `provider="didit"` → fetch JWKS (cached on first use, refreshed on `kid` mismatch with a 5-minute negative cache), verify JWS, validate claims (`iss=didit`, `aud=<expected_audience>`, `exp` not in past, `iat` not far in future).
- `provider="prebaked"` → preserve current stub but **only** when `BSS_KYC_ALLOW_PREBAKED=true`.

`BSS_KYC_ALLOW_PREBAKED` defaults to `true` in non-production envs (`BSS_ENV != production`), `false` in production. Production deployments using `prebaked` must explicitly opt in with the env var, and the choice is logged loudly at startup.

Failed verification raises `PolicyViolation` with `code=POLICY_VIOLATION`, `rule=kyc.attestation.signature_invalid`. The error envelope follows the existing policy-violation shape so the LLM observation path doesn't need to change.

#### 2.2 Soak corpus impact

v0.12's 14-day soak runs with `BSS_PORTAL_KYC_PROVIDER=prebaked` + `BSS_KYC_ALLOW_PREBAKED=true`; existing soak result remains valid because the prebaked path is preserved. Document this explicitly in the PR description.

#### 2.3 JWKS caching primitive

A small new helper at `services/crm/app/policies/_jwks_cache.py` (or as a `bss-webhooks` extension if shape allows): JWKS-by-URL with `kid`-aware refresh, default 1h TTL, negative cache 5min on fetch failure. Tested against fixture JWKS endpoints. Reused by v0.16 only if Stripe ever moves to JWS-signed webhooks (currently HMAC); stays in CRM policies if not.

### Track 3 — eSIM adapter seam

Tiny, ~1 day. Locks in the architecture so v0.16+ real-provider swap isn't a rewrite.

#### 3.1 New `EsimProviderAdapter` Protocol

```python
# services/provisioning-sim/app/domain/esim_provider.py — NEW
class EsimProviderAdapter(Protocol):
    async def order_profile(self, *, iccid: str, imsi: str, msisdn: str) -> EsimOrderResult: ...
    async def release_profile(self, *, iccid: str) -> None: ...
    async def get_activation_code(self, *, iccid: str) -> str: ...
```

`SimEsimProvider` wraps the existing `worker.process_task` simulation paths (HLR/PCRF/OCS/SM-DP+ collapsed). `worker.py` no longer hardcodes the dispatch — it calls `app.state.esim_provider.order_profile(...)`.

Stub classes for the two real providers:

```python
class OneGlobalEsimProvider(EsimProviderAdapter):
    async def order_profile(self, *, iccid, imsi, msisdn):
        raise NotImplementedError(
            "v0.16+: 1GLOBAL Connect adapter requires NDA + production credentials. "
            "Set BSS_ESIM_PROVIDER=sim until the integration ships."
        )
    # release_profile, get_activation_code — same shape

class EsimAccessEsimProvider(EsimProviderAdapter):
    # same pattern, points at eSIM Access reseller API
```

`select_esim_provider("onbglobal" | "esim_access")` returns the stub; the stub raises on first call. Startup does *not* fail — the operator might have legitimately set the env var in advance of the v0.16 integration. (This is the one place where "fail-fast" cedes to "fail-on-first-use"; the stub message is loud enough.)

Env var: `BSS_ESIM_PROVIDER=sim` (only valid in v0.15; `onbglobal`/`esim_access` accept the env but error on use).

#### 3.2 `worker.py` decoupling

`services/provisioning-sim/app/domain/worker.py` currently dispatches by task type with hardcoded sleeps. v0.15 replaces the eSIM-related branches (`ESIM_PROFILE_PREPARE`, eventually `ESIM_PROFILE_REARM`) with `app.state.esim_provider.order_profile(...)`. The other simulator branches (HLR, PCRF, OCS) stay hardcoded — they're not getting providerized in v0.16+ either.

## Doctrine and constraints

Inherited from v0.14:

1. Per-domain adapter Protocols. No mega-package. No broker container.
2. `BSS_<DOMAIN>_PROVIDER` env naming.
3. Fail-fast at startup on unknown provider, missing key, or env-mode mismatch.
4. Webhooks land on the receiving domain's service; no broker.
5. `integrations.external_call` + `integrations.webhook_event` are the forensic substrate.
6. Secrets in `.env`, never `settings.toml`.

New in v0.15:

7. **KYC verification flow lives in the portal.** Signature verification lives in BSS policy (`crm/app/policies/kyc.py`). The portal's adapter cannot be the only line of defense — a compromised portal would otherwise mint attestations.
8. **Raw KYC document numbers never cross the BSS boundary.** Hash inside `DiditKycAdapter.fetch_attestation` before returning. The BSS-side `kyc_attestation` row only ever stores hashes.
9. **No silent fallback to prebaked when Didit caps out.** Hard-block. Surface as an operator-visible event. Customer sees a templated error; ops gets the alert.
10. **Production with `provider="prebaked"` requires explicit opt-in via `BSS_KYC_ALLOW_PREBAKED=true`.** Default in production is to reject prebaked attestations.

### Anti-patterns added to CLAUDE.md

- **(v0.15+) Don't add a `services/kyc/` container.** KYC integration is channel-layer; BSS only verifies signed attestations. Adding a service inverts the doctrine. The portal IS the channel layer in this deployment.
- **(v0.15+) Don't pass raw KYC document numbers across the BSS boundary.** Hash inside `DiditKycAdapter.fetch_attestation` before returning. Greppable: `services/crm/.../*.py` and `audit.domain_event.payload` queries should never see a 9-or-12-character ID-document pattern.
- **(v0.15+) Don't silently fall back to a different KYC provider when one caps out.** Trust posture quietly degrading is exactly the failure mode KYC is supposed to prevent.
- **(v0.15+) Don't accept attestations with `provider="prebaked"` in production unless `BSS_KYC_ALLOW_PREBAKED=true` is set explicitly.** The default is reject.
- **(v0.15+) Don't fail startup on `BSS_ESIM_PROVIDER=onbglobal` or `=esim_access`.** Fail on first use with the v0.16+ pointer message. Allowing the env var to be set in advance of the integration is the affordance.

## Test strategy

- **Unit**: `KycVerificationAdapter` Protocol + `PrebakedKycAdapter` + `DiditKycAdapter` (with a faked Didit HTTP client and a fixture JWKS endpoint); `select_kyc_adapter` for unknown-name, missing-key, env-mode mismatch.
- **Integration**: `check_attestation_signature` against tampered JWS, expired JWS, wrong-issuer JWS, JWKS-key-rotation; cap-counter logic against `integrations.external_call` rows.
- **Didit sandbox test** (gated on env, skipped in CI by default): full end-to-end including real network call to Didit, manual completion of the hosted flow, callback receipt, JWS verification.
- **eSIM Protocol smoke**: existing v0.13 provisioning hero scenarios pass unchanged with `BSS_ESIM_PROVIDER=sim`. Setting `=onbglobal` and triggering a profile order produces the v0.16+ pointer error.
- **Hero scenarios**: v0.13 heroes pass three runs in `=prebaked` mode (no behavioral change). One new hero scenario: "signup with Didit sandbox" lands in CI as nightly-only.
- **Doctrine guards** added to `make doctrine-check`:
  - No `services/kyc/` directory exists.
  - `DiditKycAdapter.fetch_attestation` returns a `KycAttestation` whose `document_number_hash` is non-empty and whose payload does not contain the raw `document_number` field.
  - No code outside `select_kyc_adapter` reads `BSS_PORTAL_KYC_PROVIDER`.

**Manual testing budget for v0.15: one round.** Didit hosted flow with a sandbox workflow, including the doc-capture + liveness UX. eSIM seam validation rolls into the same round (existing provisioning hero scenarios).

## The trap

**The trap of "BSS just trusts the portal":** today's stub at `services/crm/app/policies/kyc.py:check_attestation_signature` accepts any payload with a `signature` field. The temptation in v0.15 is to leave it stub-shaped and rely on the portal to verify Didit's JWS before forwarding. *Don't.* A compromised portal could mint attestations; the policy layer is the last enforcement point. Verify on both sides — portal because that's where the JWKS lives, BSS because that's where the trust decision matters.

**The second trap:** treating the eSIM seam as a placeholder. The `SimEsimProvider` MUST behave identically to today's hardcoded dispatch — same task durations, same fault-injection hooks, same audit trail. If v0.13 hero scenarios fail after the seam lands, the seam is wrong, not the scenarios. The whole point is that v0.15 doesn't change behavior; it only changes where the behavior is defined.

**The third trap:** the Didit free-tier cap is a binding constraint and silent fallback feels helpful. It isn't. A signup that completes with a "best-effort prebaked attestation because we ran out of Didit calls this month" is an attestation the regulator will not honor. Hard-block, alert, surface the cap. The cost of refusing 50 signups in a corner case is much lower than the cost of one regulator-flagged customer who got through with no real verification.

**The fourth trap:** raw document numbers. Didit returns the document_number in plaintext as part of the attestation payload. The hash MUST happen inside `DiditKycAdapter.fetch_attestation` before the return. Once it crosses into BSS code, you have a regulatory-data-handling problem you didn't have before. Greppable: `services/crm/.../*.py` should never contain a regex matching ID-document patterns.

**The fifth trap:** building a generic JWKS validator that's "ready for any provider." Don't. v0.15 needs Didit's JWKS verified; that's it. If Stripe (v0.16) ever moves to JWS-signed webhooks, *then* generalize. Today's `_jwks_cache.py` should hard-code the issuer claim it expects and the audience it requires; the Didit JWS shape is the only one in scope.

Don't call this v0.16 — v0.16 is Payment, and Payment has its own PCI scope, its own portal UI rework, its own idempotency story. Keep them separate.
