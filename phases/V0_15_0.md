# v0.15.0 — KYC (Didit, Channel-Layer) + eSIM Adapter Seam

> **The release that puts a real third-party identity verifier in front of signup, without ever letting raw PII cross the BSS boundary, and locks in the eSIM-provider seam for v0.16+.** v0.15 inherits the v0.14 substrate (`integrations` schema, `bss-webhooks` package, `external_ref` audit envelope, `bss onboard` skeleton) and uses it for two new integrations. KYC is the headline: Didit's hosted flow handles document capture + liveness; the portal forwards a signed JWS attestation to BSS; the BSS-side policy actually verifies the JWS now (it was a stub through v0.14). The eSIM track is a thin Protocol seam over `provisioning-sim` so v0.16+ real-provider work (1GLOBAL Connect, eSIM Access — both NDA-gated) is a config swap, not a rewrite. No real eSIM provider integration ships in v0.15; that's deferred until an NDA-gated reseller agreement is in place.

The architecture pattern is inherited verbatim from v0.14. **KYC integration lives in the portal, not in any BSS service** — per CLAUDE.md "Scope boundaries: eKYC", BSS only receives signed attestations. Didit does liveness + doc capture at the portal; portal forwards the verified attestation to BSS via the existing `customer.attest_kyc` tool. The novelty in v0.15 is on the BSS side: signature verification stops being a stub and becomes a real JWS check against Didit's JWKS, with explicit allowlist for the prebaked attestation flow that scenarios depend on.

## Goal

Three deliverables, scoped to be reviewable in a single ~4–5 day window:

1. **KYC — Didit (channel-layer, lives in portal).** New `KycVerificationAdapter` Protocol with `PrebakedKycAdapter` + `DiditKycAdapter`. Portal signup flow swaps the inline prebaked-signature template for the adapter call. Didit's hosted flow runs in the customer's browser; the callback handler verifies Didit's JWS against its JWKS, hashes the document number inside the adapter, and forwards the normalized attestation to BSS.
2. **BSS-side signature verification upgrade.** `services/crm/app/policies/kyc.py:check_attestation_signature` (lines 22–31) is currently a stub that accepts any payload with a `signature` field. v0.15 upgrades it to provider-aware JWS verification with an explicit `BSS_KYC_ALLOW_PREBAKED` flag for the prebaked path. This is a doctrine change: the BSS does care that the signature is real, even though the verification *flow* lives at the portal — otherwise a compromised portal could mint attestations.
3. **eSIM SM-DP+ adapter seam.** New `EsimProviderAdapter` Protocol with `SimEsimProvider` (only impl) wrapping the existing `provisioning-sim` worker. Stub classes for `OneGlobalEsimProvider` and `EsimAccessEsimProvider` raise `NotImplementedError` with a v0.16+ pointer. `BSS_ESIM_PROVIDER=sim` is the only valid value in v0.15.

When v0.15 ships, an operator can run `bss onboard --domain kyc`, paste a Didit sandbox workflow id, restart the portal, and complete a real KYC flow against Didit's sandbox — with the JWS verified end-to-end by the BSS, the document-number hash stored only on the BSS side, and the cap counter visible via `bss external-calls --provider didit --month-to-date`.

## Deliverables

### Track 1 — KYC adapter (portal-side)

#### 1.1 New `KycVerificationAdapter` Protocol

```
portals/self-serve/bss_self_serve/kyc/
├── __init__.py
├── adapter.py        # KycVerificationAdapter Protocol + KycSession + KycAttestation dataclasses
├── prebaked.py       # PrebakedKycAdapter — preserves current behavior
├── didit.py          # DiditKycAdapter — real
└── select.py         # select_kyc_adapter()
```

Protocol shape:

```python
class KycVerificationAdapter(Protocol):
    async def initiate(self, *, email: str, return_url: str) -> KycSession: ...
    async def fetch_attestation(self, *, session_id: str) -> KycAttestation: ...
```

`PrebakedKycAdapter` returns the current `KYC_PREBAKED_SIGNATURE_TEMPLATE` flow synchronously (matches today's behavior at `signup.py:350` exactly). It exists so dev/scenario flows don't need a Didit sandbox key, and so the v0.12 14-day soak corpus continues to work.

`DiditKycAdapter`:
- `initiate()` POSTs Didit `/v2/session/` with the configured workflow id, returns `{session_id, redirect_url}`. Records to `integrations.external_call` with operation `initiate`, aggregate `(identity, IDT-...)`. (Probed against the live sandbox 2026-05-02; returns HTTP 201 with `{session_id, session_token, url, status, workflow_id, ...}`.)
- `fetch_attestation()` GETs Didit `/v2/session/{id}/decision/` **after the corroborating webhook has been received**. The webhook (HMAC-signed, verified at `/webhooks/didit`) is the trust anchor; the API response is plain TLS-only JSON used for image URLs and supplementary fields. Method blocks on the corroboration row in `integrations.kyc_webhook_corroboration` (new in v0.15) for up to 10s; falls through with `KycCorroborationTimeout` if the webhook never arrives. Returns a normalized `KycAttestation` with `{provider:"didit", provider_reference, document_type, document_number_last4, document_number_hash, document_country, date_of_birth, corroboration_id}` — see §1.4 for the full shape and §2 for the BSS-side check. **Raw document number is reduced to last-4 + hash inside the adapter before returning** — neither the raw value nor the rest of the digits ever cross the BSS boundary.

Env vars:

```bash
BSS_PORTAL_KYC_PROVIDER=didit             # prebaked|didit
BSS_PORTAL_KYC_DIDIT_API_KEY=...
BSS_PORTAL_KYC_DIDIT_WEBHOOK_SECRET=...
BSS_PORTAL_KYC_DIDIT_WORKFLOW_ID=7411e1f2-119d-4eee-9b8c-6e759933c2b8
```

Naming follows the v0.14-locked convention. Workflow id is a raw UUID — Didit's dashboard returns the id without a `wf_` prefix; do not synthesize one.

**Note on `BSS_PORTAL_KYC_DIDIT_JWKS_URL`** — earlier drafts of this spec listed a JWKS URL because the design assumed Didit returns a JWS-signed decision body. The 2026-05-02 sandbox probe disproved that assumption: the `GET /v2/session/{id}/decision/` response is plain JSON over TLS with no JWS, JWT, or signature field. Didit's trust anchor is the **HMAC-signed webhook**, not a JWS over the API response. The JWKS env var is dropped from v0.15. If Didit ever ships a signed-decision API tier, reintroduce it then.

#### 1.2 Didit free-tier cap behavior — hard-block

`DiditKycAdapter` tracks monthly call count via `integrations.external_call` query (`WHERE provider='didit' AND occurred_at >= date_trunc('month', now())`). On approach (≥450/500), emits a `kyc.cap_warning` event for ops; on cap, `initiate()` raises `KycCapExhausted`; portal renders a templated "KYC temporarily unavailable, please try again later" page; emits a high-priority `kyc.cap_exhausted` event.

**No silent fallback to prebaked, ever** — silent trust degradation is a doctrine violation. The cap behavior is deliberate: cheaper to lose a few signups for a month than to ship customers with unverified identity claims.

#### 1.3 Portal signup flow rework

`portals/self-serve/bss_self_serve/routes/signup.py` (around line 350) replaces the inline `KYC_PREBAKED_SIGNATURE_TEMPLATE` call with `request.app.state.kyc_adapter.initiate(...)`. New callback route `/signup/step/kyc/callback` resumes after the customer returns from Didit's hosted flow; calls `fetch_attestation`; passes the result to `customer.attest_kyc` via `bss-clients` (existing path).

New webhook `POST /webhooks/didit` covers the "customer closed the tab after liveness check" case — secondary path that reconciles by `session_id`. Verifies Didit HMAC signature via `bss_webhooks.signatures.verify_signature(scheme="didit_hmac", …)` (the scheme already exists from v0.14, just unused until now). Persists into `integrations.webhook_event`.

Didit Webhooks v3.0 uses a resource-generic event taxonomy (`status.updated`, `data.updated`, `user.*`, `business.*`, `transaction.*`, `activity.created`). v0.15 subscribes to **two events**:

- `status.updated` — verification status changed (`approved` / `declined` / `expired` / `in_review`). Drives the reconciliation path when the portal callback was missed.
- `data.updated` — decision payload available (extracted document fields, liveness result). Triggers `fetch_attestation`.

`user.*` (Didit persistent-user model — we don't use it), `business.*` (KYB, not KYC), and `transaction.*` (Didit billing-side) are explicitly NOT subscribed. `activity.created` is optional for forensic completeness; `bss_webhooks` will idempotently persist any unsubscribed-but-delivered event as `process_outcome='noop'`. The two-event subscription is the minimum-surface contract; expanding it requires a doctrine note.

#### 1.4 `KycAttestation` shape (privacy doctrine)

The 2026-05-02 sandbox probe confirmed Didit's `GET /v2/session/{id}/decision/` returns the raw `document_number` (e.g. `S8369796B`) plus full PII (name, DOB, address, place of birth, presigned S3 image URLs of the document and selfie). v0.15's privacy doctrine: **only last-4 digits + hash cross the BSS boundary**.

```python
@dataclass(frozen=True)
class KycAttestation:
    provider: Literal["didit", "prebaked"]
    provider_reference: str            # Didit session_id (UUID); used to look up corroboration row
    document_type: str                 # "Identity Card" | "Passport" | "Driver's License" (Didit's vocabulary)
    document_country: str              # ISO 3166-1 alpha-3, e.g. "SGP"
    document_number_last4: str         # PDPA-aligned masking; e.g. "796B" for NRIC S8369796B
    document_number_hash: str          # SHA-256 hex digest of the full normalized document number — uniqueness for dedupe / dispute resolution
    date_of_birth: date                # Required for age verification on signup
    corroboration_id: UUID             # FK into integrations.kyc_webhook_corroboration (proves a verified webhook saw this same session_id)
```

What's deliberately NOT on the attestation (and therefore never reaches BSS):

- Raw `document_number`
- `first_name`, `last_name`, `full_name`, `address`, `place_of_birth`
- Any image / video URL (presigned or otherwise)
- Liveness score, face match score, age estimation
- MRZ data, NFC payload, parsed-address breakdown

The customer's identifying details on the BSS side stay in `crm.customer` (which the customer themselves provided during signup); the attestation is purely a *verification receipt* that a regulated provider corroborated those details. If ops needs to dispute or trace a verification, they query the Didit dashboard by `provider_reference`, not by extracting fields from BSS.

Last-4 format is **the last 4 characters** of the normalized document number (matching Singapore PDPA convention for NRIC display: `*****796B`). For passports / FIN this is the last 4 alphanumerics. For numeric-only documents (e.g. some EU national IDs) it's the last 4 digits.

The hash is `sha256(normalize(document_number) + document_country + provider).hexdigest()` — domain-separated so the same number across providers / countries doesn't collide. Hash is irreversible; lookups by raw number are not supported by design.

### Track 2 — BSS-side webhook-corroboration verification

The earlier draft of this section called for JWS-against-JWKS verification on the BSS side. The 2026-05-02 sandbox probe falsified that design: Didit's decision API has no JWS. The trust anchor is the **HMAC-signed webhook**, and BSS-side verification is now corroboration-based: the policy layer checks that a verified webhook delivery exists for the session_id being attested to.

#### 2.1 `integrations.kyc_webhook_corroboration` table (NEW)

Migration 0009 in `packages/bss-models`:

```sql
CREATE TABLE integrations.kyc_webhook_corroboration (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id TEXT NOT NULL DEFAULT 'DEFAULT',
    provider TEXT NOT NULL,                            -- 'didit'
    provider_session_id TEXT NOT NULL UNIQUE,          -- Didit session UUID
    webhook_event_id UUID NOT NULL REFERENCES integrations.webhook_event(id),
    decision_status TEXT NOT NULL,                     -- 'Approved' | 'Declined' | 'Expired' | 'In Review'
    decision_body_digest TEXT NOT NULL,                -- SHA-256 of canonical-JSON of the webhook decision body
    received_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON integrations.kyc_webhook_corroboration (provider, provider_session_id);
```

Written by the `/webhooks/didit` route handler **after** `bss_webhooks.signatures.verify_signature(scheme="didit_hmac", …)` succeeds. The row is the verified-record-of-truth for this session.

#### 2.2 `services/crm/app/policies/kyc.py:check_attestation_signature`

Renamed semantically (kept method name for back-compat) to "verify the attestation has a corroborating verified-webhook row":

- `provider="didit"` → policy queries `integrations.kyc_webhook_corroboration WHERE provider='didit' AND provider_session_id=<attestation.provider_reference>`. Validates:
  - Row exists.
  - `decision_status` matches the implied status (attestation only carries verified data when status was `Approved`; an `Approved` attestation backed by a `Declined` corroboration is rejected).
  - `received_at` is within the last 30 minutes (replay guard; legitimate flows complete within this window — adjust if real-world data shows tail latency).
- `provider="prebaked"` → no corroboration row required; preserve current stub but **only** when `BSS_KYC_ALLOW_PREBAKED=true`.

`BSS_KYC_ALLOW_PREBAKED` defaults to `true` in non-production (`BSS_ENV != production`), `false` in production. Production deployments using `prebaked` must explicitly opt in; the choice is logged loudly at startup.

Failed verification raises `PolicyViolation` with `code=POLICY_VIOLATION`, `rule=kyc.attestation.uncorroborated` (was `kyc.attestation.signature_invalid` in the older JWS-shaped draft; rename clarifies the actual mechanism). The error envelope follows the existing shape.

#### 2.3 Portal-side wait-for-corroboration

`DiditKycAdapter.fetch_attestation` blocks on the corroboration row before returning. Implementation:

1. Poll `integrations.kyc_webhook_corroboration` by `(provider='didit', provider_session_id=<session_id>)` every 200ms.
2. Timeout at 10s. If the webhook hasn't arrived, raise `KycCorroborationTimeout`. Portal renders a "verification taking longer than expected, please try again in a moment" templated page with a retry button (the webhook will likely arrive momentarily; re-clicking the callback URL repolls).
3. On hit: read `webhook_event_id`, fetch the verified webhook body, build the `KycAttestation` from the webhook body (NOT the API response — the API response is unsigned and untrusted in this model). The Didit `GET /decision/` is still called for the supplementary fields (image URLs etc.) only if v0.15 ever needs them; v0.15 ships without using them.

If the user closes the tab before the portal callback fires, the `status.updated` webhook still arrives at `/webhooks/didit`. A separate reconciler (out of scope for v0.15; deferred to v0.15.1 or v0.16) walks `integrations.kyc_webhook_corroboration` rows that have no matching `crm.kyc_attestation` row and either auto-attests (if the user already has a customer record) or expires them.

#### 2.4 Soak corpus impact

v0.12's 14-day soak runs with `BSS_PORTAL_KYC_PROVIDER=prebaked` + `BSS_KYC_ALLOW_PREBAKED=true`; existing soak result remains valid because the prebaked path is preserved (no corroboration row required). Document explicitly in the PR description.

#### 4.1 `write_env_file` backup safeguard

v0.14's `cli/bss_cli/commands/onboard.py:write_env_file` does an atomic `.env.tmp → os.replace(target)` rename — safe against torn writes (process crash mid-write) but **not** against logic errors in the wizard. v0.15 is the first release where multiple domain configurators run in one wizard invocation (email + KYC), raising the blast radius if a `_configure_*()` returns malformed state.

Behavior to add (one change in `write_env_file`, applies to all domains):

1. Before the atomic rename, if `path` exists, copy it to `path.parent / f"{path.name}.backup-{YYYY-MM-DD-HHMMSS}"` via `shutil.copy2` (preserves mtime + permissions).
2. After successful write, glob `.env.backup-????-??-??-??????` in the same directory; keep the 5 most recent, delete the rest. Hand-created backups (e.g. `.env.backup-2026-05-02` without a `-HHMMSS` suffix) are NOT matched by the glob and are left alone.
3. Print the backup path in the wizard summary so the operator can `mv` it back if anything looks wrong.

Rationale: belt-and-suspenders. `os.replace` is atomic against crashes; the timestamped backup is the recourse against logic bugs. Five backups is arbitrary but bounded.

#### 4.2 `Makefile:SCENARIOS_RUN` — KYC flip-and-restore

The v0.14 macro at `Makefile:244-254` flips `BSS_PORTAL_EMAIL_PROVIDER=resend → logging` for the duration of a scenarios run, then restores via `trap` on EXIT/INT/TERM. v0.15 must extend it with the same dance for KYC, otherwise an operator with `BSS_PORTAL_KYC_PROVIDER=didit` who runs `make scenarios-hero` will burn the Didit free-tier counter on every run and the scenarios will hang waiting for a human to complete the hosted UI (no automation path).

Behavior:

1. Read `BSS_PORTAL_KYC_PROVIDER` from `.env`. If it's anything other than `prebaked` (typically `didit`), flip to `prebaked` via `sed -i.bak` and inject `BSS_KYC_ALLOW_PREBAKED=true` (the BSS-side policy requires this opt-in once Track 2 lands; scenarios don't carry production env semantics).
2. Force-recreate the affected containers (`portal-self-serve` + `crm`).
3. Trap restores the original `BSS_PORTAL_KYC_PROVIDER` and removes the wizard-injected `BSS_KYC_ALLOW_PREBAKED=true` if it wasn't set before, on EXIT/INT/TERM. Force-recreate again.
4. Print `▶ scenarios: flipping BSS_PORTAL_KYC_PROVIDER=didit → prebaked` on entry and `▶ scenarios: restored BSS_PORTAL_KYC_PROVIDER=didit` on exit, matching the email lines' style.

The production-rejection path (verification checklist line: `BSS_ENV=production` + no `BSS_KYC_ALLOW_PREBAKED` opt-in → reject) is covered by a unit test, not a hero scenario, so the Makefile change does not need to handle `BSS_ENV` at all.

Implementation lands in **Commit 2** (Tracks 1+2), not Commit 3 — the flip is load-bearing the moment `BSS_PORTAL_KYC_PROVIDER` becomes a real switch in the portal.

## Doctrine and constraints

Inherited from v0.14:

1. Per-domain adapter Protocols. No mega-package. No broker container.
2. `BSS_<DOMAIN>_PROVIDER` env naming.
3. Fail-fast at startup on unknown provider, missing key, or env-mode mismatch.
4. Webhooks land on the receiving domain's service; no broker.
5. `integrations.external_call` + `integrations.webhook_event` are the forensic substrate.
6. Secrets in `.env`, never `settings.toml`.

New in v0.15:

7. **KYC verification flow lives in the portal.** Signature verification lives in BSS policy (`crm/app/policies/kyc.py`). The portal's adapter cannot be the only line of defense — a compromised portal would otherwise mint attestations.
8. **Raw KYC document numbers never cross the BSS boundary.** Hash inside `DiditKycAdapter.fetch_attestation` before returning. The BSS-side `kyc_attestation` row only ever stores hashes.
9. **No silent fallback to prebaked when Didit caps out.** Hard-block. Surface as an operator-visible event. Customer sees a templated error; ops gets the alert.
10. **Production with `provider="prebaked"` requires explicit opt-in via `BSS_KYC_ALLOW_PREBAKED=true`.** Default in production is to reject prebaked attestations.

### Anti-patterns added to CLAUDE.md

- **(v0.15+) Don't add a `services/kyc/` container.** KYC integration is channel-layer; BSS only verifies signed attestations. Adding a service inverts the doctrine. The portal IS the channel layer in this deployment.
- **(v0.15+) Don't pass raw KYC document numbers across the BSS boundary.** Reduce to `document_number_last4` (PDPA-aligned masking) + `document_number_hash` (SHA-256, domain-separated) inside `DiditKycAdapter.fetch_attestation` before returning. Greppable: `services/crm/.../*.py` and `audit.domain_event.payload` queries should never see a 9-or-12-character ID-document pattern. The doctrine guard runs a regex sweep over the `KycAttestation` payload at adapter-return time and at policy-entry time.
- **(v0.15+) Don't store full names, addresses, or biometric URLs on the `KycAttestation`.** Only `document_number_last4`, `document_number_hash`, `document_type`, `document_country`, `date_of_birth`, `provider_reference`, and `corroboration_id` cross. Customer-identifying details on the BSS side stay in `crm.customer` from the customer's own signup form. The attestation is a verification *receipt*, not an identity dossier.
- **(v0.15+) Don't trust a Didit attestation without a corroborating verified-webhook row.** The `GET /v2/session/{id}/decision/` API response is plain TLS JSON and forgeable by a compromised portal. The trust anchor is the HMAC-signed webhook, recorded in `integrations.kyc_webhook_corroboration`. Policy rejects (`rule=kyc.attestation.uncorroborated`) any `provider="didit"` attestation whose `provider_reference` has no matching row.
- **(v0.15+) Don't silently fall back to a different KYC provider when one caps out.** Trust posture quietly degrading is exactly the failure mode KYC is supposed to prevent.
- **(v0.15+) Don't accept attestations with `provider="prebaked"` in production unless `BSS_KYC_ALLOW_PREBAKED=true` is set explicitly.** The default is reject.
- **(v0.15+) Don't fail startup on `BSS_ESIM_PROVIDER=onbglobal` or `=esim_access`.** Fail on first use with the v0.16+ pointer message. Allowing the env var to be set in advance of the integration is the affordance.

## Test strategy

- **Unit**: `KycVerificationAdapter` Protocol + `PrebakedKycAdapter` + `DiditKycAdapter` (with a faked Didit HTTP client and the fixture decision payload at `portals/self-serve/tests/fixtures/didit_decision_sample.json` — captured from the real sandbox 2026-05-02, fully PII-redacted); `select_kyc_adapter` for unknown-name, missing-key, env-mode mismatch.
- **Integration**: `check_attestation_signature` against missing-corroboration, stale-corroboration (>30min), declined-corroboration-but-approved-attestation, and tampered-body-digest cases. Cap-counter logic against `integrations.external_call` rows. Webhook-arrives-before-callback and webhook-arrives-after-callback (timing race) both pass.
- **Didit sandbox test** (gated on env, skipped in CI by default): full end-to-end including real network call to Didit, manual completion of the hosted flow, webhook delivery, corroboration row written, callback completes, attestation verified.

## The trap

**The trap of "BSS just trusts the portal":** today's stub at `services/crm/app/policies/kyc.py:check_attestation_signature` accepts any payload with a `signature` field. The temptation in v0.15 is to leave it stub-shaped and rely on the portal to verify Didit before forwarding. *Don't.* The Didit `GET /v2/session/{id}/decision/` API returns plain TLS-only JSON with no signature; a compromised portal could trivially mint attestations from synthesized API responses. The trust anchor is the **HMAC-signed webhook**, recorded in `integrations.kyc_webhook_corroboration`, and BSS policy checks for that row before accepting an attestation. Verify on both sides — portal verifies the webhook HMAC at ingestion, BSS verifies the corroboration row at attestation time.

**The second trap:** treating the eSIM seam as a placeholder. The `SimEsimProvider` MUST behave identically to today's hardcoded dispatch — same task durations, same fault-injection hooks, same audit trail. If v0.13 hero scenarios fail after the seam lands, the seam is wrong, not the scenarios. The whole point is that v0.15 doesn't change behavior; it only changes where the behavior is defined.

**The third trap:** the Didit free-tier cap is a binding constraint and silent fallback feels helpful. It isn't. A signup that completes with a "best-effort prebaked attestation because we ran out of Didit calls this month" is an attestation the regulator will not honor. Hard-block, alert, surface the cap. The cost of refusing 50 signups in a corner case is much lower than the cost of one regulator-flagged customer who got through with no real verification.

**The fourth trap:** raw document numbers and PII. The 2026-05-02 sandbox probe confirmed Didit returns the full NRIC, name, address, DOB, place of birth, and presigned image URLs of both the document and the selfie. The reduction to `document_number_last4` + `document_number_hash` MUST happen inside `DiditKycAdapter.fetch_attestation` before the return — and the names / addresses / URLs MUST be dropped, not transformed. Once any of this crosses into BSS code, you have a regulatory-data-handling problem you didn't have before. Greppable: `services/crm/.../*.py` should never contain a regex matching ID-document patterns or `front_image|back_image|portrait_image|reference_image|video_url|first_name|last_name|address`.

**The fifth trap:** assuming the Didit decision API is signed. It isn't (probed 2026-05-02). The earlier draft of this spec called for a `_jwks_cache.py` and JWS validation; both were removed in the same-day amendment. If a future Didit API tier ships signed payloads, reintroduce the verifier *then* — don't pre-build it now and let it rot.

**The sixth trap:** the timing race between webhook delivery and the customer pressing "back" in the hosted UI. Didit's webhook arrives independently of the customer-controlled redirect; in normal flows the webhook lands first (typically <1s after Didit's decision), but a slow tunnel or an aggressive user can flip the order. The portal's `fetch_attestation` MUST poll for the corroboration row, not assume one ordering. The 10s timeout is a balance — long enough to absorb any reasonable tunnel hiccup, short enough that a stuck customer sees a clean retry page rather than a hung tab.

Don't call this v0.16 — v0.16 is Payment, and Payment has its own PCI scope, its own portal UI rework, its own idempotency story. Keep them separate.
