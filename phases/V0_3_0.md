# v0.3.0 — Shared API token middleware

> **The smallest possible auth story.** One shared secret in `.env`, one middleware, one change in `bss-clients`. No OAuth, no JWT, no KYC, no RBAC — those are Phase 12 and beyond. v0.3 exists to close the "unauthenticated wide-open HTTP surface" hole that v0.1 and v0.2 leave behind, using the `auth_context.py` seam that's been sitting in every service since Phase 3 specifically for this moment.

## Goal

Every HTTP endpoint on every BSS service currently accepts any request. In a local dev environment that's fine; the moment the stack is reachable from anything beyond `localhost` or a private VPN, it's a credential-free API surface. v0.3 introduces a single shared API token — `BSS_API_TOKEN` — that every service requires in the `X-BSS-API-Token` header. Missing or wrong token → 401 before the endpoint runs. Correct token → the request proceeds and the existing `auth_context.current()` returns a hardcoded admin principal, exactly as it does today in v0.1.

The design philosophy: **do the thing that takes two hours and unlocks Tier 2 AWS deployment, not the thing that takes two weeks and delays it**. A single shared token is equivalent (for a single-operator deployment) to what mature SaaS tools call an "admin API key". It's not a multi-tenant auth system, and it's not pretending to be. When the user base ever needs real per-principal auth with roles, Phase 12 ships that through the exact same `auth_context.py` seam — this version just populates it from a token validator instead of a hardcoded admin.

## Deliverables

### 1. `packages/bss-middleware/` — shared middleware package

New workspace package. Lives parallel to `bss-clock`, `bss-events`, `bss-telemetry`.

```
packages/bss-middleware/
├── pyproject.toml
├── bss_middleware/
│   ├── __init__.py
│   ├── token_auth.py        # BSSApiTokenMiddleware
│   ├── config.py            # pydantic-settings for BSS_API_TOKEN, _REPO_ROOT pattern
│   └── startup.py           # validate_api_token_present() — fail-fast on empty token
└── tests/
    ├── test_token_auth.py
    └── test_startup.py
```

#### 1a. `BSSApiTokenMiddleware`

FastAPI/Starlette ASGI middleware. Behavior:

1. If `request.url.path` is in the **exemption allowlist** (`/health`, `/health/ready`, `/health/live`), pass through unconditionally. Everything else requires the header. No exceptions for `/docs`, `/openapi.json`, or `/metrics` — those are admin surfaces, token-protected like everything else.
2. Extract `X-BSS-API-Token` from request headers.
3. If missing → return `401 {"code": "AUTH_MISSING_TOKEN", "message": "X-BSS-API-Token header required"}`.
4. Compare with `hmac.compare_digest(provided_token, BSS_API_TOKEN)` — **timing-safe**. A naive `==` comparison on bytes is a timing oracle; `hmac.compare_digest` is the standard library answer.
5. If mismatch → return `401 {"code": "AUTH_INVALID_TOKEN", "message": "invalid API token"}`.
6. If match → set the auth_context `ContextVar` with the admin principal, then await `call_next(request)`.

**Log-once policy** on 401: the middleware logs the first 401 per `(remote_addr, path)` combo at INFO, silent thereafter for 60 seconds, to avoid log spam from probes. Not rate limiting — just log throttling.

#### 1b. Startup validator

`validate_api_token_present()` is called by each service's lifespan before any other setup:

- Reads `BSS_API_TOKEN` from env (via the package's pydantic-settings config).
- If empty, None, or `"changeme"` (a sentinel the `.env.example` ships with) → raises `RuntimeError("BSS_API_TOKEN is unset or still the default; set it in .env before starting services")`.
- If length < 32 bytes → raises `RuntimeError("BSS_API_TOKEN is too short (<32 chars); generate a proper token: openssl rand -hex 32")`.
- Otherwise returns silently.

Services fail fast on startup, not on first unauthenticated request. Compose's healthcheck surfaces the crash immediately.

### 2. Service wiring — 9 services, one shape

Each service's `main.py` adds two things. For `services/crm/app/main.py`:

```python
from bss_middleware import BSSApiTokenMiddleware, validate_api_token_present

@asynccontextmanager
async def lifespan(app: FastAPI):
    validate_api_token_present()                     # <-- fail-fast
    configure_telemetry(service_name="crm")           # (from v0.2)
    # ... existing startup
    yield
    # ... existing teardown

app = FastAPI(lifespan=lifespan)
app.add_middleware(BSSApiTokenMiddleware)             # <-- protect every endpoint
# ... existing routers
```

No router changes. No endpoint changes. No `Depends()` annotations. The middleware runs before routing, which means before the endpoint function is chosen, which means token-protection is mandatory by construction — you cannot forget to protect a new endpoint because there is no per-endpoint protection to forget.

### 3. `auth_context.py` — populate from validated token

Every service has `app/auth_context.py` that today returns a hardcoded admin principal regardless of request state. In v0.3, the middleware writes to a `ContextVar` on successful token validation, and `auth_context.current()` reads from it:

```python
# services/<svc>/app/auth_context.py
from contextvars import ContextVar
from bss_middleware.token_auth import AUTH_CONTEXT_VAR

def current() -> AuthContext:
    ctx = AUTH_CONTEXT_VAR.get(None)
    if ctx is None:
        # defensive — should only happen if a code path bypasses the middleware
        raise RuntimeError("auth_context.current() called outside a token-validated request")
    return ctx
```

The AuthContext returned is still `AuthContext(actor='system', tenant='DEFAULT', roles=['admin'], permissions=['*'])` in v0.3 — the same value the hardcoded version returned. The difference is that it's now gated by a real token check, and Phase 12 can change the principal per-request without touching policy code.

**Identical change in all 9 services.** One file per service, ~5 lines edited each.

### 4. `packages/bss-clients/` — token injection on outbound calls

`bss-clients` already has an `AuthProvider` protocol with a `NoAuthProvider` default (Phase 5). v0.3 adds a `TokenAuthProvider`:

```python
# packages/bss-clients/bss_clients/auth.py

class TokenAuthProvider(AuthProvider):
    def __init__(self, token: str):
        if not token:
            raise ValueError("TokenAuthProvider requires a non-empty token")
        self._token = token

    def get_headers(self) -> dict[str, str]:
        return {"X-BSS-API-Token": self._token}
```

The default provider used by services + CLI + orchestrator + scenario runner switches from `NoAuthProvider` to `TokenAuthProvider(token=os.environ["BSS_API_TOKEN"])`. This is a **one-line change** in each of: `cli/bss_cli/config.py`, `orchestrator/bss_orchestrator/clients.py`, each service's `main.py` lifespan (where it builds cross-service clients), and `cli/bss_cli/scenarios/actions.py`.

`TokenAuthProvider.get_headers()` merges into the existing `X-BSS-Actor` / `X-BSS-Channel` / `X-Request-ID` headers — every outbound call now carries all four.

### 5. `.env.example` + rotation procedure

`.env.example` gains:

```
# Required — generate via: openssl rand -hex 32
BSS_API_TOKEN=changeme
```

The `changeme` sentinel is what the startup validator rejects — `cp .env.example .env` without editing produces an immediate crash with a clear error message, which is the desired UX.

**Rotation procedure** (documented in RUNBOOK.md, new section "Rotating BSS_API_TOKEN"):

1. Generate: `openssl rand -hex 32 | tee new_token.txt`
2. Update `.env` on all service hosts and client hosts (CLI, orchestrator) with the new value
3. Restart services: `docker compose down && docker compose up -d`
4. Shred `new_token.txt`

**v0.3 does not support zero-downtime rotation.** A rolling strategy (services accept current OR next token during a window) is explicitly deferred to Phase 12, where real auth makes it moot. For v0.3, a 60-second compose restart is the rotation window. This is acceptable for single-operator single-tenant deployments; it would not be acceptable for Tier 3 production, which requires Phase 12 first anyway.

### 6. CLI, orchestrator, scenario runner — one-line changes each

Every component that builds a `bss-clients` client needs to pass `TokenAuthProvider` instead of the implicit `NoAuthProvider`:

| File | Change |
|---|---|
| `cli/bss_cli/config.py` | read `BSS_API_TOKEN`; pass to client factory |
| `cli/bss_cli/commands/*.py` | no change — they use the config-built clients |
| `orchestrator/bss_orchestrator/clients.py` | read `BSS_API_TOKEN`; pass to client factory |
| `orchestrator/bss_orchestrator/config.py` | declare `BSS_API_TOKEN` in settings |
| `cli/bss_cli/scenarios/actions.py` | no change — uses orchestrator clients |
| `cli/bss_cli/jaeger.py` | **no token** — Jaeger is the trace backend, not a BSS service |

The CLI fails fast if `BSS_API_TOKEN` is unset: `bss ...` exits with `error: BSS_API_TOKEN not set — check your .env`.

## Doctrine and constraints

1. **Token is all-or-nothing.** A valid token = admin access. No scopes, no roles, no per-endpoint permissions. v0.3 does not attempt to be a real auth system.
2. **Timing-safe comparison.** `hmac.compare_digest` only. A plain `==` would leak timing information about token prefix length.
3. **Fail-fast on startup.** Empty/default/short token → service refuses to start. Never ship with `BSS_API_TOKEN=changeme`.
4. **The middleware is the only chokepoint.** No `Depends(require_token)` decorators on individual endpoints. Every endpoint is protected automatically because it sits behind the middleware. If you find yourself wanting to bypass for "just this one endpoint", add it to the exemption allowlist in one place instead.
5. **Exemption list is exactly `/health`, `/health/ready`, `/health/live`.** Not `/docs`, not `/openapi.json`, not `/metrics`. The OpenAPI spec leaks endpoint signatures and is a reconnaissance aid for an attacker — token-protect it.
6. **No zero-downtime rotation in v0.3.** Restart-based rotation only. Rolling rotation is a real auth system feature; defer to Phase 12.
7. **No breaking changes to v0.1 / v0.2 APIs.** Adding a mandatory header is technically breaking for existing callers, but v0.1 and v0.2 have no external callers — all callers (CLI, orchestrator, scenario runner, integration tests) are in-repo and updated in the same commit.
8. **Campaign OS untouched.** The middleware wraps BSS services only. `campaignos` schema access and the separate production workload are unaffected (and remain on their own auth, whatever that is).
9. **<50 ms p99 internal API still holds.** Middleware overhead is ~50 µs per request (one dict lookup + one `compare_digest` call). Re-measure; no budget headroom loss expected.
10. **The token never appears in logs.** `structlog` has a redaction filter — add `X-BSS-API-Token` to the redacted-headers list. Audit events never store the token. Error messages for auth failures never echo the provided (wrong) token back.

## Test strategy

1. **Unit — `packages/bss-middleware/tests/`.** `BSSApiTokenMiddleware` returns 401 on missing header, 401 on wrong token, passes through on correct token. `validate_api_token_present` raises on empty/default/short/unset. `hmac.compare_digest` usage verified by inspection (not a runtime test — a mutation test that changes `compare_digest` to `==` should be caught in review).
2. **Per-service — `services/*/tests/test_api_token.py`.** One file per service. Three cases minimum: (a) no header → 401, (b) wrong token → 401, (c) correct token → 200 on a representative endpoint. Uses the existing httpx ASGITransport fixture pattern from Phase 5; conftest.py sets `BSS_API_TOKEN` to a known test value via monkeypatch.
3. **Cross-service integration — `tests/integration/v0_3_0/test_token_propagation.py`.** Spin up the full compose stack. Fire a request via the CLI (which builds clients with `TokenAuthProvider`). Assert the request reaches the target service. Then forcibly unset `BSS_API_TOKEN` on the CLI side and assert the request fails with a clean error.
4. **Hero scenarios still pass unmodified.** All three v0.1 hero scenarios should continue to pass without YAML edits — the token plumbing is transparent to the scenario runner because it goes through the same bss-clients path. This is the implicit regression gate for v0.3. If a scenario breaks, the token plumbing leaked into the public surface where it shouldn't.
5. **`usage.simulate` still works for scenarios.** The scenario runner path through `actions.py` → bss-clients → mediation needs to keep carrying the token. Part of the cross-service integration test.

## Out of scope

- **OAuth2, JWT, SAML, OIDC.** All deferred to Phase 12. `auth_context.py` remains the seam.
- **Per-principal auth.** One token, one admin principal. No user accounts.
- **Roles, scopes, permissions beyond `['*']`.** Deferred to Phase 12.
- **mTLS between services.** Shared token is enough for the BSS internal surface.
- **Zero-downtime token rotation.** Restart-based rotation only. Rolling rotation requires a real auth system.
- **Token revocation.** Equivalent to rotation — there's one token, revoking it = rotating.
- **Multi-tenant tokens.** `tenant_id='DEFAULT'` stays everywhere. Multi-tenancy is a v0.3+ concern tracked separately.
- **Rate limiting per token.** Single token = single caller; no rate-limit policy makes sense.
- **Audit log of auth failures.** The log-once-per-60s policy is deliberate — full audit trail of 401s is noise. Real auth failure auditing is Phase 12.
- **UI for token management.** It's one env var. No UI.

## The trap

**Do not add `Depends(require_token)` on individual endpoints.** The middleware is the chokepoint. Per-endpoint dependencies are how you end up with an endpoint that was added in a hurry and forgot the decorator. Middleware means "protected by default, and the only way to unprotect is to edit one centralized allowlist."

**Do not use `==` on tokens.** `hmac.compare_digest`. Yes, in 2026, with cloud networks and rate limits, the practical exploitability of a token timing oracle is low. Use `compare_digest` anyway — it costs nothing, it's the right primitive, and reviewers will trust the code more.

**Do not add `/docs` or `/openapi.json` to the exemption list.** OpenAPI schemas are a reconnaissance aid. If someone hits `/docs` without a token, they should see 401, not the endpoint catalog.

**Do not try to build rolling rotation into v0.3.** "Accept current OR next token for 60 minutes" is a real auth system feature. v0.3 is restart-based rotation. If you start down the rolling path, you're rebuilding Phase 12 without the rest of it.

**Do not smuggle role-based checks in "while we're touching auth_context".** v0.3's `auth_context` returns the same admin principal it did in v0.1. The point is the seam, not the content. Phase 12 fills the seam with real claims — v0.3 just gates it behind a token.

**Do not forget the scenario runner.** It calls services via orchestrator → bss-clients. If `TokenAuthProvider` isn't wired there, scenarios start returning 401 and the regression gate breaks. It's one line in `cli/bss_cli/scenarios/actions.py` (or wherever the shared client factory lives — verify during reconnaissance).

**Do not let the token leak into logs or error responses.** The error message for `AUTH_INVALID_TOKEN` says "invalid API token" — it does not echo back the provided token, even partially. That's a reconnaissance aid for the attacker trying to probe length or character set.

**And do not declare v0.3 done before running every hero scenario three times in a row.** The token plumbing is only right if every existing flow works transparently.
