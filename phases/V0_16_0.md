# v0.16.0 — Payment (Stripe, Card-on-File, Webhook-Driven Reconciliation)

> **The release that lets a real customer's real card mint a real Stripe payment_method, get charged for a real bundle, and have the result reconciled by a signed webhook — without a single PAN ever touching BSS.** v0.16 is the largest of the three provider-integration releases (v0.14 email → v0.15 KYC + eSIM seam → v0.16 payment) and gets its own window because the work is genuinely heavier: a brand-new `TokenizerAdapter` Protocol where v0.14/v0.15 just extended existing patterns, a Stripe Elements rework of the portal's COF flow (the most invasive single change of the three releases), a webhook receiver in the payment service that reconciles charges + records out-of-band refunds + records chargebacks (record-only — motto #1), and the documented cutover playbook for an operator switching from mock tokens to live Stripe.

The architecture pattern is inherited from v0.14: per-domain adapter Protocols, no broker container. The novelty is that **v0.16 is the first integration where the portal UI itself changes shape** — not just an adapter swap, but a rework of how the customer's browser submits card details. Stripe.js + Elements collects the PAN client-side, hands the portal a `pm_*` id, and the BSS payment service charges that id off-session. The doctrine line: PAN never touches BSS in production. The startup-time guard refuses to boot the portal in production mode if a card-number `<input>` field is in any rendered template.

## Goal

Five deliverables, scoped to a ~7–8 day window with two manual-test rounds:

1. **`TokenizerAdapter` Protocol + Stripe + Mock impls.** New Protocol at `services/payment/app/domain/tokenizer.py`; `MockTokenizerAdapter` wraps the existing mock tokenizer (preserving FAIL/DECLINE test affordances); `StripeTokenizerAdapter` calls `stripe.PaymentIntent.create` with idempotency keys. `select_tokenizer()` dispatches at startup. `PaymentService.__init__` takes the adapter as a constructor arg.
2. **Stripe Elements portal rework (production mode).** Replace the server-rendered card-number form at `signup.py:393+` with a Stripe.js + Elements iframe when `BSS_PAYMENT_PROVIDER=stripe`. Mock-mode keeps the existing form for dev affordance. Startup guard refuses to boot in `production` + `stripe` mode if any card-number input survives in templates.
3. **Stripe webhook receiver.** `POST /webhooks/stripe` on payment service (port 8003). Verifies `Stripe-Signature` via `bss_webhooks.signatures.verify_signature(scheme="stripe", …)` (the scheme already exists from v0.14, just unused until now). Reconciles `charge.succeeded` / `charge.failed`, records `charge.refunded` (out-of-band refund — emits domain event for the cockpit to surface), records `charge.dispute.created` (chargeback — record-only, no auto-action — motto #1).
4. **Idempotency + cutover playbook.** `idempotency_key = f"ATT-{attempt_id}-r{retry_count}"` with documented retry semantics. Migration story for existing mock-format `payment_method.token` rows when an operator switches to Stripe: lazy-fail by default (next charge fails cleanly with `mock_token_in_stripe_mode`); `bss payment cutover --invalidate-mock-tokens` for proactive cutover.
5. **Three-provider end-to-end soak.** v0.14 + v0.15 + v0.16 together: a hero scenario that runs end-to-end with `BSS_PORTAL_EMAIL_PROVIDER=resend` + `BSS_PORTAL_KYC_PROVIDER=didit` + `BSS_PAYMENT_PROVIDER=stripe` against all three sandboxes, three runs in nightly CI. This is the "production-shape" gate; without it, "v0.14 + v0.15 + v0.16 ship together" is a claim, not a proof.

When v0.16 ships, the four-mottos-aligned MVNO has all three real providers wired, with a documented homelab and cloud deploy story. v1.0 then becomes a question of "did an operator actually run this for 90 days," not "is the integration code real."

## Deliverables

### Track 1 — `TokenizerAdapter` Protocol + impls

#### 1.1 New Protocol

```python
# services/payment/app/domain/tokenizer.py — NEW
class TokenizerAdapter(Protocol):
    async def charge(
        self,
        token: str,                       # Stripe pm_..., or mock tok_...
        amount: Decimal,
        currency: str,
        *,
        idempotency_key: str,
        purpose: str,
        customer_external_ref: str | None,  # Stripe customer id; None for mock
    ) -> ChargeResult: ...

    def tokenize(
        self, card_number: str, exp_month: int, exp_year: int, cvv: str
    ) -> TokenizeResult: ...

    async def attach_payment_method_to_customer(
        self, *, payment_method_id: str, customer_id: str
    ) -> None: ...

    async def ensure_customer(
        self, *, bss_customer_id: str, email: str
    ) -> str: ...   # returns provider-side customer id (cus_... for Stripe)
```

Three impls in `services/payment/app/domain/`:

- `MockTokenizerAdapter` — wraps existing `mock_tokenizer.tokenize_card` + `charge`. `tokenize()` works (dev affordance). `attach_payment_method_to_customer` is a no-op. `ensure_customer` returns a mock `cus_mock_<bss_id>`. Preserves FAIL/DECLINE/`tok_FAIL_*` test affordances exactly.
- `StripeTokenizerAdapter`:
  - `tokenize()` raises `NotImplementedError("server-side tokenization is forbidden in production; portal uses Stripe.js")`. The error message is loud because anyone calling this is making a security mistake.
  - `charge()` calls `stripe.PaymentIntent.create(amount=..., currency=..., payment_method=token, customer=customer_external_ref, off_session=True, confirm=True, idempotency_key=idempotency_key)`. Records to `integrations.external_call`. Returns a normalized `ChargeResult` with Stripe's `decline_code` passed through.
  - `attach_payment_method_to_customer` calls `stripe.PaymentMethod.attach`.
  - `ensure_customer` upserts a Stripe customer keyed on `bss_customer_id` metadata; returns the `cus_*` id and stores it in `payment.customer_external_ref` (new column, see schema below).
- (Reserved) `AdyenTokenizerAdapter` — Protocol shape avoids Stripe-specific concepts (no `PaymentIntent` in the API surface) so v1.0+ portability is real. Not implemented in v0.16.

#### 1.2 Wiring

`services/payment/app/dependencies.py` constructs the adapter at lifespan startup via `select_tokenizer(os.environ["BSS_PAYMENT_PROVIDER"])`, hangs off `app.state.tokenizer`. `PaymentService.__init__` takes `tokenizer: TokenizerAdapter` as a constructor arg, replacing the bare module import at `payment_service.py:13`. Existing call sites at `payment_service.py:65` switch from `mock_tokenizer.charge(...)` to `self.tokenizer.charge(...)`.

#### 1.3 New env vars

```bash
BSS_PAYMENT_PROVIDER=stripe                 # mock|stripe
BSS_PAYMENT_STRIPE_API_KEY=sk_live_...
BSS_PAYMENT_STRIPE_PUBLISHABLE_KEY=pk_live_...
BSS_PAYMENT_STRIPE_WEBHOOK_SECRET=whsec_...
```

Naming follows the v0.14-locked convention (`BSS_<DOMAIN>_PROVIDER` + `BSS_<DOMAIN>_<NAME>_<KEY>`).

Startup guards:
- `BSS_PAYMENT_PROVIDER=stripe` + `BSS_ENV=production` + `BSS_PAYMENT_STRIPE_API_KEY=sk_test_*` → fail-fast with explicit message.
- Same for live-vs-test publishable key (`pk_live_*` vs `pk_test_*`).
- `BSS_PAYMENT_PROVIDER=stripe` without webhook secret → fail (webhook receiver would silently 401 every call otherwise).

#### 1.4 Schema additions

Alembic migration (`0017_v0160_payment_provider_columns.py` — assuming v0.15's migration was 0016):

```sql
ALTER TABLE payment.customer
    ADD COLUMN customer_external_ref text,         -- Stripe cus_*; NULL pre-cutover
    ADD COLUMN customer_external_ref_provider text; -- 'stripe' | NULL

ALTER TABLE payment.payment_method
    ADD COLUMN token_provider text NOT NULL DEFAULT 'mock';   -- 'mock' | 'stripe'
-- existing 'token' column now stores either tok_<uuid> (mock) or pm_<...> (stripe)

ALTER TABLE payment.payment_attempt
    ADD COLUMN provider_call_id text,               -- pi_*
    ADD COLUMN decline_code text;                   -- Stripe decline_code passed through
```

The `token_provider` column gives the lazy-fail cutover behavior something concrete to check: a mock-token row attempted under stripe-mode raises `mock_token_in_stripe_mode` cleanly.

### Track 2 — Stripe Elements portal rework

The most invasive single change in the three-release set. ~2 days of work; budget accordingly.

#### 2.1 Mode-switched COF template

`portals/self-serve/bss_self_serve/routes/signup.py` (around line 393+) and `templates/signup/cof.html`:

- `BSS_PAYMENT_PROVIDER=mock` → renders existing card-number form (dev affordance).
- `BSS_PAYMENT_PROVIDER=stripe` → renders Stripe Elements iframe; mounts Stripe.js (CDN-loaded directly per Stripe's docs — no bundler, no npm); on submit, JS posts the resulting `pm_...` id to the existing portal route; portal forwards to BSS payment service via the existing `bss-clients` path.

The mode switch is server-side template selection (a `payment_provider` variable passed from the route). HTMX integration is preserved — the form submit still hits the same HTMX endpoint; the difference is that in stripe-mode the form has no PAN field, only a hidden `payment_method_id` populated by Stripe.js.

#### 2.2 PCI scope guard (startup-time)

A startup self-check at portal lifespan:

- Walk all Jinja templates under `portals/self-serve/bss_self_serve/templates/`.
- Refuse to boot if any template contains `<input type="text" name="card_number">` (or close variants) AND `BSS_ENV=production` AND `BSS_PAYMENT_PROVIDER=stripe`.

The scan is a regex; false positives are acceptable (a template that mentions card_number in a comment will trip the scan, fix the comment). The scan is cheap (~few ms at boot) and exists explicitly so a future template edit doesn't silently regress PCI scope.

#### 2.3 `bss payment add-card` CLI dev command

The existing dev-only CLI at `cli/bss_cli/commands/payment.py` that calls `mock_tokenizer.tokenize_card` directly. v0.16 gates it on `BSS_PAYMENT_PROVIDER=mock`:

```
$ BSS_PAYMENT_PROVIDER=stripe bss payment add-card 4242...
ERROR: bss payment add-card is dev-only and requires BSS_PAYMENT_PROVIDER=mock.
       In stripe mode, customers add cards via the self-serve portal (Stripe Elements).
       For test data, use the portal with Stripe test cards (4242 4242 4242 4242).
```

The dev tokenizer affordance survives, mode-locked.

### Track 3 — Stripe webhook receiver

#### 3.1 New route

`POST /webhooks/stripe` in `services/payment/app/api/webhooks.py`. Verifies `Stripe-Signature` via `bss_webhooks.signatures.verify_signature(scheme="stripe", …)`. Persists into `integrations.webhook_event`.

#### 3.2 Event routing

Three event-type families, each with a documented behavior:

- **`charge.succeeded` / `charge.failed` / `payment_intent.payment_failed`** — mostly redundant with the synchronous charge result. Idempotently reconcile `payment_attempt` row state. If the row is already in the matching state, no-op + record `process_outcome='noop'`. If the webhook says success but the row says failed (or vice versa — should never happen but defend in depth), emit a `payment.attempt_state_drift` event for ops. Webhook is the *secondary* source of truth; the synchronous Stripe response from the `charge()` call is primary.
- **`charge.refunded`** — out-of-band refunds via Stripe dashboard. Emit `payment.refunded` domain event with the refund amount + reason. Operator cockpit surfaces it. **No automatic balance adjustment** — refund-driven balance changes are a v1.0 concern; v0.16 records the refund but doesn't reverse the bundle. (This stays inside motto #1: failed charge = block; out-of-band refund is operator-initiated, doesn't auto-restore service.)
- **`charge.dispute.created`** — chargeback. Persist webhook + emit `payment.dispute_opened` event. **No automatic case creation, no service block.** Operator handles via existing case workflow if they choose. Motto #1 is preserved: BSS-CLI does not do dunning, collections, or auto-action on payment-collection-adjacent events.

Webhook routes are exempt from `BSSApiTokenMiddleware` (already added in v0.14). Auth is provider signature only.

### Track 4 — Idempotency + cutover playbook

#### 4.1 Idempotency

`idempotency_key = f"ATT-{attempt_id}-r{retry_count}"`. Retry-count semantics:

- **Same key on BSS-crash-restart retry** — we want Stripe to dedupe and return the original outcome. Distinguishable by: `payment_attempt` row already exists with `status='charging'` and a recorded `idempotency_key`.
- **New key on user-initiated retry after UI-level failure** — we want a fresh charge attempt. Distinguishable by: a new `payment_attempt` row was created (the user re-clicked Pay).

The PaymentService's `attempt_charge` method computes the key once at attempt-row creation and records it on the row. Restarts re-read the recorded key. UI retries create a new row → new key.

Documented in `docs/runbooks/payment-idempotency.md` with the exact sequences.

#### 4.2 Cutover playbook

When an operator switches `BSS_PAYMENT_PROVIDER=mock` → `=stripe`, existing `payment_method.token` rows are mock format and unusable.

**Default behavior — lazy-fail.** Next charge attempt against a mock-token row in stripe-mode raises `mock_token_in_stripe_mode` (checked via the new `payment_method.token_provider` column). The error surfaces to the customer as "your saved payment method is no longer valid; please add a new one"; the existing self-serve "add COF" flow handles the rest. Honest with bundled-prepaid posture (failed charge = block).

**Optional behavior — proactive cutover.** New CLI:

```
bss payment cutover --invalidate-mock-tokens
  Found 47 active payment_methods with token_provider='mock'.
  Mark all as expired so customers re-add proactively? [y/N]
  ...
```

The CLI emits a customer-notification event per row so the operator's email-template flow (now Resend-backed, v0.14) can send a "please update your payment method" mailing.

Documented in `docs/runbooks/stripe-cutover.md`.

### Track 5 — Three-provider end-to-end soak

The "production-shape" gate. A hero scenario that runs end-to-end with all three real providers in sandbox mode:

```
BSS_PORTAL_EMAIL_PROVIDER=resend
BSS_PORTAL_KYC_PROVIDER=didit
BSS_PAYMENT_PROVIDER=stripe
BSS_ESIM_PROVIDER=sim                  # still simulator in v0.16
BSS_ENV=staging
```

Steps:
1. New customer signup via portal → Didit KYC sandbox → Stripe Elements COF → BSS issues subscription → Resend sends confirmation email.
2. Time-skip via `bss-clock` to renewal boundary → Stripe charge against the saved `pm_*` → webhook reconciles → Resend sends renewal receipt.
3. Trigger a Stripe test-mode dispute via `stripe trigger charge.dispute.created` → webhook records → cockpit surfaces it.
4. Out-of-band refund via Stripe dashboard → webhook records → cockpit surfaces it.

Runs nightly in CI (gated on `BSS_NIGHTLY_SANDBOX=true` + secrets injected from CI vault). Three consecutive green runs = soak pass. A flake → dig in *before* shipping; do not paper over.

This is the test that proves "v0.14 + v0.15 + v0.16 work together." Without it, the three-release split is a claim, not a proof.

## Doctrine and constraints

Inherited from v0.14 + v0.15:

1–10 (see those specs).

New in v0.16:

11. **PAN never touches BSS in production.** Stripe.js + Elements only when `BSS_PAYMENT_PROVIDER=stripe` + `BSS_ENV=production`. Server-side card-number forms forbidden in prod; startup template scan enforces.
12. **Webhooks are secondary source of truth for charge state.** The synchronous Stripe response from the `charge()` call is primary; webhooks reconcile and detect drift. A webhook saying success when the row says failed (or vice versa) is a `payment.attempt_state_drift` ops alert, not a state overwrite.
13. **No automatic action on chargebacks (`charge.dispute.created`).** Record-only. Operator handles via case workflow. Motto #1 (no dunning, no collections) stays intact.
14. **No automatic balance adjustment on out-of-band refunds.** Refund event is recorded; balance change is operator-initiated via existing tools. Bundle-prepaid posture unchanged.
15. **Hero scenarios pass three runs in BOTH mock-mode and live-sandbox-mode.** Mock-passes-prod-fails is a real-provider edge case the mock doesn't model — fix the mock, not the test.

### Anti-patterns added to CLAUDE.md

- **(v0.16+) Don't render server-side card-number inputs when `BSS_PAYMENT_PROVIDER=stripe` and `BSS_ENV=production`.** Startup template scan refuses to boot. PCI scope (SAQ A) depends on this. Greppable: in production deployments, `rg 'name="card_number"' portals/self-serve/.../templates/` must stay empty.
- **(v0.16+) Don't auto-open a case on `charge.dispute.created`.** Record-only. Motto #1: no dunning, no collections, no auto-action on payment-collection-adjacent events. The operator opens the case if they choose.
- **(v0.16+) Don't auto-adjust balances on out-of-band refunds.** v0.16 records the refund event; balance reversal is operator-initiated via existing tools. The product is bundled-prepaid; refund is an exception, not a flow.
- **(v0.16+) Don't reuse an idempotency key across user-initiated retries.** Same key only on BSS-crash-restart retries. Greppable: any code constructing `f"ATT-{x}-r0"` outside the attempt-creation path is wrong.
- **(v0.16+) Don't log raw Stripe responses.** Use `bss_webhooks.redaction.redact_provider_payload(provider="stripe", body=…)` — Stripe responses contain customer email, last4, billing address. The existing v0.14 doctrine entry covers this; v0.16 re-asserts because Stripe is the highest-volume call site.
- **(v0.16+) Don't call `mock_tokenizer.charge` directly from `payment_service.py`.** v0.16 retires that call site. All charges go through `self.tokenizer.charge(...)`. Greppable: `rg 'mock_tokenizer\.(tokenize|charge)' services/payment/app/services/` must match only test fixtures, never production code.
- **(v0.16+) Don't switch providers without a documented cutover.** A `payment_method.token` minted as a mock token is unusable when Stripe is selected. Run `bss payment cutover` or accept lazy-fail with documented customer-comms; do not flip the env var blind.

## Test strategy

- **Unit**: `TokenizerAdapter` Protocol + `MockTokenizerAdapter` + `StripeTokenizerAdapter` (with a faked `stripe` SDK + `respx` for HTTP); `select_tokenizer` for unknown-name, missing-key, env-mode mismatch (sk_test_* in production refusal).
- **Integration**: webhook receiver against fixture-signed Stripe payloads (`charge.succeeded`, `charge.failed`, `charge.refunded`, `charge.dispute.created`, plus a tampered-signature negative test); idempotency-key uniqueness logic; lazy-fail cutover behavior with mock-format tokens in stripe-mode.
- **Stripe sandbox test** (gated on env, skipped in CI by default; runnable nightly): full signup → Elements → COF → renewal-time charge → webhook reconcile, against Stripe test mode.
- **Three-provider end-to-end soak** (Track 5): nightly in CI when `BSS_NIGHTLY_SANDBOX=true`, three consecutive green runs gate the v0.16 release.
- **Hero scenarios — dual run.** Every existing v0.13/v0.14/v0.15 hero scenario runs twice in CI: once with all providers mock/prebaked/sim/logging, once with all providers set to live-sandbox. Both pass three runs each.
- **Migration test**: `payment_method.token_provider='mock'` → switch `BSS_PAYMENT_PROVIDER=stripe` → next charge fails cleanly with `mock_token_in_stripe_mode`; `bss payment cutover --invalidate-mock-tokens` works end-to-end.
- **PCI scope guard test**: a fixture template with `name="card_number"` + `BSS_ENV=production` + `BSS_PAYMENT_PROVIDER=stripe` → portal lifespan refuses to start.
- **Doctrine guards** added to `make doctrine-check`:
  - No `mock_tokenizer.(tokenize|charge)` calls outside test fixtures.
  - No raw Stripe response logging.
  - No code outside `select_tokenizer` reads `BSS_PAYMENT_PROVIDER`.

**Manual testing budget for v0.16: two rounds.** Round 1: Stripe Elements UI smoke + sandbox SetupIntent + sandbox charge + webhook receipt. Round 2: dispute simulation via `stripe trigger`, refund via dashboard, cutover CLI on a mock-data deployment.

## Out of scope

- **Bounce-suppression list (Resend).** Post-v0.16. v0.14 records bounces; acting on them is its own design exercise.
- **Refund-driven balance adjustment.** v0.16 records refunds; balance reversal stays operator-initiated.
- **Auto-case-open on chargeback.** Record-only.
- **Adyen / Checkout.com / alt-payment adapters.** Stripe only in v0.16. Adyen Protocol shape compatibility is preserved (no `PaymentIntent` in the API) but no impl ships.
- **DB-backed multi-tenant `tenant_config`.** Post-v0.16. v0.14–v0.16 all single-tenant.
- **Cloud-specific secret stores (AWS Secrets Manager, GCP Secret Manager, k8s sealed-secrets).** v0.14–v0.16 stay `.env`. The deploy story is host-agnostic by design — homelab and cloud are both first-class.
- **Customer-facing UI for viewing receipts / managing saved cards beyond the existing flow.** v0.16 reworks COF capture; the rest of the customer payment UX is unchanged.
- **Real eSIM SM-DP+ integration (1GLOBAL Connect, eSIM Access).** Still NDA-gated. Post-v0.16.

## The trap

**The trap of "swap a function call":** v0.16 looks like "replace `mock_tokenizer.charge` with `self.tokenizer.charge`" plus some env vars. It is not. The Stripe Elements portal rework is the single largest UI change in the BSS-CLI codebase since the v0.4 portal landed. Budget two days, not one. Stripe.js loads from a CDN and runs client-side; the server-side template has to mount it cleanly, the form has to submit a `payment_method_id` instead of a card number, the HTMX endpoint has to accept the new shape, and the existing mock-mode form has to keep working in dev. There is no "small version" of this change that ships safely.

**The second trap: webhook-as-primary-source-of-truth.** Stripe's webhook is at-least-once and runs on Stripe's clock. The synchronous response from the `charge()` call is the primary truth; webhooks reconcile and detect drift. A webhook handler that overwrites `payment_attempt.status` based on what the webhook says will eventually fight a sync response that already wrote a different status — and the result is non-deterministic depending on which write wins. Treat webhooks as "did the world agree with what we already recorded?", not "what did the world decide for us?"

**The third trap: idempotency keys.** Reuse on user retries → silent double-charge protection that hides real bugs. Fresh keys on BSS restart → silent double-charge. Get the "is this the same attempt or a new one?" check right via the `payment_attempt` row's existence and recorded key; do not trust caller intent.

**The fourth trap: motto #1 erosion via chargebacks.** It is genuinely tempting to auto-open a case when a chargeback fires — CSR ergonomics improve, and the chargeback is a real signal. Do not. Motto #1 says no dunning, no collections, no auto-action on payment-collection-adjacent events. Record-only. The operator opens the case if they choose. The version of BSS-CLI that auto-acts on chargebacks is a different product.

**The fifth trap: cutover.** A `payment_method.token` minted as a mock token is unusable when Stripe is selected. The first real-world cutover will not be the first time anyone thinks about this — that's why the runbook exists in v0.16 and not v0.17. Test the lazy-fail path. Test the proactive cutover CLI. Document the customer-comms sequence. The cost of a bad cutover is "every customer's next charge fails simultaneously"; the cost of a good cutover is "47 customers re-add their card over a week."

**The sixth trap: `BSS_ENV=production` is a doctrine, not a string.** Every guard in v0.16 (sk_test_* refusal, PCI template scan, prebaked-KYC opt-in from v0.15) hangs off this env var being set correctly. A staging deployment that says `BSS_ENV=development` will silently let test keys + card-number forms through. The fix is to set the env var honestly; the danger is shipping with a default that lets the guards no-op.

**The seventh trap: three-provider soak is not a checkbox.** Track 5 is a real test that catches real bugs only the integration of three real providers can produce: a Resend-bounce that races a Stripe-renewal-receipt, a Didit-cap-exhausted that hits during a peak signup window, a Stripe-webhook-delay that confuses the renewal-time idempotency key. Three consecutive green runs against all three sandboxes is the gate. A flake → dig in *before* tagging the release. Do not paper over.

Don't call this v1.0. v1.0 is when an MVNO has actually run this in production for 90 days, paying real money, charging real cards, sending real emails, and the only complaint is a Stripe rate-limit you didn't tune. v0.14 + v0.15 + v0.16 just give the seams to find out.
