# v0.5.0 — CSR agent console

> **The v0.4 pattern reflected back at the internal user.** A small FastAPI + Jinja + HTMX console that lets a CSR operator find a customer, open a 360 view, and ask the LLM agent to investigate and act. Reuses every piece of infrastructure v0.4 introduced (`portals/` workspace, agent log widget, SSE streaming, pre-baked auth). The new capability is operator-initiated, context-enriched LLM interaction: the operator types a question about a specific customer, the agent plans and executes tool calls in the customer's context, and the 360 view auto-refreshes to reflect the changes.

## Goal

v0.4 proved the portal-as-channel pattern for customer signup. v0.5 proves the same pattern for operators by building the CSR's workbench:

1. **Customer search and 360 view.** Find a customer by name or MSISDN. Open their profile. See subscriptions, cases with child tickets, payment methods, recent interactions. The 360 view is a re-skin of the existing ASCII `customer.py` renderer's data model; the layout is HTML but the content and structure are the same.
2. **"Ask the assistant about this customer."** The 360 view has a natural-language input box. The operator types a question — *"why is this customer's data not working?"*, *"top up 5GB to their plan"*, *"open a ticket about the missed renewal"* — and submits. The agent receives the question enriched with the customer's current state as context, plans a tool chain, executes it, and streams events into the same agent log widget v0.4 introduced. When the agent completes, the 360 view auto-refreshes to show any state changes.
3. **Interactions panel shows every action as it happens.** The hero scenario from v0.1 (`llm_troubleshoot_blocked_subscription.yaml` — blocked customer → LLM diagnoses → VAS purchase → unblock) now runs through this UI instead of the CLI. The operator watches the agent work. The audit trail shows every action attributed to `channel=portal-csr` (or `channel=llm` for agent-mediated actions, depending on how the channel header is wired — decision in §3).

Hard constraint inherited from v0.4: **every write goes through the LLM orchestrator.** The CSR cannot bypass the agent. The agent is the operator's interface to making changes. The reason is the same — it keeps the policy-layer chokepoint honest and amplifies the agent story. Reads can go direct.

Everything else is scaled to the same demo-grade ceiling as v0.4. Stub login that accepts any credentials. No real auth, no SSO, no role-based UI filtering, no ticket-routing engine, no SLA timers in the UI. One operator at a time.

### 3. Stub login — `routes/auth.py`

`GET /login`: renders a two-field form (username, password) with a note: *"Accepts any credentials in dev mode. This is a demo console — real auth ships in Phase 12."*

`POST /login`: accepts anything. On submit:
- Creates a session token (UUID) and stores it in the in-memory session store with `operator_id=<whatever username was typed>`
- Sets an HTTP-only cookie `bss_csr_session=<token>`
- Redirects to `/search`

`POST /logout`: clears the cookie and the session, redirects to `/login`.

Every other route checks the cookie and redirects to `/login` if missing. This is **not real auth** — it's a session mechanism for the demo, so the operator's name appears in the topbar and the agent prompt has an `operator_id` to attribute to.

**Channel attribution:** every outbound call from the CSR portal carries `X-BSS-Channel: portal-csr` and `X-BSS-Actor: <operator_id>`. The agent bridge overrides these per-call (the LLM's tool calls land with `channel=portal-csr` and `actor=<operator>`, not `channel=cli`). This is the audit trail the hero scenario asserts.

### 4. Search — `GET /search?q=...`

Simple LIKE query. The CRM service already has `customer.list` supporting `name_contains` filter (per TOOL_SURFACE.md). v0.5 adds an MSISDN lookup path: the CSR portal route tries both `customer.list(name_contains=q)` and `customer.find_by_msisdn(q)` (new tool if it doesn't exist — `crm` → `inventory.get_msisdn_assignment(msisdn)` → owning customer).

Results table:
- Customer ID | Name | Email | Phone | Active Subscription (count + state) | Last interaction
- Click a row → `/customer/{id}`

No fuzzy search, no Elasticsearch, no ranking. If the operator types a partial name and gets 20 results, they scroll. If the operator types an MSISDN and gets one result, they click it.

### 5. Customer 360 — `GET /customer/{id}`

Re-skin of `cli/bss_cli/renderers/customer.py`'s data model in HTML. Sections:

- **Top card** (`customer_summary.html`) — name, email, phone, customer state, KYC status, created-at. Re-fetched on auto-refresh.
- **Subscriptions** — cards for each subscription with state, plan, balance bars (data + voice), next renewal. Blocked subscriptions highlighted.
- **Cases & Tickets** — each case as a card with child ticket summary. Click a case → `/case/{id}` drill-in.
- **Payment methods** — list of cards on file with last-4 and brand.
- **Recent interactions** — table of last 20 interactions with timestamp, channel, type, description. Ordered newest first. Auto-refreshed when the agent completes.
- **Ask the assistant** (`ask_input.html`) — a single-line input + submit button. Placeholder: *"Ask about this customer…"*. Below the input, hint text: *"Examples: 'why is their data not working?', 'top up 5GB to their plan', 'open a technical ticket about the last outage'"*.

All data fetched via direct `bss-clients` reads. No agent involvement in rendering the 360. The agent only runs on the `ask` submit path.

### 6. Case thread — `GET /case/{id}`

Drill-in view for a single case. Shows:
- Case header (ID, subject, state, priority, assigned agent, created/updated timestamps)
- Child tickets list (ID, type, state, assigned agent)
- Notes timeline — most recent at the bottom, chronological like a chat thread

Each section is a `bss-clients` read. No writes in v0.5 case thread view — if the operator wants to add a note or transition a ticket, they do it via the "Ask the assistant" input on the parent 360 view: *"Add a note to CASE-042: 'followed up with provisioning team'"*.

### 7. `ask` endpoint — `POST /customer/{id}/ask`

The hero integration. Request body: `{question: "why is their data not working?"}`. Response: `303 See Other` to `/customer/{id}/progress?session=<session_id>` which opens the SSE stream.

`agent_bridge.ask_about_customer(customer_id, question, operator_id)`:

```python
async def ask_about_customer(customer_id: str, question: str, operator_id: str):
    # Fetch minimal context
    customer = await customer_client.get(customer_id)
    subscriptions = await subscription_client.list_for_customer(customer_id)

    prompt = build_csr_prompt(
        operator_id=operator_id,
        customer_id=customer_id,
        question=question,
        customer_snapshot={
            "name": customer.name,
            "email": customer.email,
            "state": customer.state,
            "kyc_status": customer.kyc_status,
        },
        subscription_snapshot=[
            {"id": s.id, "state": s.state, "offering": s.offering_id}
            for s in subscriptions
        ],
    )
    async for event in astream_once(
        prompt, allow_destructive=False, channel="portal-csr", actor=operator_id
    ):
        yield event
```

**Snapshot inclusion is deliberate.** Without it, the agent burns tool calls rediscovering what the operator already sees on screen. With it, the agent lands on the right subtree immediately — `subscription.get_balance` if the question is about data, `case.list` if it's about an existing issue.

Prompt template lives in `bss_csr/prompts.py`. It uses `KNOWN_LEADS` few-shot examples for the common patterns (blocked-subscription troubleshoot, top-up, open-ticket) so a small model stays on rails.

### 8. Auto-refresh after agent completion

When `astream_once` yields `AgentEventFinalMessage`, the SSE handler emits one additional frame with an HTMX `hx-trigger` event:

```
event: agent-complete
data: <div hx-get="/customer/{{ id }}/summary" hx-trigger="load delay:100ms" hx-target="#customer-summary" hx-swap="outerHTML"></div>
```

The client-side HTMX extension for SSE picks up the `agent-complete` event and inserts the div, which triggers the HTMX GET, which re-renders `customer_summary.html` and swaps it in place.

Sections that auto-refresh:
- `#customer-summary` — top card
- `#subscriptions-list` — subscription cards
- `#cases-list` — case cards
- `#interactions-list` — recent interactions (the key visible signal that the agent's actions landed)

All four fire in parallel. Any section where state didn't change renders the same HTML and the swap is a no-op visually.

### 9. Telemetry + auth + channel attribution

Same pattern as v0.4:

- `configure_telemetry(service_name="portal-csr")` in lifespan
- Portal's own HTTP surface is behind the stub login (cookie-gated); no `BSSApiTokenMiddleware` on the portal itself
- **Outbound** calls to services carry `BSS_API_TOKEN` via `TokenAuthProvider` (v0.3)
- `X-BSS-Channel: portal-csr` on every outbound call
- `X-BSS-Actor: <operator_id>` populated from the cookie session
- The agent's internal tool calls inherit these via the orchestrator's existing context-var plumbing

One wrinkle: **the agent's tool calls land in interaction logs with `channel=portal-csr, actor=<operator>`**, not `channel=llm`. The LLM is an execution mechanism, not an actor. Auditing "who did what" should attribute to the operator who asked the question. If forensic detail about *which model* executed the request matters, that's in `audit.domain_event` (richer actor field includes `llm-<model-slug>`) from Phase 10. Documented in DECISIONS.

## Doctrine and constraints

1. **Every write goes through the orchestrator.** Same as v0.4. The CSR portal's route handlers never import `SubscriptionClient.purchase_vas`, `CaseClient.open`, or any mutating client method. Every change is operator-asks-agent-agent-acts. Grep guard enforces.
2. **Reads go direct.** The 360 view and search pages use `bss-clients` read methods without LLM mediation. LLM-mediating a `customer.get` is wasteful and adds latency.
3. **Stub login, no real auth.** Any credentials work. Operator session is a cookie with a UUID. Lost on server restart. Real auth is Phase 12.
4. **Context-enriched prompts are non-negotiable.** The agent receives the customer snapshot with the question. Without it, a small model wastes 3-4 tool calls rediscovering visible-to-the-operator state.
5. **Agent actions attributed to the operator.** `X-BSS-Channel: portal-csr`, `X-BSS-Actor: <operator_id>`. Not `channel=llm`. See §9 + DECISIONS.
6. **Auto-refresh via HTMX, not client-side state management.** When the agent completes, the SSE stream emits an event that triggers a GET-and-swap on the 360 sections. No Redux, no Vuex, no client-side reactivity.
7. **One operator at a time.** Concurrent operator sessions are technically supported (session store is keyed by cookie) but untested and uninteresting for a demo. If two operators ask about the same customer simultaneously, the agent handles whichever call came in first and the other waits — fine for v0.5.
8. **Footprint stays under budget.** CSR portal ~150 MB (same as self-serve). Bundled-infra total: ~3.35 GB. Still under 4 GB.
9. **Campaign OS untouched.** Grep guard.
10. **The demo is the hero scenario.** If the hero scenario passes three runs in a row, v0.5 is shippable. If it flakes, the LLM path is not reliable enough and v0.5 holds.

## Security model

v0.5 adds an internal operator console. The threat model is *stricter* than v0.4 because the CSR portal routes prompts on behalf of *any* customer in the database, not just a signup flow.

**What the agent can do for any CSR prompt.** Every tool in the registry except the destructive ones (v0.1 gate). The agent can open cases, add ticket notes, purchase VAS on a customer's behalf, resolve tickets, update contact mediums, issue refunds (if a refund tool is ever added) — on *any* customer the portal's search can find. That's every customer in the database. Combined with CRM's auto-logged interaction trail, a prompt submitted through the CSR portal is effectively an admin action against a live account.

**Who can invoke the agent through the CSR portal.** Anyone who can reach port 9002 and POST credentials to `/login`. The stub login accepts any username + any password.

**The stub login is NOT a security control.** It exists to populate `X-BSS-Actor` with an operator ID so the audit trail has something to attribute to. It is a UX mechanism, not a lock. Anyone who can reach port 9002 bypasses it by submitting the form.

**What actually keeps the CSR portal safe.** Network-level exposure control, same as v0.4, but the stakes are higher. A leaked URL + a publicly-reachable port 9002 is equivalent to an open admin console against the entire customer base. The v0.3 `BSS_API_TOKEN` is irrelevant here — the attacker never needs it because the agent carries it on their behalf.

**What's deferred to Phase 12.**

- Real CSR auth — OAuth via Keycloak / Cognito / Entra, session lifetime, session revocation
- Role-based tool gating (senior_agent vs csr vs billing_analyst — permissions derived 1:1 from tool names per CLAUDE.md)
- Per-operator rate limits on agent prompts
- Suspicious-prompt alerting ("an operator just asked to purchase VAS for 100 customers in 5 minutes")
- Explicit human-in-loop confirmation on destructive actions even through the agent
- Audit on login attempts (not in scope while the login is stub)

**What you must NOT do before Phase 12:**

- Do not publish port 9002 on a public IP, ever.
- Do not allow the CSR portal to run on any network that isn't operator-trusted (Tailscale / VPN / ops LAN).
- Do not remove the `allow_destructive=False` pin on the CSR `agent_bridge.ask_about_customer`. It's the last guardrail while the door is unlocked.
- Do not ship demo recordings or LinkedIn screenshots that include the actual portal URL if the port is published non-locally.
- Do not assume "it's just a demo" is a defense — a demo instance with prod-shaped customer data is a data breach waiting to happen.

**The right mental model:** v0.5's CSR portal is a privileged admin console on a trusted network. Treat it like your database credentials — never exposed beyond the operator subnet.

## Out of scope

- **Real auth.** Stub login only. Phase 12 adds real CSR auth (likely Keycloak or Cognito).
- **Role-based UI filtering.** All operators see all sections. Senior-agent-only actions, auditor-read-only, etc. — Phase 12.
- **Ticket-routing engine, queue management.** v0.1's ticket model supports single assignee; real CSR workbenches have skill-based routing, round-robin queues, escalation timers — none of that here.
- **SLA countdown timers in the UI.** The data is in `crm.case.due_by` — render if trivial, don't if not.
- **Multi-operator concurrent sessions.** Technically supported, untested, not a v0.5 concern.
- **Bulk actions across customers.** Nothing that iterates over many customers.
- **Knowledge-base integration (RAG over runbooks).** That's Phase 11's `knowledge.search` tool — if it exists by v0.5, expose it to the agent; don't build UI for it.
- **Case creation / ticket transitions via direct form.** The agent does all writes. The operator's surface for state changes is the ask box.

## The trap

**Do not build the second portal before extracting the shared pieces from the first.** Copy-paste the agent log widget and SSE plumbing from `portals/self-serve/` into `portals/csr/` and you've guaranteed they'll drift. Every subsequent fix needs to land in both places, half the time only lands in one, and the demo bug reports start a month later. Extract `bss-portal-ui` on day one — before CSR exists as a directory.

**Do not let the 360 view call a mutating tool.** Every time the operator submits the ask form, the agent is the only thing that writes. The temptation to add an "Open Ticket" button on the case drill-in that calls `CaseClient.open` directly is real — resist it. If tickets need opening from the UI, the operator asks the agent: *"open a technical ticket about the missed renewal"*.

**Do not send the agent a bare question.** A small model like MiMo v2 Flash given *"why is this customer's data not working?"* with no context will start with `customer.list(name_contains="")` and hallucinate an ID. With the customer snapshot pre-fed, it goes straight to `subscription.get_balance` and the tool chain is predictable. The prompt builder is where model reliability lives or dies.

**Do not attribute agent actions to `channel=llm`.** The audit surface is "what did this operator cause to happen", and the operator is human. The LLM is the execution layer. Forensic per-model attribution is in `audit.domain_event.actor` (`llm-<slug>`) — that's the right place for it. Collapsing "who did this" with "what executed it" in the interaction log makes the CSR view less useful, not more.

**Do not use client-side state management to handle the auto-refresh.** The HTMX SSE extension dispatches custom events; `hx-trigger="sse:agent-complete"` is the one-line pattern. Reaching for React Context / Redux / signals is how a demo becomes a framework commitment.

**Do not hide the ask input behind tabs, accordions, or modals.** It's the interface. It sits front and center on the 360 view, below the customer summary, above the subscriptions section. The viewer scanning the screenshot should find it in under two seconds.

**Do not forget the hero scenario three-in-a-row gate.** The CSR portal wraps the flakiest v0.1 hero scenario (LLM diagnosing a blocked subscription) in an additional UI layer. If anything's flaky, it'll flake here first. Ship when three pass. Not before.
