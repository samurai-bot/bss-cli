# v0.9.0 — Per-portal service identity (named tokens at the BSS perimeter)

> **The smallest possible blast-radius reduction.** v0.3 introduced `BSS_API_TOKEN`, a single shared secret carried by every internal caller. v0.9 splits it: the customer-facing self-serve portal gets `BSS_PORTAL_API_TOKEN`, a separately-named identity that the BSS perimeter middleware validates on its own. A leaked portal credential no longer grants orchestrator-level access. `service_identity` flows into `auth_context` and `audit.domain_event` so every write is answerable to "which surface initiated this." No OAuth, no JWT — Phase 12 still owns those. v0.9 is the bridge.

## Goal

Three deliverables, each small, that together make the perimeter aware of *which* internal process is calling, not just whether *some* internal process is.

1. **`bss-middleware` accepts a named-token map.** `BSSApiTokenMiddleware` is upgraded to validate the request token against multiple env-loaded secrets, each tied to a `service_identity` string. Hits attach the identity to `request.state` and `auth_context`. Misses still return 401. Backwards compatible — single-token deployments work unchanged.
2. **`bss-clients` can carry a named identity outbound.** A new `NamedTokenAuthProvider` reads its token from a configurable env var. The orchestrator and CSR console keep using `BSS_API_TOKEN` (default identity). The self-serve portal builds its clients with `NamedTokenAuthProvider("portal_self_serve", "BSS_PORTAL_API_TOKEN")`. The portal's chat surface — which calls the orchestrator's `astream_once` — propagates the identity into the agent's downstream tool calls.
3. **`service_identity` flows into audit + logs.** Every `audit.domain_event` row carries the resolved identity (header-untrusted; resolved from the validated token map). `structlog` request-context processor attaches it. Operators can answer "which surface did this write?" by reading the audit log.

When v0.9 ships, the perimeter has multiple named identities. A compromised portal token rotates independently of the orchestrator's token. Audit gains a critical dimension. None of the business logic changes.

## Deliverables

### Track 1 — `bss-middleware` named-token map

#### 1.1 Token map model

`packages/bss-middleware/bss_middleware/api_token.py`:

- New `TokenMap` dataclass: `{ token_value (hashed): service_identity (str) }`. Loaded once at app startup; immutable thereafter.
- Loader function `load_token_map_from_env() -> TokenMap`:
  - Reads `BSS_API_TOKEN` (the existing default — identity `"default"`).
  - Reads any env var matching `BSS_*_API_TOKEN` (the named token pattern). Each non-empty value becomes an entry; identity is derived from the env-var name (`BSS_PORTAL_API_TOKEN` → `"portal"`, `BSS_CSR_API_TOKEN` → `"csr"`, `BSS_PARTNER_API_TOKEN_ACME` → `"partner_acme"`).
  - Each token must be ≥32 chars; sentinel `"changeme"` rejected. Rules unchanged from v0.3.
  - Tokens hashed via HMAC-SHA-256 with a fixed salt (so the map can be safely logged at debug level for diagnosis without leaking secrets).
- Startup validator `validate_token_map(token_map)`:
  - At least the `"default"` identity must be present (this preserves v0.3 behaviour for deployments that don't enable named tokens).
  - All named identities (if any) must be unique.
  - All tokens must be unique across the map (two identities can't share a token).
  - Failures raise `TokenMapInvalid` with a clear message; service refuses to boot.

#### 1.2 Middleware upgrade

`BSSApiTokenMiddleware` (existing pure-ASGI middleware):

- Constructed with a `TokenMap` instead of a single token.
- On each request: read `X-BSS-API-Token` header → hash with the same salt → lookup in map.
- Hit: attach `request.scope["service_identity"] = <identity>` and proceed. Add `auth_context` enrichment hook (Track 2).
- Miss: return 401 with the existing structured error shape — unchanged from v0.3.
- Exempt paths unchanged (`/health`, `/health/ready`, `/health/live`).
- Comparison is `hmac.compare_digest` against the hashed token map. (Hashing the env tokens once at startup avoids timing-leaking the env values themselves; comparison is still constant-time.)

#### 1.3 Backwards compat

If only `BSS_API_TOKEN` is set, the loader produces a single-entry map (`{token_hash: "default"}`). Behaviour is identical to v0.3 except that `service_identity = "default"` is now present in `request.state` and `auth_context`. Existing v0.3 tests pass without modification.

#### 1.4 `validate_api_token_present()` rename + alias

The v0.3 startup helper is renamed to `validate_token_map_present()` and gains the validation logic from §1.1. The old name is kept as an alias (deprecation warning logged) so existing service `main.py` files don't break in mid-deploy. Both names removed in v1.0.

### Track 2 — `auth_context` enrichment

#### 2.1 `auth_context.current()` shape

Every service has its own `auth_context.py`. The current shape (v0.3) returns a struct with `principal`, `tenant`, `roles`. v0.9 adds `service_identity`:

```python
@dataclass
class AuthContext:
    principal: str          # "admin" today; per-principal in Phase 12
    tenant: str             # "DEFAULT" today
    roles: list[str]        # ["admin"] today
    service_identity: str   # "default" / "portal" / "csr" / "partner_<name>" — v0.9 new
    actor: str | None       # X-BSS-Actor header (v0.x existing — customer or operator id)
    channel: str | None     # X-BSS-Channel header (v0.x existing — "cli", "portal_self_serve", etc.)
```

The `service_identity` field is populated by the request-context middleware from `request.scope["service_identity"]` (set by `BSSApiTokenMiddleware`). Never populated from a header — it's authoritative because it comes from token validation.

#### 2.2 RequestIdMiddleware update

The middleware that already plumbs `actor` / `channel` into `auth_context` from headers also reads `request.scope["service_identity"]` and stamps it. This requires the middleware order: `BSSApiTokenMiddleware` runs first (sets the scope key), then `RequestIdMiddleware` (consumes it for auth_context).

#### 2.3 Per-service updates

Each of the 9 BSS services + 2 portals receives a small update to their `auth_context.py` and middleware wiring:

- Add `service_identity` to the `AuthContext` dataclass.
- Update the request-context middleware to read it.
- Default to `"default"` if for any reason it's missing (e.g., test direct-instantiation paths that bypass middleware).

This is a mechanical sweep. The CSR console and self-serve portals also pick up the field, even though only the customer-facing one will populate it from anything other than `"default"` in v0.9.

### Track 3 — `bss-clients` named outbound

#### 3.1 `NamedTokenAuthProvider`

`packages/bss-clients/bss_clients/auth.py`:

- New class alongside the existing `NoAuthProvider` and `TokenAuthProvider`:
  ```python
  class NamedTokenAuthProvider(AuthProvider):
      def __init__(self, identity: str, env_var: str):
          self._identity = identity
          self._env_var = env_var
          # token loaded lazily on first use; cached
      def headers(self) -> dict[str, str]:
          return {"X-BSS-API-Token": self._token()}
      def _token(self) -> str:
          # read from os.environ; cache; raise if missing
  ```
- Identity is informational (used for log fields, not header-stamped — the receiving service derives identity from token validation).

#### 3.2 Self-serve portal client construction

New module `portals/self-serve/bss_self_serve/clients.py`:

- `get_clients()` builds the standard `BSSClients` bundle (Customer, Subscription, Catalog, etc.) with `NamedTokenAuthProvider("portal_self_serve", "BSS_PORTAL_API_TOKEN")`.
- Used by every direct `bss-clients` call in the portal. (As of v0.8 the only direct calls from the portal are reads — for the dashboard placeholder, plans browse, etc. v0.10 expands this.)
- Imported in `portals/self-serve/bss_self_serve/main.py` lifespan; passed to routes via dependency injection (mirrors the orchestrator pattern).

#### 3.3 Orchestrator + CSR console unchanged at the perimeter

- Orchestrator's `bss-clients` continue using `TokenAuthProvider` against `BSS_API_TOKEN`. Identity = `"default"`.
- CSR console likewise. v0.12 is when CSR gets its own named token (`BSS_CSR_API_TOKEN`).

#### 3.4 `astream_once` identity propagation

`orchestrator/bss_orchestrator/agent_bridge.py`:

- `astream_once(question, ..., service_identity: str | None = None)` — new optional parameter.
- When set, the agent's downstream tool calls (which go through `bss-clients`) carry that identity instead of the orchestrator's default. Implementation: the parameter populates a `ContextVar` that `bss-clients` reads when constructing the per-call `headers()` dict. Without the var set, `bss-clients` falls back to whatever provider the client was constructed with (default behaviour, i.e., `BSS_API_TOKEN`).
- The self-serve portal's chat handler in v0.11 will pass `service_identity="portal_self_serve"` so chat-driven writes are auditable to the portal token, not the orchestrator's.
- v0.9 ships the parameter and the propagation but doesn't yet use it from the portal — the chat surface is wired up in v0.11.

## Doctrine and constraints

1. **Each external-facing surface gets its own named token.** Self-serve portal in v0.9. CSR console in v0.12. Future partner integrations get their own. Sharing one token across surfaces defeats the blast-radius point.
2. **`service_identity` is authoritative; not header-trusted.** Resolved from the validated token map. Adding a separate `X-BSS-Service-Identity` header is the wrong pattern — it would let the caller assert any identity.
3. **No OAuth, no JWT in v0.9.** Phase 12 owns those. v0.9 is the bridge that makes Phase 12 a per-service middleware swap, not a business-logic rewrite.
4. **Tokens loaded at startup, not at request time.** Cached. Rotation is restart-based. Per-request env reads are slow and hard to observe.
5. **Hashed token map.** The in-memory map stores HMAC-SHA-256 hashes of tokens, not the raw values. This lets ops log the map at debug level without leaking, and aligns with how passwords/peppered-secrets are typically handled.
6. **At least one named identity must be present.** `BSS_API_TOKEN` is required (the `"default"` identity). Named tokens (`BSS_PORTAL_API_TOKEN` etc.) are optional — without them, v0.9 behaviour reduces to v0.3 plus the `service_identity = "default"` enrichment.
7. **Backwards compatibility.** A single-token deployment (only `BSS_API_TOKEN` set) works unchanged. Existing v0.3 / v0.6 tests pass without modification.
8. **Per-portal `bss-clients` instances.** Each portal builds its own `BSSClients` bundle with the right `NamedTokenAuthProvider`. No shared/global clients across portals.
9. **`astream_once(service_identity=...)` is opt-in.** v0.9 ships the parameter; v0.11 portal chat surface uses it. CLI scenarios and CSR console don't pass it (default behaviour preserved).
10. **`bss_clock.now()` for everything.** v0.6 doctrine guard still passes.

   - Missing env var → raises at first use.
   - Caches token after first read.

## Out of scope

- **OAuth2 / JWT.** Phase 12 territory. v0.9 is the bridge.
- **Per-principal authentication.** v0.9 doesn't change the principal/role model. `auth_context.principal` remains `"admin"` for the default identity. Per-principal auth is Phase 12.
- **Customer-facing token (browser-side).** `BSS_PORTAL_API_TOKEN` is server-side only. The customer's browser holds a session cookie (v0.8); the portal server holds the BSS token. Never expose the BSS token to the browser.
- **Token rotation automation.** v0.9 is restart-based. Auto-rotation (e.g., every 30 days, with overlap windows) is a v1.x story.
- **CSR console named token.** Stays on `BSS_API_TOKEN` in v0.9. v0.12 picks this up alongside CSR auth.
- **Per-route token scoping.** Named tokens identify a surface, not a permission set. A leaked portal token still grants full BSS read/write access — it just leaves a different audit trail. Per-permission scoping is Phase 12.
- **Rate limiting per named token.** No per-identity rate limits at the BSS perimeter in v0.9. The portal's own customer-facing rate limits (login, chat) live in `bss-portal-auth`.
- **External / partner API tokens.** The pattern is in place (`BSS_PARTNER_API_TOKEN_<NAME>`), but no actual partner integration ships in v0.9. v1.x.
- **Token revocation list.** Rotation = restart, no persistent revocation list. v0.9 scale doesn't need it; Phase 12 will via JWT `kid` rotation.
- **Audit row per failed token attempt.** The 401 path emits a structlog warning (existing behaviour). It does not write to `audit.domain_event` — that's reserved for successful business-domain events. A failed-auth audit table can come later if ops needs it.

## The trap

**Do not trust `X-BSS-Service-Identity`.** The whole point of named tokens is that identity is *derived* from successful token validation, not asserted by the caller. If a deliverable starts adding "let the portal pass its own identity in a header for clarity," stop — that header is forgeable, the validated map is not.

**Do not read tokens at request time.** Even with caching, every `os.environ.get("BSS_PORTAL_API_TOKEN")` per request is a code smell — it implies the token might change between requests, which it can't (rotation is restart-based). Load once, cache, done.

**Do not let one named token cover multiple surfaces.** It's tempting because "the portal and the orchestrator are both internal." They're not equivalent — the portal is reachable from the public internet (via reverse proxy), the orchestrator is not. Distinct tokens = distinct rotation cadences and distinct blast radii.

**Do not shortcut the audit migration with a NULL-default column.** Backfilling existing rows to `'default'` is the right call (rationale: that's what they were under the v0.3 single-token regime), but it has to be in the migration with NOT NULL applied after, not "we'll backfill in v0.10." Same trap as the v0.7 price snapshot.

**Do not let the hashed-map abstraction leak through to callers.** The middleware's internal storage being hashed is a security detail, not an interface. Callers that need to know "did this token validate?" go through the middleware response (200/401), not by inspecting the map. No `is_valid_token(value)` helper exposed.

**Do not bypass the middleware in tests.** It's tempting to write tests that construct `AuthContext` directly and skip the middleware. Do that for unit tests. Integration tests must round-trip through the actual middleware so the token-resolution path is exercised — otherwise you have green tests against a perimeter that never sees real traffic.

**Do not add named tokens for things that aren't external surfaces.** Internal per-service identities ("billing service identifies itself differently from rating") are a *role* concern, not a *named token* concern. Phase 12 handles that with JWT claims. v0.9 is for distinguishing externally-addressable surfaces.

**Do not let the deprecation alias `validate_api_token_present` linger past v1.0.** Add it in v0.9 to avoid breaking deploys mid-rollout, but the v1.0 release notes should explicitly retire it. Don't carry forever-aliases.

**Do not ship without rotation runbook walked through.** A theoretical rotation procedure that's never been exercised is a runbook that breaks the first time it's needed. Either run a real rotation against a dev stack or pair-walk the procedure with someone unfamiliar with it; whichever the team does, record the outcome in the runbook.
