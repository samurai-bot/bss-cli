# V1.6.0 — Cockpit CRM workbench

**Status:** shipped (2026-06-10)
**Branch:** `feature/v1.6-cockpit-crm`
**Decision record:** DECISIONS 2026-06-10 (Phase 0 amendment, operator-directed)

## Why

The v0.13 browser was "a thin veneer over the Conversation store" by
doctrine. The conversational write path earned its keep (v1.5 multi-step
autonomy, propose-then-`/confirm`, policy gating) — but the *read*
ergonomics didn't: triaging a case queue, scanning a customer 360, or
comparing catalog rows one chat bubble at a time is slower than a table.
Operator directive: make the cockpit a full CRM — chat stays the
centrepiece, UI supplements it.

## The three rules

1. **Reads are screens.** Direct `bss-clients` reads via the
   orchestrator's `get_clients()` bundle; every 360 section degrades
   independently (one service down ≠ blank page).
2. **Routine writes are policy-gated forms.** Non-destructive CRM verbs
   only — log interaction, open case, add note, case transitions
   (take/await/resume/resolve), priority, open/assign/resolve/close
   ticket. One route → one `bss-clients` call; `PolicyViolation`
   flashes back verbatim.
3. **Destructive / money-moving / compound verbs hand off to chat.**
   `DESTRUCTIVE_TOOLS` verbs + charges (order submit, VAS, renew-now)
   render as "Ask the agent" buttons → `POST /cockpit/handoff` opens a
   session pinned to the customer with the verb **drafted, never
   auto-sent**. Doctrine pinned by
   `portals/csr/tests/test_routes_crm.py::test_crm_routes_never_call_destructive_clients`.

## Bugs fixed en route (workbench surfaced them)

- CRM `PATCH /case/{id}` deserialized triggers only → bss-clients
  `update_case_priority` (and the `case.update_priority` tool) 422'd
  since v0.13. PATCH now takes trigger and/or priority/category;
  field updates policy-gated (`case.update.case_is_closed`,
  `case.update.invalid_priority`) + `case.updated` event.
- `CRMClient.list_cases` sent `agentId`; service reads
  `assignedAgentId` — filter silently ignored. Fixed; `limit`/`offset`
  added.
- `CRMClient.transition_case` couldn't express `resume`
  (pending_customer → in_progress collides with `take` in the
  to_state map). Direct `trigger=` override added.
- v0.13 case page read TMF camelCase off the snake_case Case DTO —
  customer/opened-at silently blank. `bss_csr.views.field` reads both
  spellings everywhere now.
- COM `GET /productOrder` required `customerId`; now optional with
  `state`/`limit`/`offset`, newest-first (motto #7 — reads are free).
- Ticket client surface had drifted wholesale from the CRM service
  (live smoke caught it; mocks couldn't): `open_ticket` omitted the
  required `customerId`/`caseId` fields (422 always), `assign_ticket`
  sent `assignedAgent` instead of `assignedToAgentId` (silent no-op),
  `transition_ticket` sent `toState` instead of `trigger` (422
  always). Fixed with a ticket state→trigger map; `in_progress`
  disambiguates via one `get_ticket` read (start/resume/reopen).

## v1.6.1 — same-day operator review amendments

Operator feedback on the v1.6.0 build reversed rule 3 and forced a UI
overhaul (DECISIONS 2026-06-10 amendment #2):

1. **Direct CRUD restored.** Orders create/submit/cancel, catalog
   add-offering/add-price/set-window, customer name + contact CRUD +
   close, subscription plan-change/renew/VAS/terminate, case close,
   ticket cancel — all direct policy-gated writes. Destructive and
   money-moving verbs sit behind a **two-step UI confirm**
   (`crm-danger-form` → `confirm=yes`; routes refuse without it;
   test-pinned both directions). Handoff buttons remain as the chat
   seam; promo assignment stays conversational (loyalty pairing).
2. **Fixed-viewport app shell.** `body` = `100dvh` flex column,
   `overflow: hidden`; `.cockpit-main` / thread stream are the only
   scroll panes. Kills the iPad "scroll to reach the compose box"
   defect — the old `calc(100vh - 53px)` math broke when the header
   wrapped and under Safari's dynamic toolbars.
3. **Design system.** Accent links everywhere (no default blue on the
   dark palette), panel cards with small-caps labels, three button
   weights + danger, dot badges, two-column detail grids
   (`crm-grid`), dense tables with mono identifiers.
