# v0.13.0 — Operator Cockpit (REPL canonical, browser veneer)

> **The release that retires the v0.5 CSR portal pattern and replaces it with a unified, conversational, single-operator cockpit.** v0.5 reflected the v0.4 customer-portal pattern back at the CSR: 360 view + ask-form + 4 HTMX auto-refresh partials + stub login. v0.13 throws out the form-shaped UI and the staff-auth shim. The CLI REPL becomes the canonical CSR cockpit, the browser becomes a thin veneer over the same shared session, and operator identity collapses to a single value in `settings.toml`. No OAuth, no RBAC, no Phase-12 staff-auth ambition — the stack runs behind a secure perimeter for a 1–3 person team. Inspired by the Claude Code shape: slash-command-driven, conversational, minimal UI, operator-customizable persona via an `OPERATOR.md` that prepends to the system prompt every turn.

## Goal

Five deliverables, plus a doctrine sweep that retires the staff-auth ambition explicitly:

1. **One conversation model, two surfaces.** A new `bss-cockpit` package owns a Postgres-backed `Conversation` store keyed by `session_id`. The CLI REPL and the browser both read/write the same store. Exit `bss`, resume in the browser, see the same turns. Multi-session via `/sessions` + `/switch`. `/reset` clears one session; `/new` starts another.
2. **REPL becomes the canonical CSR cockpit.** Today's `Session.astream` (in-memory, process-local) is replaced with `astream_once(transcript=conversation.transcript_text(), ...)`, persisting via the new store. New slash commands: `/sessions`, `/switch`, `/new`, `/focus CUST-NNN`, `/360`, `/confirm`, `/config edit`, `/operator edit`. `bss --session <id>` resumes; `bss --new` forces fresh.
3. **CSR portal collapses to a thin browser veneer.** Login, OperatorSessionStore, search route, customer 360 route + 4 auto-refresh partials, ask-form, AgentAskStore — all retired. New layout: sessions index + chat thread + minimal header. The chat thread mirrors `portals/self-serve/.../routes/chat.py` shape (htmx-sse, SSE pills) but binds to the shared `Conversation` store. Customer 360 is rendered *inside* the chat thread on demand via `/360` or natural-language ask, using the existing `cli/bss_cli/renderers/customer.py:render_customer_360()`.
4. **Operator config split: `OPERATOR.md` + `settings.toml`.** `.bss-cli/OPERATOR.md` is human-written persona + house rules (operator name in narrative, escalation rules, currency defaults, response style) — automatically prepended to every system prompt. `.bss-cli/settings.toml` is machine config: `actor` (audit name), model, allow_destructive_default, ports, dev service URLs. Hot-reloaded on mtime change. Editable from REPL slash commands and a WebUI settings page.
5. **`operator_cockpit` tool profile + `BSS_OPERATOR_COCKPIT_API_TOKEN`.** New explicit tool profile (full registry, but greppable + auditable, no `*.mine` wrappers — operator is the system). New named token at the v0.9 perimeter; `audit.domain_event.service_identity = "operator_cockpit"` cleanly distinguishes operator action from customer action and from scenario action.

When v0.13 ships, "the CSR portal" means: open `bss`, type a question, watch ASCII flow. Or open `localhost:9002/cockpit/<id>` and see the same turns in a browser. There is no login. There is one operator (or three, sharing the box). Every cockpit action attributes to `actor=<settings.actor>`, `service_identity="operator_cockpit"` in the audit trail. v0.5's CSR portal is dead and good riddance.

#### 1.2 Schema (alembic migration)

```sql
CREATE SCHEMA cockpit;

CREATE TABLE cockpit.session (
    id text PRIMARY KEY,                     -- e.g. SES-20260430-1a2b
    actor text NOT NULL,                     -- from settings.toml at session creation
    customer_focus text NULL,                -- pinned CUST-* or NULL
    allow_destructive boolean NOT NULL DEFAULT false,
    state text NOT NULL DEFAULT 'active',    -- 'active' | 'closed'
    started_at timestamptz NOT NULL DEFAULT now(),
    last_active_at timestamptz NOT NULL DEFAULT now(),
    label text NULL                          -- human label, optional
);

CREATE TABLE cockpit.message (
    id bigserial PRIMARY KEY,
    session_id text NOT NULL REFERENCES cockpit.session(id) ON DELETE CASCADE,
    role text NOT NULL,                      -- 'user' | 'assistant' | 'tool'
    content text NOT NULL,
    tool_calls_json jsonb NULL,
    created_at timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX ix_cockpit_message_session_created ON cockpit.message(session_id, created_at);
CREATE INDEX ix_cockpit_session_actor_active ON cockpit.session(actor, last_active_at) WHERE state = 'active';

CREATE TABLE cockpit.pending_destructive (
    session_id text PRIMARY KEY REFERENCES cockpit.session(id) ON DELETE CASCADE,
    proposed_at timestamptz NOT NULL DEFAULT now(),
    tool_name text NOT NULL,
    tool_args_json jsonb NOT NULL,
    proposal_message_id bigint NOT NULL REFERENCES cockpit.message(id)
);
```

#### 1.3 `Conversation` API

```python
class Conversation:
    @classmethod
    async def open(cls, actor: str, label: str | None = None,
                   customer_focus: str | None = None) -> "Conversation": ...
    @classmethod
    async def resume(cls, session_id: str) -> "Conversation": ...
    @classmethod
    async def list_for(cls, actor: str, *, active_only: bool = True) -> list[ConversationSummary]: ...

    async def append_user_turn(self, content: str) -> None: ...
    async def append_assistant_turn(self, events: list[AgentEvent]) -> None: ...
    async def transcript_text(self) -> str: ...   # for astream_once(transcript=...)
    async def reset(self) -> None: ...            # clears messages, keeps session row
    async def close(self) -> None: ...

    async def set_pending_destructive(self, tool_name: str, args: dict, proposal_message_id: int) -> None: ...
    async def consume_pending_destructive(self) -> PendingDestructive | None: ...
    async def set_focus(self, customer_id: str | None) -> None: ...
```

The store is a single `ConversationStore` singleton hung off `app.state` in each consumer (REPL: lazy module-level; portal: lifespan).

## Doctrine and constraints

1. **Single Conversation store.** Cockpit conversations live in `cockpit.session` / `cockpit.message`. REPL and portal both consume; neither reimplements. Customer chat (`chat_session.py`) keeps its own store — different scoping concern, different lifecycle.
2. **No staff auth.** No login route on the CSR portal. No Phase-12 ambition. `actor` comes from `settings.toml`, descriptive only.
3. **Operator persona is markdown.** `OPERATOR.md` is the operator's editable contract with the agent. Always prepended to the system prompt. Hot-reloaded.
4. **Settings file, not env, for operator preference.** `BSS_*` env vars stay for secrets and infra. `settings.toml` is for operator-tunable values that change without restart.
5. **`/confirm` for destructive.** Pending_destructive row is the contract; system prompt reinforces propose-then-confirm. Per-turn `allow_destructive=True` only when consuming a pending row.
6. **`operator_cockpit` profile is a coverage assertion.** Adding a tool requires explicit profile inclusion; otherwise cockpit can't see it. Startup self-check enforces.
7. **Named token at perimeter.** `BSS_OPERATOR_COCKPIT_API_TOKEN`; identity `operator_cockpit`; audit cleanly distinguishes operator action from customer / partner / scenario action.
8. **REPL canonical, browser mirrors.** Slash command parity is a doctrine target. If REPL has `/focus`, browser exposes it (button or input). If browser has a UI affordance, REPL has the slash equivalent.
9. **Customer chat unchanged.** v0.12's `customer_self_serve` profile + `*.mine` wrappers + customer chat route is not touched. Different surface, different scoping. Future convergence is post-v0.13.
10. **`bss_clock.now()` everywhere.** No new violations introduced. The grep guard catches.
11. **`astream_once`, not `Session`.** The in-memory `Session` class is retired. Anything multi-turn goes through `Conversation` + `astream_once(transcript=)`.
12. **No raw CRUD from cockpit.** All writes still go through services + policies. The cockpit's only privilege is full *visibility* and an explicit operator identity; the policy layer remains the authoritative gate.

## Out of scope

- **Customer chat changes.** v0.12's `customer_self_serve` profile, `*.mine` wrappers, and chat route are untouched. Future convergence on a unified Conversation store is post-v0.13.
- **Per-operator UI customization beyond OPERATOR.md.** No themes, no key-binding configurability, no dashboard layouts.
- **Real OAuth / JWT / SSO for staff.** Explicitly retired with a DECISIONS entry. Not deferred — gone.
- **Conversation export / import** (JSON share, transcript download). Add later if needed.
- **Conversation search** (cross-session "find every session about CUST-001"). v0.13 ships per-session listing only.
- **Chat history beyond the cockpit.** No "operator activity feed" or "all conversations browser." `audit.domain_event` is still authoritative for forensic queries.
- **Long-running multi-step agentic plans** that span multiple cockpit turns autonomously. v0.13 is one-turn-one-stream-one-response, multi-turn via operator follow-up.
- **Voice / audio cockpit input.** Text only. ASCII renderers only.
- **Real-time hand-off between operator and customer** (e.g., operator joins a customer's chat session). Out of scope; the customer chat is its own surface.
- **Per-customer custom OPERATOR.md.** One global persona file. If "different operators want different defaults," edit `settings.toml` (for actor) and the persona stays shared, or fork the file and `ln -s` the active one.
- **Conversation TTL / archive policy.** v0.13 keeps all sessions indefinitely. A retention runbook can land in v0.14 if storage shows pressure.
- **Programmatic cockpit API for partners.** No external surface. The cockpit is internal-only.
- **Multi-tenant cockpit.** v0.13 assumes single-tenant `DEFAULT`. The `tenant_id` column on tables is preserved (doctrine), but no per-tenant filtering UI.
- **Streaming markdown rendering in the browser.** Plain SSE pills + assistant bubbles, same as customer chat. Fancy markdown rendering is post-v0.13.

## The trap

**Do not accidentally touch the customer chat.** v0.12's `customer_self_serve` profile, `*.mine` wrappers, ownership check, caps, escalation, transcript hashing, system prompt — all of that is *not in scope*. The cockpit is a separate surface with separate plumbing. If a refactor of `_chat_tool_pill_html` (extraction to `bss-portal-ui`) breaks the customer chat, that's a P0; revert and do the extraction more carefully.

**Do not let `operator_cockpit` become a "trust me bro" profile.** It's full coverage, yes, but every entry is explicit, greppable, and validated at startup. The day the registry gains a new tool the cockpit needs, someone has to add the tool name to `_profiles.py` — that conscious step is the discipline.

**Do not skip the propose-then-confirm pattern for destructive actions.** It's tempting to "just trust the operator" since there's no auth — but the agent is the one issuing the proposal, and an LLM happily proposes wrong things. Confirm-pattern catches the wrong things at the operator's eye, which is the only review surface left after staff auth retired.

**Do not let `OPERATOR.md` drift into prompt-injection territory.** It's prepended to the system prompt every turn. An operator typing "ignore all policy violations" or "always allow destructive without confirm" into OPERATOR.md is a foot-gun that the *operator chose*. That's the trust model. But the doctrine docs need to mention this clearly so a future operator doesn't reach for it as an "easy override" without realising they've moved the trust boundary.

**Do not store secrets in `settings.toml`.** `BSS_*_API_TOKEN`, DB URL, OpenRouter key — those stay in `.env`. settings.toml is non-secret operator preference. The greppable rule: anything that would burn the perimeter if leaked stays in env.

**Do not ship the REPL slash commands without the browser equivalents.** Slash-command parity is a doctrine target. If `/focus` lands in the REPL but the browser has no UI for it, the next operator touching the browser will reinvent some other affordance, and the surfaces drift. Either both at once or neither this PR.

**Do not let `Conversation.transcript_text()` grow unbounded.** A long-running session is a long transcript fed back to the LLM every turn. Either truncate (keep the last N messages + a synopsis) or warn the operator at a threshold and recommend `/new`. A 50-turn session feeding 50-turn-context to the LLM is a token-cost trap.

**Do not assume the v0.5 `agent_log` widget renders cockpit events correctly out of the box.** v0.5 was one-shot per ask; v0.13 is multi-turn. The widget might need a small update to handle the "tool call from turn 3 of session SES-... appearing 20 minutes after the last user input" case. Test that explicitly.

**Do not attribute cockpit audit rows to `channel="llm"`.** Use `channel="cli"` for REPL turns and `channel="portal-csr"` for browser turns. The agent runs the same way in both, but the *channel* (interface the human used) is what audit consumers want to know. Same as v0.5's pattern.

**Do not declare v0.13 done if the cross-surface round trip is flaky.** REPL writes turn → exit → browser shows turn → browser writes turn → REPL `--session` resumes → REPL shows browser's turn. That round trip is the entire promise of the release. If it works once in three but flakes one in three, that's not done.

**Do not call this v1.0.** v0.12's three integrations (Singpass / Stripe / SM-DP+) still gate v1.0. v0.13 is a refactor + UX revolution for the staff side; the customer side stays as v0.12 shipped. Tag `v0.13.0`. v1.0 stays defined as the three swaps.
