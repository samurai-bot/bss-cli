# v0.7.0 — Catalog versioning + plan change machinery

> **The plumbing release.** No portal UX, no new login flow, no chat scoping. v0.7 makes the catalog support the things every real MVNO does and v0.x has been silently unable to do: add a tier without a code change, run a CNY-style promo on a date window, change a plan's price without breaking existing customers' deals, and let a customer switch from PLAN_M to PLAN_L at next renewal. Everything in this version is data-model and tool-surface; the portal pages that consume it land in v0.10.

## Goal

Two coupled deliverables, both invisible to the customer until v0.10 wires them up:

1. **Time-bounded catalog rows.** `product_offering` and `product_offering_price` gain `valid_from / valid_to`. `subscription` gains a price snapshot copied at order time. Renewal charges the snapshot, never the catalog. Catalog price changes affect new orders only — existing customers are grandfathered until explicitly migrated.
2. **Plan change as a scheduled-at-renewal switch.** A new tool `subscription.schedule_plan_change` records a pending offering on the subscription. The next renewal applies it: charges the new offering's snapshot price, swaps the active offering, clears the pending fields. No proration, no immediate effect, no shortcut around doctrine. Operators get a separate `subscription.migrate_to_new_price` for catalog-driven price migrations with notice.

When v0.7 ships, the operator can use the admin CLI to run a CNY promo, add an XS or XL plan, schedule a price change for an existing plan with 30-day notice, and let any future portal page schedule a customer's plan change for next renewal. The substrate is in place; v0.10 builds the customer-facing UX on top.

## Deliverables

### Track 1 — Schema migration

#### 1.1 Alembic migration

Single migration in `packages/bss-models/alembic/versions/0xxx_catalog_versioning.py`. Adds:

- `catalog.product_offering.valid_from TIMESTAMPTZ NULL`
- `catalog.product_offering.valid_to TIMESTAMPTZ NULL`
- `catalog.product_offering_price.valid_from TIMESTAMPTZ NULL`
- `catalog.product_offering_price.valid_to TIMESTAMPTZ NULL`
- `subscription.subscription.price_amount NUMERIC(10,2) NULL` (NOT NULL after backfill)
- `subscription.subscription.price_currency TEXT NULL` (NOT NULL after backfill)
- `subscription.subscription.price_offering_price_id TEXT NULL` (FK to `catalog.product_offering_price.id`, NOT NULL after backfill)
- `subscription.subscription.pending_offering_id TEXT NULL`
- `subscription.subscription.pending_offering_price_id TEXT NULL`
- `subscription.subscription.pending_effective_at TIMESTAMPTZ NULL`

#### 1.2 Backfill step (in the same migration)

Before flipping the three snapshot columns to NOT NULL, the migration backfills every existing subscription:

```sql
UPDATE subscription.subscription s
SET price_amount = p.amount,
    price_currency = p.currency,
    price_offering_price_id = p.id
FROM catalog.product_offering_price p
WHERE p.offering_id = s.offering_id
  AND s.price_amount IS NULL;
```

Migration logs `backfilled_subscriptions=<N>` and asserts no remaining NULLs before applying NOT NULL constraints. If any subscription has no matching price (which shouldn't happen but is worth defending against), the migration aborts loudly with the offending IDs listed.

#### 1.3 Updated `bss_models` SQLAlchemy models

`packages/bss-models/bss_models/catalog.py` (or wherever offering/price live) — add the four columns to the mapped classes. `packages/bss-models/bss_models/subscription.py` — add the six new columns. Keep the foreign-key relationship from `subscription.price_offering_price_id` → `product_offering_price` as a one-way reference (no back-population needed; we don't enumerate subscriptions from a price row).

#### 1.4 Updated seed

`packages/bss-seed/bss_seed/catalog.py` — extend the existing seed so `valid_from = NULL, valid_to = NULL` is the explicit "always available" default for the three seeded offerings and three prices. No behavioural change versus today; just makes the columns non-mysterious in seed dumps.

### Track 2 — Active-catalog queries

#### 2.1 Repository methods

`services/catalog/bss_catalog/repository.py` (post-v0.6 layout):

- `async def list_active_offerings(at: datetime | None = None) -> list[Offering]`
  - `at` defaults to `bss_clock.now()`.
  - Filter: `(valid_from IS NULL OR valid_from <= at) AND (valid_to IS NULL OR valid_to > at) AND is_sellable AND lifecycle_status = 'active'`.
  - Returns offerings ordered by price (ascending) for deterministic display.
- `async def get_active_price(offering_id: str, at: datetime | None = None) -> Price`
  - Filter: same time bounds on `product_offering_price`.
  - If multiple rows match, **lowest amount wins** — this is the documented promo-discount semantic.
  - Raises `PolicyViolation(rule="catalog.price.no_active_row")` if no active price exists. This protects renewal from silently charging a phantom-row price.
- `async def get_offering_price_by_id(price_id: str) -> Price | None`
  - Direct lookup by snapshot — used by renewal-time math.
  - No time filter; the price row is fetched for its amount/currency regardless of whether it's still active. The snapshot remembers history.

#### 2.2 HTTP routes

Add to `services/catalog/bss_catalog/routes/product_offering.py` and `product_offering_price.py`:

- `GET /tmf-api/productCatalogManagement/v4/productOffering?activeAt=ISO-8601` (defaults to now) — TMF-shaped query parameter for the time filter.
- `GET /tmf-api/productCatalogManagement/v4/productOfferingPrice/{id}/active?activeAt=...` — returns the active price row for an offering at a given moment.

Existing TMF endpoints (`GET /productOffering` without `activeAt`, `GET /productOffering/{id}`) keep their current "all-rows" semantics for back-compat with v0.6 catalog tests.

#### 2.3 `bss-clients` CatalogClient updates

`packages/bss-clients/bss_clients/catalog.py` — add `list_active_offerings(at=None)` and `get_active_price(offering_id, at=None)` methods that hit the new endpoints. Existing `list_offerings()` / `get_offering()` stay.

### Track 3 — Snapshot at order time

#### 3.1 COM order creation

`services/com/app/services/order_service.py:create_order`:

- After `check_offering_exists` succeeds, call `catalog_client.get_active_price(offering_id)` to get the price row that's active *now*.
- Stamp the price snapshot onto the order item characteristics: `priceAmount`, `priceCurrency`, `priceOfferingPriceId`.
- The `order.in_progress` event payload gains the same three fields under `item.priceSnapshot`.
- New policy `check_offering_currently_sellable` (in `services/com/app/policies/order.py`) — wraps the active-offering check, raises `policy.offering.not_sellable_now` if the offering is outside its `valid_from / valid_to` window. Replaces the existing `check_offering_exists` for the create-order path; the old policy stays for paths that just need existence (e.g., reads).

#### 3.2 SOM forwarding

`services/som/app/services/som_service.py` consumes `order.in_progress`, reads `priceSnapshot` from the item, forwards it onto `service_order.completed` so subscription receives it.

#### 3.3 Subscription persistence

`services/subscription/app/services/subscription_service.py:create_from_service_order`:

- Read `priceAmount / priceCurrency / priceOfferingPriceId` from the event payload.
- Persist into `subscription.price_amount / price_currency / price_offering_price_id`.
- Reject (raise) if any of the three is missing — fail loudly during transition rather than store NULL.

#### 3.4 Renewal reads snapshot

`services/subscription/app/services/renewal_service.py` (or wherever the renewal charge is built):

- Read `subscription.price_amount` and `subscription.price_currency` from the row.
- **Do not** call `catalog_client.get_offering_price` or `get_active_price` for the renewal-time price. The snapshot is the contract.
- The renewal payment.charge call uses the snapshot amount; the `subscription.renewed` event payload includes the snapshot for downstream visibility.

### Track 4 — Plan change

#### 4.1 New service method

`services/subscription/app/services/subscription_service.py:schedule_plan_change(subscription_id, new_offering_id)`:

- Validates via policies (Track 4.2).
- Calls `catalog_client.get_active_price(new_offering_id)` to snapshot the price row that's active *at scheduling time* — the customer is committing to that price for the new plan.
- Writes `pending_offering_id`, `pending_offering_price_id`, `pending_effective_at = subscription.next_renewal_at`.
- Emits `subscription.plan_change_scheduled` event with `{ subscription_id, current_offering_id, new_offering_id, new_price_amount, effective_at }`.
- No state change on the subscription FSM — pending fields are advisory.

`services/subscription/app/services/subscription_service.py:cancel_pending_plan_change(subscription_id)`:

- Clears the three pending fields.
- Emits `subscription.plan_change_cancelled`.
- Idempotent — calling on a subscription with no pending change is a no-op (returns success).

#### 4.2 Policies

`services/subscription/app/policies/plan_change.py` (new file):

- `check_subscription_active_or_pending_renewal(sub)` — only `active` and `pending_renewal` states can schedule. Reject `terminated`, `suspended`, `pending_activation`.
- `check_offering_sellable_now(catalog_client, new_offering_id)` — rejects if the offering isn't in `list_active_offerings(now)`.
- `check_not_same_offering(sub, new_offering_id)` — rejects switching to the current plan.
- `check_no_pending_change(sub)` — only one pending switch at a time. To replace, the customer must cancel first.

Each raises a structured `PolicyViolation` with rule names like `subscription.plan_change.not_eligible_state`, `subscription.plan_change.target_not_sellable_now`, etc.

#### 4.3 Renewal-time application

`renewal_service.charge_renewal(subscription)`:

- Before computing the renewal charge, check `pending_offering_id`. If set and `pending_effective_at <= now`:
  1. Read the new price via `get_offering_price_by_id(pending_offering_price_id)` — direct lookup, no time filter.
  2. Charge that amount via `payment.charge`.
  3. On success: update `subscription.offering_id = pending_offering_id`, copy pending price snapshot to active price snapshot, **reset bundle allowances per the new offering's allowances**, clear pending fields.
  4. Emit `subscription.plan_changed` event.
  5. Continue with normal renewal post-state (bundle reset, period roll, etc.).
- If the charge fails, do **not** roll over the plan change — leave the subscription on the old plan, mark it `blocked` per existing renewal-failure policy, leave pending fields intact so a manual recharge or top-up could retry. Emit `subscription.plan_change_payment_failed`.

#### 4.4 Tools

`orchestrator/bss_orchestrator/tools/subscription.py`:

- `subscription.schedule_plan_change(subscription_id: SubscriptionId, new_offering_id: OfferingId) -> dict` — wraps the service method via `bss-clients`. Doc-string explains the next-renewal semantics.
- `subscription.cancel_pending_plan_change(subscription_id: SubscriptionId) -> dict` — wraps the cancel.
- Both registered in `TOOL_REGISTRY`. Both pass through the standard `PolicyViolationFromServer` translation so the LLM and CLI receive structured errors.

### Track 5 — Operator price migration

#### 5.1 Migrate-price service method

`services/subscription/app/services/subscription_service.py:migrate_subscriptions_to_price(filter, new_price_id, effective_from, notice_days, initiated_by)`:

- `filter` accepts `{"offering_id": "PLAN_M"}` only in v0.7. The interface is wider for future filters, but only the offering-id case is implemented.
- For each matched subscription:
  - Validate the new price belongs to the subscription's current offering. (Operator should not accidentally switch a PLAN_M sub onto a PLAN_L price row.)
  - Write `pending_offering_id = current offering_id` (same plan), `pending_offering_price_id = new_price_id`, `pending_effective_at = effective_from + notice_days`.
  - Emit `subscription.price_migration_scheduled` event with `{ subscription_id, old_amount, new_amount, effective_at, initiated_by }`.
- Returns a count of affected subscriptions plus the list of IDs.
- **Admin-only.** New policy `check_admin_role` (reads from `auth_context.current()`) gates the call. v0.7 hardcodes the admin role check to the existing v0.3 single-admin model; Phase 12 will swap the check for real RBAC.

#### 5.2 Renewal-time application — same machinery

Track 4.3's renewal-time application code handles both customer-scheduled plan changes and operator-scheduled price migrations — they're the same shape (pending offering + pending price + effective date). The renewal flow doesn't need to know which kind it's applying.

The only distinction is the emitted event after success: if `pending_offering_id == current offering_id` (i.e., a price migration, same plan), emit `subscription.price_migrated` instead of `subscription.plan_changed`. Downstream consumers interested in "the plan changed under me" can filter accordingly.

#### 5.3 Notification (logging-only adapter)

After scheduling a price migration, the service writes one row per affected subscription to a new event type `notification.requested` with `{ customer_id, channel: "email", template: "price_migration_notice", template_args: {...} }`. v0.7 has no real email delivery — there's just a `LoggingNotificationConsumer` that pretty-prints these events to stdout / a dev mailbox file. Real SMTP / SES adapter is deferred to v1.0 alongside other production integrations.

## Doctrine and constraints

1. **No proration.** Plan change is next-renewal only. If a deliverable suggests "let the customer switch immediately and refund the unused days," stop. The doctrine in CLAUDE.md is explicit.
2. **Snapshot is the contract.** Renewal reads `subscription.price_amount`, never the catalog. Any code path that hits `get_active_price` from a renewal-time stack frame is a bug.
3. **Catalog edits are admin-only.** No customer-facing surface and no LLM tool can edit the catalog. CLI only in v0.7. The CSR console gets HTML versions in a later version.
4. **Existing customers grandfathered.** Catalog price changes never silently touch existing subscriptions. Migrations are explicit, operator-initiated, and carry a notice period.
5. **Promos are exactly two patterns.** Limited-time offering (windowed `product_offering`) or discounted price on existing offering (windowed `product_offering_price`). No coupon codes, referral credits, stacked discounts, or A/B price tests in v0.7. Anything outside these two needs its own phase.
6. **No raw SQL in the admin CLI.** Every write goes through the policy layer via `bss-clients`. The CLI is a transport.
7. **Backfill is part of the migration, not a separate step.** A two-step "deploy then run script" loses idempotency and can leave subscriptions in NULL-snapshot state. Backfill in-migration, NOT NULL constraint right after.
8. **No portal UX in v0.7.** If a deliverable starts touching `portals/`, it belongs in v0.10. v0.7 is plumbing.
9. **`bss_clock.now()` everywhere.** No `datetime.now()` / `datetime.utcnow()` in any v0.7 code path. The v0.6 doctrine guard still passes.

## Test strategy

1. **Unit tests**:
   - Catalog active-row math: at exactly `valid_from`, just before `valid_to`, exactly `valid_to` (exclusive boundary), no active rows (raises), multiple overlapping rows (lowest amount wins).
   - Snapshot copy at order: COM persists snapshot, SOM forwards, subscription persists.
   - Plan-change scheduling: each policy variant is exercised (wrong state, target not sellable, same offering, pending change exists).
   - Renewal-time application: pending-set + pending-effective-at-passed → applies; pending-set + future effective → doesn't apply; pending charge fails → subscription blocks, pending intact.
   - Cancel pending: idempotent on no-pending; clears fields when present.

2. **Integration tests** (require running stack):
   - End-to-end "schedule plan change at v0.7": order PLAN_M → schedule switch to PLAN_L → freeze clock to next renewal → trigger renewal → assert subscription is on PLAN_L at PLAN_L's snapshot price → assert allowances reset to PLAN_L's allowances.
   - End-to-end "CNY promo": admin adds windowed offering PLAN_CNY → list_active_offerings before window returns 3 plans → list during window returns 4 → list after window returns 3 → a customer who ordered PLAN_CNY during the window has it persisted on their subscription.
   - End-to-end "discount price on existing offering": admin adds windowed price for PLAN_M ($20 instead of $25 from 2026-02-10 to 2026-02-24) → get_active_price during window returns $20 → an order placed during the window snapshots $20 → the order's renewal charges $20.
   - End-to-end "operator migration with notice": migrate-price PLAN_M to a new $30 row with 30-day notice → 10 subscriptions affected → 30 days pass (clock-frozen) → renewals charge $30 → subscriptions that terminated during the notice period skip the migration.

3. **Hero scenario suite update**:
   - All existing hero scenarios pass three runs in a row unchanged. The catalog versioning columns default to NULL valid bounds, so behaviour is identical to v0.6.
   - One new hero scenario: `catalog_versioning_and_plan_change.yaml` — drives the four integration test paths above as a single end-to-end flow. Three runs in a row green before tag.

4. **Doctrine guards**:
   - `rg 'get_active_price|get_offering_price' services/subscription/app/services/renewal_service.py` → empty. The renewal must not query the catalog.
   - `rg 'datetime\.(now|utcnow)' services/ packages/bss-models/` → empty (the v0.6 grep guard still applies).
   - Backfill assertion: post-migration test `SELECT count(*) FROM subscription.subscription WHERE price_amount IS NULL` returns 0.

5. **Migration replay safety**:
   - Apply migration on a copy of the v0.6 dev database; assert backfill populates every existing subscription; assert all NOT NULL constraints hold; downgrade and re-upgrade cleanly (the migration's `downgrade()` drops the columns and works against a backfilled-then-downgraded state).

## Out of scope

- **Customer-facing UX for plan change.** No `/plan/change` page in v0.7 — that's v0.10. The tools are callable from the CLI and orchestrator only.
- **Coupon codes, referral credits, gift cards, loyalty tiers, stacked discounts, A/B price tests, free-month bundles.** Promos are the two patterns in §5. Anything else needs its own phase.
- **Real notification delivery.** v0.7 logs `notification.requested` events; no SMTP / SES. Real email is v1.0.
- **Multi-tenancy in catalog.** Catalog is single-tenant in v0.7 (as it is now). Multi-tenant catalog gating is a Phase 12+ concern.
- **Bulk-import of offerings.** Admin CLI adds one at a time. If you need 50 plans, run the command 50 times or write a shell loop.
- **Catalog write API for partners.** External catalog editing is not exposed at any HTTP surface in v0.7. CLI only.
- **Versioning of bundle allowances.** v0.7 versions price and offering. Allowances change with offering. If you need "PLAN_M with 30GB until June, then 35GB after" without changing the price, retire the offering and add a new one rather than versioning allowances.
- **Plan-change preview in the orchestrator.** "What would my new bill be?" is a chat / dashboard concern landed in v0.10 / v0.11. The tool surface in v0.7 is action-only.

## The trap

**Do not implement plan change as terminate-and-recreate.** It's tempting because it reuses the existing terminate + create-from-order paths. It's wrong: the customer's pending VAS top-ups are lost, the audit log shows voluntary churn, and there's a window where the customer has no active line. Pending fields + renewal-time pivot is the only correct shape.

**Do not skip the backfill.** Shipping NULL price snapshots with NOT NULL deferred to a later migration means the first renewal cycle on real subscriptions blows up. Backfill is fifteen lines of SQL — do it as part of the migration, not "later."

**Do not let renewal touch the catalog.** The whole point of snapshot-at-order is that catalog changes don't disturb existing customers. If renewal looks up `get_active_price`, the doctrine collapses. Greppable guard, run it in CI.

**Do not generalise the "promotions" model beyond the two patterns.** It's tempting to add a "campaign" abstraction, a "discount stack," a "coupon engine." Resist. Two patterns cover what a small MVNO actually does in its first year. New mechanics get new phases.

**Do not let the admin CLI bypass policies.** Operator commands feel different — "I trust myself, why the policy round-trip?" — but the audit trail, the structured errors, and the consistency with future-RBAC-via-Phase-12 all depend on every write going through `services/.../app/policies`. Raw SQL in the admin CLI is a v0.7 failure even if no test catches it.

**Do not let the operator-migration tool become a one-shot batch script.** Each affected subscription gets its own pending fields and its own per-subscription event. If you implement it as a single "UPDATE all matching rows" SQL statement, you lose the per-subscription audit trail and the per-subscription cancel-by-terminating path.

**Do not let "lowest active wins" silently mask a misconfigured catalog.** If you see two simultaneously active price rows for the same offering and they aren't a deliberate promo, that's a configuration bug. Add a startup sanity check (or a `bss admin catalog show --at now` review step in the runbook) — don't just let the lower one win and never tell anyone.

**Do not start the portal work in v0.7.** Every time a deliverable here suggests "and then the portal page would call this," the answer is "yes, in v0.10." v0.7 is plumbing. v0.10 is the customer-facing render of that plumbing. Mixing them slips both.
