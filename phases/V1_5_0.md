# V1.5.0 — Multi-step orchestration in the cockpit

**Status:** planned (2026-05-26)
**Branch:** `feat/v1.5-multi-step-cockpit` (to be cut)
**Migration head at start:** 0030 (post-v1.4.1 `promotion.state='exhausted'`)

## Why now

Three layers of the cockpit stack to make the agent effectively one-step
today, even though the ReAct loop mechanically supports chaining:

1. **Safety contract.** Destructive tools (`subscription.terminate`,
   `case.close`, `customer.close`, …) gate on operator `/confirm`. By
   design — `bss_orchestrator/safety.py:24`. No change needed there.
2. **No batched mode.** Every destructive step in a compound action
   requires its own propose + `/confirm`. There is no operator escape
   hatch that says "I trust the plan, run it." Compare loyalty-cli's
   `LOYALTY_REPL_LLM_AUTONOMY={granular,batched}`.
3. **v0.19 anti-hallucination prompt** instructs the agent to reply
   with one short sentence and stop after every renderer-backed read.
   That was the right fix for the `customer.get` → "Product Catalog:
   PLAN_L… PLAN_M…" re-fabrication bug (`bss_cockpit/prompts.py:128–134`),
   but it ALSO suppresses chained reads — so the operator can't say
   "investigate CASE-042" and have the agent walk `case.get` →
   `customer.get` → `subscription.list_for_customer` → `case.list_tickets`
   in one turn. Case investigation, signup-by-conversation, and
   compound writes ("create customer + issue welcome offer") all hit
   this wall.

Loyalty-cli landed the equivalent unlock at v0.11 (DECISIONS 2026-…,
`cli/loyalty-cli/src/loyalty_cli/repl/llm.py`). The design is
proven in a sister project, doctrinally compatible, and lifts cleanly
into bss-cli without touching the policy layer.

## Scope (this release)

Five pieces. None add a container; none add a schema; all sit in
`bss-orchestrator` + `bss-cockpit`.

### 1. `BSS_REPL_LLM_AUTONOMY` env var

```python
# orchestrator/bss_orchestrator/autonomy.py  (new)

VALID_AUTONOMY_MODES = ("granular", "batched")

class AutonomyMisconfigured(RuntimeError):
    """Raised at orchestrator startup if BSS_REPL_LLM_AUTONOMY is set
    to an unrecognised value. Fail-closed."""

def read_autonomy_mode() -> str:
    raw = os.environ.get("BSS_REPL_LLM_AUTONOMY", "granular").strip().lower()
    if raw not in VALID_AUTONOMY_MODES:
        raise AutonomyMisconfigured(...)
    return raw
```

- `granular` (default) — current behaviour. Each destructive step in a
  multi-step loop propose-then-`/confirm`s separately. Maximum
  operator control. This matches v1.4.1 behaviour exactly.
- `batched` — the FIRST destructive step in a loop gates on `/confirm`;
  subsequent destructive steps in the same `/confirm`-resumed loop
  execute autonomously. Operator confirms the plan once and the loop
  runs to completion.
- Unknown values raise `AutonomyMisconfigured` at orchestrator boot —
  same fail-closed shape as `BSS_API_TOKEN=changeme`.

Honoured identically in the REPL (`bss`) and the v0.13 browser cockpit
(`portal-csr`). Read once at process start, cached on `app.state`.

### 2. `ITERATIVE FLOW` block in the cockpit system prompt

Add to `packages/bss-cockpit/bss_cockpit/prompts.py` `_COCKPIT_INVARIANTS`,
positioned after the renderer-backed-read rules. Authorises compound
actions with bss-shaped worked examples:

```
ITERATIVE FLOW (multi-step compound actions): you may call multiple
tools across iterations to accomplish a compound request. After each
tool call, you'll see the tool's result back — use it to decide the
next step. Patterns:

- "show me CUST-001 and then list their subscriptions" → call
  customer.get first; on its result, call subscription.list_for_customer
  customer_id=CUST-001; finally reply with a brief one-sentence summary.
- "investigate CASE-042" → call case.get first; on its result, call
  customer.get on the linked customer + subscription.list_for_customer
  + case.list_tickets in any order; reply with a brief situational
  summary and a proposed next action (without proposing destructive
  tools unless the operator asks).
- "register CUST-XYZ + create order for PLAN_M" → call customer.create
  (destructive — operator will /confirm); on resume, call
  customer.attest_kyc + payment.add_card + com.create_order in order.
- When a tool returns a structured error (PolicyViolation,
  DESTRUCTIVE_OPERATION_BLOCKED), ADAPT — try different args, ask the
  operator for missing info, or give up cleanly. Don't loop on the same
  broken call.

End a compound action by returning a FINAL TEXT message (no tool call)
summarising what you did and any remaining steps for the operator.
```

### 3. Soften the v0.19 "Done." rule

Today (`bss_cockpit/prompts.py:113–118`):

> After a renderer-backed tool call, your assistant reply MUST be one of:
>   - a single short sentence acknowledging completion ("Done." / "Found
>     3 customers." / "Catalog above." — that's it).
>   - …

Soften to:

> After a renderer-backed tool call, your assistant text reply MUST be
> a single short sentence (or empty). However, you MAY emit ANOTHER
> tool call instead of a text reply — the rendering rule applies only
> to the text reply that terminates a compound action. The
> anti-duplication rule (don't re-render data the cockpit already
> showed as ASCII) is unchanged.

Net effect: the agent can chain `customer.get` → `subscription.list_for_customer`
without being forced into a "Done." pause between them, but the loop
still terminates with a one-sentence reply (no markdown table
re-rendering). Anti-hallucination doctrine intact.

### 4. 3-strike bail for tool-error loops

```python
# orchestrator/bss_orchestrator/session.py  (in astream_once)

MAX_CONSECUTIVE_TOOL_FAILURES = 3

# In the ReAct loop's tool-result handler:
if tool_result is PolicyViolation or tool_result is DESTRUCTIVE_OPERATION_BLOCKED:
    consecutive_failures += 1
    if consecutive_failures >= MAX_CONSECUTIVE_TOOL_FAILURES:
        # Bail the loop with a structured "couldn't recover" message
        # rendered as a clear cockpit panel. Operator sees what was
        # tried and the final failure context.
        raise AgentLoopBailout(reason="3 consecutive tool failures", history=...)
else:
    consecutive_failures = 0
```

Catches LLM thrash where the same broken call recurs (Gemma's known
failure mode on long compound prompts). Loyalty-cli has this; bss-cli
doesn't.

### 5. Cockpit-chrome filter on history rehydration

When loading prior turns from `cockpit.message` into the LLM context
for `astream_once(transcript=...)`, strip rows that are cockpit-emitted
chrome (the `⚠ PROPOSE`, `REFUSED`, `/confirm` confirmation banners).
The LLM should see only genuine user / assistant / tool turns — its
own prior chrome leaking back in confuses subsequent reasoning.

Lift the pattern from `cli/loyalty-cli/src/loyalty_cli/repl/loop.py`'s
`_is_cockpit_chrome` / `_strip_fake_propose`. New module:
`packages/bss-cockpit/bss_cockpit/chrome_filter.py` with a frozen list
of known cockpit prefixes.

## Design rules

1. **Lift from loyalty-cli; don't reinvent.** The autonomy env, the
   iterative-flow prompt block, the 3-strike bail, and the chrome
   filter all have working precedents in `cli/loyalty-cli/`. Read
   those files first, port the shapes, adapt the constants. Don't
   redesign.
2. **No change to the safety contract.** `DESTRUCTIVE_TOOLS` stays
   exactly as it is. `/confirm` stays the gate. Batched mode is about
   how many `/confirm`s a compound action needs, NOT which tools
   require one. A future `subscription.terminate` is still gated by
   the wrapper, full stop.
3. **No new container, no new schema, no migration.** v1.5 is a
   behaviour-change release in two packages
   (`bss-orchestrator`, `bss-cockpit`) plus a prompt edit. Migration
   head stays at 0030.
4. **Doctrine intact.** Write through policy; renderers own ASCII
   output; agent never paraphrases tool results; knowledge.search
   still mandatory for prose answers. The anti-duplication rule
   stays sharp — what changes is the next-action permission, not the
   text-reply contract.
5. **Single autonomy state per process.** No per-session override (yet).
   Operator sets the env var, restarts the orchestrator + cockpit, and
   the choice applies to every astream_once in that process. A
   per-session `/autonomy granular|batched` slash command is a v1.5.x
   open question.
6. **Customer chat is unaffected.** The `customer_self_serve` profile
   does not expose destructive tools to begin with (`*.mine` wrappers
   over reads + `case.open_for_me` + a few writes already in the
   profile). `BSS_REPL_LLM_AUTONOMY` applies to the
   `operator_cockpit` profile only. Doctrine guard: customer chat
   prompt MUST NOT include the ITERATIVE FLOW block.

## The autonomy contract

**Granular (default — current behaviour):**

```
> register CUST-XYZ then issue them PLAN_M
⚠  PROPOSE: customer.create id="CUST-XYZ" email="..."
> /confirm
(customer card renders)
⚠  PROPOSE: com.create_order customer_id="CUST-XYZ" offering="PLAN_M"
> /confirm
(order card renders, plus agent's wrap-up sentence)
```

**Batched (opt-in via `BSS_REPL_LLM_AUTONOMY=batched`):**

```
> register CUST-XYZ then issue them PLAN_M
⚠  PROPOSE: customer.create id="CUST-XYZ" email="..."
   (next destructive steps in this loop will run without re-asking)
> /confirm
(customer card renders)
(order card renders — auto-executed, no second /confirm)
(agent's wrap-up sentence)
```

The PROPOSE banner explicitly tells the operator what "/confirm" is
authorising in batched mode — the *current* step plus any subsequent
destructive steps in this loop. This is the doctrine choice: opt-in,
visible, no surprise authorisation. Mirrors loyalty-cli's panel text.

## Open questions deferred to v1.5.x

1. **Per-session autonomy override.** `/autonomy granular` / `/autonomy
   batched` slash command in the REPL that overrides the env var for
   the current session. Useful for "I'm doing a risky migration, drop
   me back to granular even though my default is batched." v1.5.1.
2. **Per-tool autonomy.** Annotate some destructive tools as "always
   granular even in batched mode" (`admin.reset_operational_data`,
   `admin.force_state`). Probably worth doing — opening as a v1.5.2
   question rather than a v1.5.0 default.
3. **Investigation playbook tools.** Composite read tools like
   `case.investigate(case_id)` that bundle the read chain server-side
   and return one structured payload. v1.5 unlocks the LLM-driven
   path; v1.6 could add the server-side fast path for high-frequency
   investigations.
4. **Visibility/debt-closure work** from the alternative v1.5 pitch
   (outbox/inbox dashboards, usage.rated consumer + live verify, CI
   integration for e2e, parallel-spec safety) — keep on the v1.6
   candidate list.
