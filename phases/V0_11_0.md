# v0.11.0 — Signup funnel goes direct (extending the post-login carve-out)

> **The doctrine consolidation release.** v0.4 made the signup funnel orchestrator-mediated as the demo artifact for "the agent pattern works on a customer-facing flow." v0.10 carved out post-login customer self-serve as direct-API. v0.11 finishes the carve-out: signup is a deterministic happy path with no LLM judgment calls, so it goes direct. The chat surface remains the only orchestrator-mediated route — and v0.12 narrows that to a per-customer scoped tool profile.

## Why

The v0.4 demo's signature artifact — the agent log streaming during signup — was load-bearing when the project's purpose was *demonstrating the agent pattern is viable on a customer-facing flow*. Once v0.10 shipped post-login direct-API, the architecture statement was made: BSS-CLI separates **deterministic routine flows** (direct, sub-second, deterministic) from **judgment-required flows** (orchestrator-mediated, LLM-reasoning, slower).

Signup is a deterministic routine flow:

1. Pick plan → catalog read.
2. Pick MSISDN → inventory list + reserve.
3. Enter name + email + phone → form validation.
4. KYC attestation → channel-layer signed JWT in, BSS records it.
5. COF → tokenize + create_payment_method.
6. Place order → COM creates a ProductOrder, SOM activates it.

Each step has one correct next step. There is no branching that benefits from LLM reasoning. The 85+ second wall time on the v0.10 hero (`portal_self_serve_signup_v0_8`) is the cost of running an LLM round-trip per step on a flow that doesn't need one.

The educational artifact (watching an agent drive a customer flow) is preserved by:

- The existing `llm_troubleshoot_blocked_subscription` hero — the canonical "LLM adds value where judgment is needed" demo.
- The `portal_csr_blocked_diagnosis` hero — same story, CSR-side.
- The chat surface (still orchestrator-mediated, scoped tighter in v0.12).

If anyone needs the v0.4-style "watch the agent sign someone up" demo for a presentation, the v0.10 hero suite + the v0.4 git tag remain accessible — they just stop being the primary signup path.

## Doctrine extension

Update `CLAUDE.md` anti-pattern from:

> **(v0.4–v0.9 / pre-login + signup + chat)** The signup funnel and the chat surface go through the orchestrator via `agent_bridge.*` → `astream_once`.

to:

> **(v0.4–v0.10 / signup + chat)** The signup funnel and the chat surface go through the orchestrator via `agent_bridge.*` → `astream_once`.
>
> **(v0.11+ / chat only)** Only the chat surface goes through the orchestrator. The signup funnel writes directly via `bss-clients`, with the same shape as v0.10 post-login routes (one route → one BSS write, ownership-bound, audited where applicable).

Greppable doctrine guard updates: `make doctrine-check`'s portal-handlers-don't-write rule narrows from "everything except v0.10 post-login routes" to "only the chat route" — i.e. the signup route files (`signup.py`, `msisdn_picker.py`, `confirmation.py`, `activation.py`, `agent_events.py`) join the v0.10 carve-out whitelist.

## Goal

Migrate the signup funnel from orchestrator-mediated (LLM tool chain via SSE) to direct API calls from route handlers, **without** changing the URL shapes the v0.4/v0.8 customers know:

- `/welcome` (public) — unchanged.
- `/plans` (public) — unchanged.
- `/signup/{plan_id}` — the form. Now POSTs directly to a route handler that orchestrates the deterministic chain.
- `/signup/{plan_id}/msisdn` — number picker. Same shape, direct read.
- `/signup/{plan_id}/progress` — replaced or simplified. The agent-log SSE widget is removed (no LLM driving anything); the page either shows a brief deterministic progress trace (5-step timeline) or redirects directly to `/activation/{order_id}` once SOM completes.
- `/activation/{order_id}` — unchanged shape. Renders LPA QR + activation code.
- `/confirmation/{subscription_id}` — unchanged.

Sub-second per step except the SOM activation wait (still polled, still sub-2s with the simulator).

## Deliverables

### Track 1 — Doctrine + docs (precedes code)

1. **CLAUDE.md anti-pattern split** — extend the v0.10 carve-out per the wording above.
2. **DECISIONS entry** — capture the doctrine consolidation, alternatives rejected (dual-path, status quo), consequences (v0.4 demo artifact retired from signup; chat surface remains the primary "agent in the path" demo).
3. **ARCHITECTURE.md update** — the topology diagram's solid line (direct) extends to cover signup; the dashed line (orchestrator) remains only for `/chat`.

### Track 2 — Signup route migration

4. Replace the agent-driven SSE flow with a direct route-handler chain that calls `customer.create` → `customer.attest_kyc` → `payment.add_card` (via the v0.10 client-side tokenizer) → `order.create` → `order.wait_until(completed)`. Each of the five is one bss-clients call from the route handler — composition is **server-side** (the order activation is already a coordinated COM/SOM/provisioning workflow), not in the route. The route just orchestrates the user-facing form sequence.
5. **Step-up auth** — signup writes happen **before** the customer has a session, so step-up doesn't apply. The session is established on first email-verify (v0.8 flow). The signup route is gated by `requires_verified_email` (which v0.8 already enforces on `/signup/*`).
6. **Audit** — each direct write produces an `audit.domain_event` row downstream. The portal additionally writes a `portal_action` row capturing each signup step (`signup_create_customer`, `signup_attest_kyc`, etc.); add labels to `SENSITIVE_ACTION_LABELS` if the v0.10 catalogue is the source of truth, OR introduce a separate `SIGNUP_ACTION_LABELS` set if the step-up vs no-step-up distinction makes mixing them awkward (decide during implementation).

### Track 3 — Replace `agent_events.py` with deterministic progress

7. The v0.4 SSE widget streamed agent events. The v0.11 progress page either:
   - **Option A:** small deterministic 5-step timeline that polls `order.get(order_id)` every 500ms and ticks each step as it completes. No SSE, no agent log. ~30 lines of HTMX.
   - **Option B:** synchronous POST that runs the chain inline and returns a redirect when done. Simpler, no progress UI, but the user sees ~3-5s of blank-screen wait.
   - Pick A for the customer experience win.
8. Delete `agent_events.py` and the SSE plumbing it owns. The `agent-log-events` widget in `base.html` becomes chat-only (lives on the chat page, removed from signup pages). `bss_self_serve.routes.signup` now contains the deterministic chain; `agent_bridge.drive_signup` is removed.

### Track 4 — Hero scenarios

9. Replace `portal_self_serve_signup_v0_8.yaml` with `portal_self_serve_signup_direct.yaml`. Same coverage (full happy path, KYC + COF + order placed + activation completed + LPA visible), but asserts no `astream_once` was called on the orchestrator side and that the wall time is under 10 seconds (instead of ~85). The v0.8 LLM-mediated scenario is archived (kept in the repo at `scenarios/archive/` as a regression reference, not run by `make scenarios-hero`).
10. Three runs in a row green; doctrine guards (`rg 'astream_once' portals/self-serve/bss_self_serve/routes/` matches **only** chat) clean.

### Track 5 — Cross-cutting

11. **`agent_bridge.drive_signup` deletion** — the function and its `astream_once` invocation are removed. Anything that imports it gets cleaned up. The orchestrator's signup-related tool prompts can stay in the registry (chat may still want to talk about signup steps) but they're no longer driven from the portal.
12. **CSR portal untouched** — v0.5/v0.10 CSR's pre-baked-admin pattern remains. CSR has its own separate doctrine path and v0.12 / v1.x will revisit it.
13. **`make doctrine-check`** updates the carve-out whitelist; `signup.py`, `msisdn_picker.py`, `confirmation.py`, `activation.py` join the v0.10 list.

## Constraints (non-negotiable)

1. **The chat surface stays orchestrator-mediated.** v0.11 narrows the carve-out to *only* the chat route. v0.12 then scopes the chat further (per-customer profile + caps).
2. **No new BSS endpoints.** v0.11 reuses every endpoint v0.10 + earlier shipped. The change is purely "who calls them" (route handler instead of orchestrator).
3. **No URL changes on the customer-facing surfaces.** Old signup links keep working.
4. **The v0.4 demo artifact is retired from the primary signup path** — that's the doctrine commitment. If anyone needs the agent-log demo for a presentation, the git history at `tag v0.10.0` is where they go.
5. **One route → one BSS write.** Same as v0.10. The signup chain has 5 writes; they live in 5 routes (or 5 forms in one wizard) — not a single mega-handler that calls all 5.
6. **Wall-time budget.** The new hero must complete in under 10 seconds of wall time, three runs in a row.

## The trap

**Do not preserve the agent-log widget on signup "just in case someone wants it."** It's a feature flag that nobody owns and it'll rot. Either commit to the direct path or don't ship v0.11. The git history is the demo archive.

**Do not extend the carve-out to chat in this phase.** Chat is the one route that's still LLM-mediated by design. v0.12 narrows it; v0.11 only finishes signup.

**Do not invent a "minimal LLM" hybrid signup.** Some intermediate forms are tempting ("LLM picks the MSISDN", "LLM does the form fill"). Each of those reintroduces the latency + nondeterminism v0.11 exists to remove. If a flow needs LLM judgment, route it through `/chat`, not through signup.
