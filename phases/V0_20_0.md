# v0.20.0 — Knowledge tool (handbook RAG) + catalog roaming flag

> **The release that makes the cockpit literate about its own runbooks, and closes the v0.17 catalog gap nobody noticed.** Two features, one schema (a) the long-reserved `knowledge` schema in CLAUDE.md, one column add to `bundle_allowance`. The cockpit gains a `knowledge.search` / `knowledge.get` tool pair that reads `docs/HANDBOOK.md` + `CLAUDE.md` + the runbooks; the LLM cites a section anchor for any answer not derivable from real tool output. Catalog admin gains the `--data-roaming-mb` flag so v0.17's roaming allowance type stops requiring a SQL workaround or a seed-module edit.

## Prerequisites (operator one-time setup)

**`pgvector` must be available on the target Postgres** before `make migrate` runs migration `0022`. Stock `postgres:16` and `postgres:16-alpine` images do **not** include it.

**Bundled mode (`docker-compose.infra.yml`) — image swap:**

```yaml
postgres:
  image: pgvector/pgvector:pg16    # was: postgres:16-alpine
```

Drop-in same-major replacement; data directory format is identical, the existing `postgres_data` volume mounts unchanged, no `pg_dump`/`pg_restore` needed. After `docker compose up -d postgres`:

```bash
docker exec postgres psql -U bss -d bss -c "CREATE EXTENSION IF NOT EXISTS vector;"
```

**BYOI mode:** operator runs `CREATE EXTENSION IF NOT EXISTS vector;` once per BSS database on their existing Postgres before `make migrate`.

**Pre-swap verification (always):** confirm the running Postgres is on major version 16 — `psql -c "SELECT version();"`. Same-major (16.x → 16.x) swap is data-safe; downgrades are not supported by Postgres and major upgrades require a separate `pg_upgrade` flow that is **not** part of this phase. The pgvector image tracks current 16.x.

**Rollback:** the swap is reversible in 30 seconds — point `image:` back at `postgres:16` (or `postgres:16-alpine`) and `docker compose up -d postgres`. The data dir stays valid; only the `vector` extension becomes unavailable, which is fine because Tier 0 (FTS) doesn't need it. The migration's `CREATE EXTENSION IF NOT EXISTS` is idempotent.

A new runbook ships with this phase — `docs/runbooks/knowledge-indexer.md` — which carries the full backup/swap/verify/rollback procedure for the bundled path and a one-liner for BYOI hosts. The migration's pgvector activation block points operators at this runbook on failure.

## Goal

Two long-tail closures.

**Knowledge.** The cockpit's failure mode at v0.19 is that an operator asks "how do I rotate the cockpit token?" or "what's the prebaked-KYC env flag?" and the LLM either confidently paraphrases an outdated answer or admits it doesn't know. The substrate to fix this has been listed as frozen tech-stack in `CLAUDE.md` since v0.1 (`pgvector` extension on the same Postgres instance, schema `knowledge`) and reserved as planned in `TOOL_SURFACE.md` ("Knowledge tools (planned, Phase 11)"). v0.20 activates that plan with the smallest possible shape: index every authoritative `.md` in the repo, expose two tools, enforce citation-or-don't-claim. Tier 0 (Postgres FTS) ships first; Tier 1 (pgvector embeddings) ships in the same release once Tier 0 has proven the plumbing — same tool signature, swap the search backend.

**Catalog roaming flag.** v0.17 added `data_roaming` as a first-class allowance type (rating routes by `roaming_indicator`; subscription `is_exhausted` ignores roaming; VAS `purchase_vas` materializes the balance row). Seed data uses it correctly (`PLAN_M = 500mb`, `PLAN_L = 2gb`, `VAS_ROAMING_1GB`). But `bss admin catalog add-offering` never grew a `--data-roaming-mb` flag, so adding a new roaming-included plan requires either a `psql` `INSERT INTO catalog.bundle_allowance` or an edit + re-seed of `packages/bss-seed/bss_seed/catalog.py`. Acceptable as a documented gap (the v0.20 refresh of `add-product-offering.md` calls it out) but trivial to close while we're touching catalog admin paths anyway.

## Schema delta — single migration `0022`

`packages/bss-models/alembic/versions/0022_v0200_knowledge_chunks.py` (down_revision="0021"):

- New schema `knowledge` (CLAUDE.md has reserved this since v0.1).
- New table `knowledge.doc_chunk`:
  - `id TEXT PRIMARY KEY` (deterministic hash of `source_path + anchor`)
  - `source_path TEXT NOT NULL` (e.g. `docs/HANDBOOK.md`)
  - `anchor TEXT NOT NULL` (e.g. `81-catalog--add-an-offering-with-roaming`)
  - `heading_path TEXT NOT NULL` (e.g. `Part 8 → 8.1 → Roaming gap and workaround`)
  - `kind TEXT NOT NULL` (`handbook` / `doctrine` / `runbook` / `architecture` / `decisions` / `tool_surface`)
  - `content TEXT NOT NULL` (the section body)
  - `content_tsv tsvector GENERATED ALWAYS AS (to_tsvector('english', content)) STORED`
  - `content_hash TEXT NOT NULL` (sha256 of content; drives idempotent reindex)
  - `embedding vector(1024)` (NULL until Tier 1; column added in this migration so Tier 1 doesn't need a second migration)
  - `source_mtime TIMESTAMPTZ NOT NULL`
  - `indexed_at TIMESTAMPTZ NOT NULL DEFAULT now()`
- Indexes: GIN on `content_tsv` (Tier 0 search), HNSW on `embedding` (Tier 1; `WHERE embedding IS NOT NULL`).
- `CREATE EXTENSION IF NOT EXISTS pgvector` — first activation; gated behind a try/except so BYOI hosts without pgvector get a fail-fast `RuntimeError` at indexer startup pointing at `docs/runbooks/knowledge-indexer.md` rather than a silent migration-time crash.

No FK from `audit.domain_event` to `knowledge.doc_chunk` even when a turn cites — the citation is a free-text anchor in `audit.chat_usage.citations` (jsonb add, see below) so reindexing doesn't break audit history.

`packages/bss-models/alembic/versions/0023_v0200_chat_usage_citations.py` (down_revision="0022"):

- `audit.chat_usage` add column `citations jsonb NOT NULL DEFAULT '[]'::jsonb`. Each cockpit / customer-chat turn that calls `knowledge.search` records `[{anchor, source_path}]` here; replaces nothing, just adds forensic visibility.

## What ships

### 1. The indexer

**New module** — `packages/bss-knowledge/bss_knowledge/indexer.py`:

- `Indexer.reindex(*, repo_root, force=False) -> ReindexReport` — walks the **explicit** allowlist of doc paths (CLAUDE.md, ARCHITECTURE.md, DECISIONS.md, TOOL_SURFACE.md, ROADMAP.md, CONTRIBUTING.md, docs/HANDBOOK.md, docs/runbooks/*.md), chunks each on `### `-prefixed headings (handbook), `## `-prefixed headings (CLAUDE.md, ARCHITECTURE.md, runbooks), or per-entry on dated headings (DECISIONS.md). Returns `(added, updated, deleted, skipped_unchanged)`.
- Mtime cache: skip if `source_mtime` matches and `content_hash` unchanged. `--force` bypasses.
- Path allowlist is **explicit**, not a glob. `phases/V0_*.md` is intentionally NOT indexed (historical build plans mislead the LLM; doctrine note in CLAUDE.md cites this).
- Anchor generation uses the same algorithm GitHub/Obsidian use (lowercase, spaces → hyphens, strip non-alphanumeric except hyphens) so anchors in citations resolve in the editor.

**New CLI** — `cli/bss_cli/commands/admin_knowledge.py`:

- `bss admin knowledge reindex [--force]` — operator-driven reindex.
- `bss admin knowledge search <query> [--k 3]` — debug surface; same shape as the LLM tool returns.
- `bss admin knowledge list` — paginated browse over `knowledge.doc_chunk` (anchor, kind, heading_path, content_hash, indexed_at).

**Make target** — `make knowledge-reindex` (delegates to `bss admin knowledge reindex`). Listed in `make help`. NOT run by `make seed` — reindex is operator-initiated, not part of `up-all` boot, because the docs change with PRs and the indexer talks to a running Postgres.

### 2. The tools

**New module** — `orchestrator/bss_orchestrator/tools/knowledge.py`:

- `knowledge.search(query: str, k: int = 3, kinds: list[Literal[...]] | None = None) -> list[dict]` — Tier-0 FTS via `ts_rank` over `content_tsv`. Each hit returns `{anchor, source_path, heading_path, kind, snippet}` where `snippet` is `ts_headline` of the matched span (~30 words). `kinds` filter lets the LLM scope to e.g. `["doctrine"]` for "is this allowed?" questions.
- `knowledge.get(anchor: str, source_path: str) -> dict` — pulls full section text. Composite key (anchor + source_path) so identical anchors across files don't collide. Returns `{anchor, source_path, heading_path, kind, content, indexed_at}`.

Both tools registered in `operator_cockpit` profile only. **NOT** in `customer_self_serve` — the customer chat must not see operator runbooks (would leak destructive-flow hints, perimeter posture, internal env vars). `validate_profiles()` enforces.

Tool descriptions in the registry quote the citation rule verbatim so the LLM sees it on every tool-call decision: *"Use this when the operator asks a how-to / what-is question that no other tool can answer. Always cite the returned `anchor` and `source_path` in your reply. Do NOT paraphrase without citing."*

### 3. Tier-1 embedding upgrade (same release, gated by env)

**New module** — `packages/bss-knowledge/bss_knowledge/embedder.py`:

- `Embedder.embed_chunks(chunks)` — calls OpenRouter's embeddings endpoint (`text-embedding-3-small`-equivalent, 1024-dim). Sync, wrapped in `asyncio.to_thread`. Batches of 32; retries with backoff on rate limit. Off-network fallback: `sentence-transformers/all-mpnet-base-v2` if `BSS_KNOWLEDGE_EMBEDDER=local` is set (768 dim → padded to 1024 to keep the column shape stable; doctrine note in `knowledge-indexer.md`).
- Indexer becomes two-pass when `BSS_KNOWLEDGE_BACKEND=hybrid`: pass 1 writes content + tsvector; pass 2 fills `embedding` for any row where it's NULL or `content_hash` changed.
- Search becomes hybrid: cosine similarity over `embedding` for top-30, re-rank by `ts_rank` for top-K. Falls back to FTS-only when `BSS_KNOWLEDGE_BACKEND=fts`. Defaults to `fts` so first boot works without an embeddings provider.

### 4. Citation-or-don't-claim guard (cockpit + chat)

**Modified** — `cli/bss_cli/repl.py` and `portals/csr/bss_csr/routes/cockpit.py`:

- `_RE_KNOWLEDGE_CLAIM` regex catches first-person handbook/runbook/doctrine claims ("the handbook says", "per the runbook", "according to doctrine", "per CLAUDE.md", "per DECISIONS"). If the assistant turn matches AND no `knowledge.*` tool call fired this turn, the reply is replaced by the templated `_KNOWLEDGE_HALLUCINATION_FALLBACK`: *"I don't have a citation for that. Run `bss admin knowledge search "<query>"` or open `docs/HANDBOOK.md` for the authoritative answer."* Same pattern as v0.12's `_RE_ESCALATION_CLAIM` anti-hallucination guard for the customer chat.

**Modified** — `packages/bss-cockpit/bss_cockpit/prompts.py`:

- `_COCKPIT_INVARIANTS` gains one rule: *"For how-to / what-is questions outside the deterministic tool surface, call `knowledge.search` first and cite the returned `anchor` (e.g. `[HANDBOOK §8.4](docs/HANDBOOK.md#84-rotate-api-tokens)`) in your reply. If the search returns nothing relevant, say so explicitly — do not paraphrase from training data."*

## Doctrine amendments

Three new v0.20+ anti-pattern entries in `CLAUDE.md`:

- **Don't index `phases/V0_*.md` in the knowledge tool.** Phase docs are historical build plans — they describe what was *planned* at a point in time, not what *is*. Indexing them lets the LLM cite a deferred feature as if shipped, or quote a draft env var name that was renamed before merge. Index `CLAUDE.md`, `docs/HANDBOOK.md`, `docs/runbooks/*.md`, `ARCHITECTURE.md`, `DECISIONS.md`, `TOOL_SURFACE.md`, `ROADMAP.md`, `CONTRIBUTING.md` — and that's it. The allowlist in `bss_knowledge.indexer.INDEXED_PATHS` is the source of truth; adding a new path requires a doctrine review.
- **Don't expose the knowledge tools in `customer_self_serve`.** The handbook and runbooks describe destructive operator flows, perimeter posture, named-token rotation, KYC bypass flags, the cutover runbook for Stripe. None of that belongs in the customer chat — leaking it would teach a prompt-injected LLM exactly which flag to ask the operator about. The greppable rule: `knowledge.search` and `knowledge.get` appear in `TOOL_PROFILES["operator_cockpit"]` only; `validate_profiles()` enforces.
- **Don't paraphrase doctrine without citing.** When the operator asks a how-to / what-is question, the cockpit's contract is `knowledge.search` → cite anchor, OR say "I don't have a citation for that." The `_RE_KNOWLEDGE_CLAIM` guard enforces by replacing un-cited handbook claims with the fallback template. A model that reliably trips the guard is a doctrine bug — fix the prompt or the search relevance, don't relax the regex.

One v0.20+ surface convention in `CLAUDE.md`:

- **(v0.20+) Roaming offerings use `--data-roaming-mb` on `add-offering`.** The v0.17–v0.19 SQL workaround documented in earlier `add-product-offering.md` is retired. Existing rows seeded by hand stay valid (idempotent migration); new offerings go through the flag. The seed module (`packages/bss-seed/bss_seed/catalog.py`) keeps emitting raw inserts for determinism; that's the seed's contract, not an operator path.

## Known limits (intentional)

- **Reindex is operator-initiated.** No file-watcher in the cockpit container. Editing `docs/HANDBOOK.md` requires `make knowledge-reindex` (or restart with `BSS_KNOWLEDGE_REINDEX_ON_BOOT=true`, which v0.20 adds for dev convenience but defaults off in production). Watcher would couple the cockpit container to the repo filesystem in a way that doesn't work in BYOI deployments where the container only has `/app`.
- **Tier 1 embeddings are off by default.** `BSS_KNOWLEDGE_BACKEND=fts` is the v0.20 default. Operators who actually need recall beyond FTS flip to `hybrid`; the embedding cost is ~$0.01 per full reindex of the current corpus (~120 chunks × ~500 tokens each). Negligible against existing chat budget but real, so opt-in.
- **No cross-document graph.** A query that needs information from two sections returns two separate hits and the LLM stitches them together. Graph-RAG / multi-hop is a post-v0.20 question; current corpus size doesn't justify the complexity.
- **Customer chat does NOT get a public-docs RAG variant.** A future minor release could add `customer/HANDBOOK_PUBLIC.md` indexed under a `customer_self_serve_knowledge` profile but the doctrine cost (one more thing to keep in lock-step with code) is real and the customer chat's tool surface already carries the deterministic answers it needs. Defer.
- **No DLQ for failed embeddings.** A row whose embedding call fails stays at NULL and falls back to FTS-only on next search. Reindex retries on next run. Acceptable; loud-fail isn't useful here.

## Rollout

1. **Operator runs the prereq once** (see [Prerequisites](#prerequisites-operator-one-time-setup) above): bundled = swap image to `pgvector/pgvector:pg16`; BYOI = `CREATE EXTENSION` on the target Postgres. Migration `0022` is the first activation site.
2. **Apply migrations 0022 + 0023.** Migration's `CREATE EXTENSION IF NOT EXISTS vector;` is idempotent; if the extension wasn't installed at step 1, this raises a clear error pointing at `docs/runbooks/knowledge-indexer.md` rather than failing silently mid-migration.
3. **`make knowledge-reindex`** on first deploy. Subsequent reindexes happen on docs PR merges.
4. **Default backend = FTS.** Operators who want embeddings flip `BSS_KNOWLEDGE_BACKEND=hybrid` and run a reindex (the embedding pass populates the existing rows in place — no schema change).
5. **Catalog roaming flag is purely additive.** Existing `add-offering` invocations without the flag behave unchanged. Existing seeded roaming rows stay valid (idempotent on `id`). The handbook + `add-product-offering.md` get refreshed in the same PR; the SQL workaround section is replaced with the standard flag example, and a `(v0.17–v0.19 historical)` note points readers at the prior approach for reference.
6. **DECISIONS.md entry** dated `2026-05-XX`: "Activate the long-reserved `knowledge` schema with a Tier-0 FTS shape (and Tier-1 pgvector path in same release). Operator cockpit only — customer chat stays scoped. Reindex operator-initiated. Phase docs intentionally not indexed. Bundled-mode Postgres image swapped from `postgres:16-alpine` to `pgvector/pgvector:pg16` (same major; drop-in)." Plus a sibling entry: "Close the v0.17 catalog admin gap by adding `--data-roaming-mb` to `bss admin catalog add-offering`. SQL workaround retired from `add-product-offering.md`."

## What v0.21+ might consider

- **File-watcher reindex** in dev (not prod), gated by `BSS_KNOWLEDGE_DEV_WATCH=true`.
- **Per-section freshness scoring** — DECISIONS.md entries dated > 18 months ago down-weighted in re-ranking.
- **Operator feedback loop** — `bss admin knowledge feedback <chunk_id> --useful=false --reason "outdated since v0.X"` writes a row in `knowledge.chunk_feedback` consumed by the indexer.
- **Public customer knowledge** — separate `customer_self_serve_knowledge` tool against a curated subset of FAQs.
