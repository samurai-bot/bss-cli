# v0.8.0 — Portal authentication foundation (account-first signup)

> **The release that makes the self-serve portal a real surface, not a demo.** v0.4 shipped a portal where every visitor was anonymous and every flow ran through the LLM. v0.8 puts a login wall in front. Email + magic link, server-side sessions, step-up auth for sensitive actions, audit trail per attempt. The signup funnel inverts: account is created first, KYC + customer record are deferred until the moment of first paid order. CSR console is deliberately untouched in v0.8 — it stays on its v0.5 pre-baked-admin pattern. KYC remains the existing mock attestation; this version does not start the Singpass migration.

## Goal

Three deliverables, each small, that together turn the self-serve portal into something a real customer could use:

1. **`packages/bss-portal-auth` — the identity layer.** New workspace package owning email-based identity, login codes, server-side sessions, step-up tokens, rate limiting, audit. Pluggable email adapter (logging-only in v0.8; real SMTP/SES is v1.0). Clean Python API that the portal consumes — no HTTP service of its own.
2. **Self-serve portal becomes login-gated.** Session middleware reads cookies, resolves identities, attaches to `request.state`. New `/auth/*` routes for login / verify / step-up / logout. Existing pre-login surfaces (`/welcome`, `/plans` browse) stay open; everything else gates on session.
3. **Account-first signup funnel.** Visitor verifies email → lands on an empty dashboard → clicks "get a line" → goes through KYC (still mock attestation) + customer creation + order placement, all under the verified session. The `link_to_customer` step happens at the moment of first KYC + customer creation success, in the same transaction.

When v0.8 ships, every page on the self-serve portal is either explicitly public (welcome, plans browse, login, verify) or gated on a verified session. The CSR console works exactly as it did in v0.5 — no auth changes there. KYC and payment remain mock-shaped behind the same providers v0.4 / v0.5 used.

## Deliverables

### Track 1 — `packages/bss-portal-auth`

#### 1.1 Workspace package skeleton

`packages/bss-portal-auth/bss_portal_auth/` with the standard `uv` workspace member shape (mirroring `bss-clock`, `bss-clients`, `bss-middleware`).

- `pyproject.toml` registers the package; `[tool.uv.sources]` workspace-aware.
- Adds itself to root `pyproject.toml` workspace members list.
- Self-serve portal's `pyproject.toml` adds `bss-portal-auth = { workspace = true }` to its dependencies.

#### 1.2 Identity store schema

New schema `portal_auth` (separate from `crm` — auth identity is portal-side, not BSS-core). Migration in `packages/bss-models/alembic/versions/0xxx_portal_auth.py`:

- `portal_auth.identity (id text pk, email text not null, customer_id text null, email_verified_at timestamptz null, status text not null default 'unverified', created_at timestamptz not null default now(), last_login_at timestamptz null)`. Unique partial index on `(email)` where `status != 'deleted'` so the same email can't have two active identities. `customer_id` is NULL until first KYC success and is then linked.
- `portal_auth.login_token (id text pk, identity_id text not null fk, kind text not null, code_hash text not null, issued_at timestamptz not null, expires_at timestamptz not null, consumed_at timestamptz null, ip text null, user_agent text null)`. `kind` ∈ `{magic_link, otp, step_up}`. Index on `(identity_id, kind, consumed_at)` for fast active-token lookup.
- `portal_auth.session (id text pk, identity_id text not null fk, issued_at timestamptz not null, expires_at timestamptz not null, last_seen_at timestamptz not null, ip text null, user_agent text null, revoked_at timestamptz null)`.
- `portal_auth.login_attempt (id bigserial pk, email text null, ip text null, ts timestamptz not null default now(), outcome text not null)`. Append-only, source for rate limiting and audit.

All tables carry `tenant_id` per the existing convention, defaulting to `'DEFAULT'`. (Tenant separation for portal auth comes alongside the rest in Phase 12.)

#### 1.3 Public Python API

`bss_portal_auth/__init__.py` re-exports a small surface:

- `start_email_login(session, email, ip=None, user_agent=None) -> LoginChallenge`
  - Rate-check first (Track 1.4). On exceed → raise `RateLimitExceeded`.
  - Create `identity` row if email isn't known (`status='unverified'`).
  - Mint a 6-digit OTP and a 32-char URL-safe magic-link token (both stored as HMAC-SHA-256 with the server pepper).
  - Insert two `login_token` rows (one OTP, one magic_link), both with the same `expires_at` (default 15 min).
  - Hand the codes to the configured `EmailAdapter.send_login(email, otp, magic_link)`.
  - Return a `LoginChallenge` containing only non-secret state (`identity_id`, `expires_at`).

- `verify_email_login(session, email, code, ip=None, user_agent=None) -> Session | LoginFailed`
  - Look up active tokens for the identity matching `email`.
  - Timing-safe compare against both OTP and magic-link hashes.
  - On success: mark token consumed, mint a `session` row, mark identity `email_verified_at = now()` if first time, update `last_login_at`, write `login_attempt` row with `outcome='success'`, return `Session(id, identity_id, expires_at)`.
  - On failure: write `login_attempt` row with `outcome='wrong_code'` (or `'expired'`, `'no_active_token'`), return `LoginFailed(reason)`. Distinct outcomes are logged but the customer-facing response is intentionally generic ("incorrect or expired code").

- `start_step_up(session, session_id, action_label, ip=None, user_agent=None) -> StepUpChallenge`
  - Issues a fresh OTP scoped to `action_label`.
  - Short expiry (5 minutes). Stored as a `login_token` with `kind='step_up'` and `code_hash`-bound `action_label`.
  - Hand to `EmailAdapter.send_step_up(email, otp, action_label)`.

- `verify_step_up(session, session_id, code, action_label) -> StepUpToken | StepUpFailed`
  - Validates against the stored `step_up` token, scoped to the same `action_label`.
  - On success: returns a one-shot `StepUpToken(token, expires_at, action_label)` valid for the next 60 seconds. The portal uses it as a header / form field on the next sensitive request.

- `consume_step_up_token(session, token, action_label) -> bool`
  - Called by the portal route handler at the moment of a sensitive write.
  - Validates the token matches `action_label`, hasn't expired, hasn't been consumed.
  - Marks consumed atomically. Returns True/False.

- `current_session(session, cookie_value) -> Session | None`
  - Used by middleware. Resolves cookie → session row → identity. Updates `last_seen_at`. Returns None if revoked / expired.

- `revoke_session(session, session_id)` — for explicit logout; sets `revoked_at = now()`.

- `link_to_customer(session, identity_id, customer_id)` — called by the signup flow at the moment of first KYC + customer creation success; sets `identity.customer_id` and `status='registered'`. Idempotent on already-linked.

#### 1.4 Rate limits + brute-force protection

Configurable via env, sane defaults:

- `BSS_PORTAL_LOGIN_RATE_PER_EMAIL=3/15min` — max login starts for one email.
- `BSS_PORTAL_LOGIN_RATE_PER_IP=10/hour` — per-IP cap.
- `BSS_PORTAL_VERIFY_RATE_PER_EMAIL=10/15min` — max code-attempts for one email.
- `BSS_PORTAL_STEPUP_RATE=5/15min` — per-session cap.

Implementation reads `login_attempt` rows for the relevant key + window. Exceeded → `RateLimitExceeded(retry_after_seconds)`. Audit row recorded with `outcome='rate_limited'`.

#### 1.5 Token discipline

- OTP: 6 numeric digits, generated via `secrets.choice('0123456789')` (no `random`).
- Magic link: 32-char URL-safe via `secrets.token_urlsafe(24)`.
- Stored as HMAC-SHA-256 with server-side pepper from env `BSS_PORTAL_TOKEN_PEPPER` (≥32 chars, validated at startup).
- Comparison is `hmac.compare_digest`.
- Pepper rotation runbook documented (Track 4).

#### 1.6 Email adapter interface

`bss_portal_auth/email.py`:

- `class EmailAdapter(Protocol)` with three methods: `send_login(email, otp, magic_link)`, `send_step_up(email, otp, action_label)`, `send_email_change_verification(new_email, otp, magic_link)`.
- `LoggingEmailAdapter` — writes the codes + magic links to stdout (dev) or a file (`BSS_PORTAL_DEV_MAILBOX_PATH`). The default in v0.8.
- `NoopEmailAdapter` — for tests.
- `SmtpEmailAdapter` is **explicitly not implemented in v0.8.** Stub class with a `NotImplementedError` and a comment pointing at v1.0.

The portal selects the adapter via env: `BSS_PORTAL_EMAIL_ADAPTER=logging|noop`. Default `logging`. Anything else raises at startup.

### Track 2 — Self-serve portal session middleware

#### 2.1 `PortalSessionMiddleware`

New file `portals/self-serve/bss_self_serve/middleware/session.py`:

- Reads `bss_portal_session` cookie.
- Resolves to a `Session` via `bss_portal_auth.current_session()`.
- On hit: attaches `request.state.session = Session`, `request.state.identity = Identity` (loaded once), `request.state.customer_id = identity.customer_id` (may be None for unlinked).
- On miss: attaches `None` to all three.
- On request out, sets the cookie with: `HttpOnly`, `Secure` (skipped only when `BSS_PORTAL_DEV_INSECURE_COOKIE=1`), `SameSite=Lax`, `Path=/`, `Max-Age=86400`. Cookie value is the session id (an opaque string), nothing more.
- Sliding expiration: `last_seen_at` updated on every request; when `last_seen_at - issued_at > 24h`, session is rotated (new id minted, old revoked, cookie reissued).
- On step-up success: cookie is rotated immediately to defang fixation.

#### 2.2 Portal main wiring

`portals/self-serve/bss_self_serve/main.py`:

- Mount `PortalSessionMiddleware` after request-id middleware, before route resolution.
- Add a startup hook that calls `bss_portal_auth.validate_pepper_present()` (alongside the existing `validate_api_token_present`).
- Register the dev-mailbox file path if `LoggingEmailAdapter` is selected.

#### 2.3 Route gating helpers

`bss_self_serve/security.py`:

- `requires_session(request) -> Session` — raises `RedirectTo("/auth/login?next=...")` if `request.state.session is None`. Used as a FastAPI dependency.
- `requires_verified_email(request) -> Identity` — same plus `identity.email_verified_at is not None`.
- `requires_linked_customer(request) -> str` — same plus `identity.customer_id is not None`. Returns the `customer_id`. Used by every post-first-order page.
- `requires_step_up(action_label)` — dependency-factory that reads a `X-BSS-StepUp-Token` header (or `step_up_token` form field), calls `consume_step_up_token`. Raises `StepUpRequired(redirect="/auth/step-up?action=...&next=...")` if missing or invalid.

### Track 3 — Auth routes + funnel

#### 3.1 `/auth/*` routes

New module `portals/self-serve/bss_self_serve/routes/auth.py`:

- `GET /auth/login` — email entry form. Carries optional `?next=` for post-login redirect (validated against an allowlist of internal paths).
- `POST /auth/login` — calls `start_email_login`, on success redirects to `GET /auth/check-email?email=<masked>&next=...`.
- `GET /auth/check-email` — interstitial: "We sent a code and a link to your email — click the link or enter the code below."
- `POST /auth/check-email` — accepts the OTP, calls `verify_email_login`, on success sets the session cookie and redirects to `next` (or `/`).
- `GET /auth/verify` — magic-link landing. Reads `?token=...&email=...`, calls `verify_email_login`, redirects similarly.
- `POST /auth/logout` — calls `revoke_session`, clears cookie, redirects to `/welcome`.
- `GET /auth/step-up` — form for step-up OTP entry. Reads `?action=...&next=...`.
- `POST /auth/step-up/start` — calls `start_step_up`. Renders the entry form.
- `POST /auth/step-up` — calls `verify_step_up`. On success, attaches the one-shot token to the session (via a short-lived signed cookie) and redirects to `next`. The next request's route uses `requires_step_up(action_label)` and consumes it.

#### 3.2 Welcome / plans browse — public

- `GET /welcome` — current landing, modified to show a "Sign in / Get started" CTA (both lead to `/auth/login`).
- `GET /plans` — public catalog browse using `catalog.list_active_offerings(now)`. No CTA on a plan card unless the visitor is signed in. If signed in, the CTA is "Choose this plan" and routes to the signup funnel.

These two routes are explicitly excluded from session-required gating.

#### 3.3 Account-first signup funnel

The post-v0.8 signup flow (replacing the v0.4 anonymous-form approach):

1. Visitor at `/plans` clicks "Choose PLAN_M".
2. Redirected to `/auth/login?next=/signup/plan/PLAN_M`.
3. Enters email → magic link / OTP → session created → `email_verified_at` stamped.
4. Lands on `/signup/plan/PLAN_M`. The page shows the plan recap and asks for the KYC mock attestation (same form fields as v0.4: name, NRIC-shaped string, DOB).
5. POST `/signup/plan/PLAN_M`:
   - Validates form.
   - Calls existing mock KYC attestation flow (returns synthesised attestation payload — unchanged from v0.4).
   - Calls `customer.create` via the orchestrator (preserves the v0.4 doctrine that signup writes go through the agent).
   - On customer creation success, calls `link_to_customer(identity_id, customer_id)`.
   - Calls `payment.method_create` via the orchestrator (mock provider, accepts test patterns).
   - Calls `order.create` via the orchestrator → COM/SOM → activation.
   - Renders the v0.4 confirmation card (eSIM QR).

The orchestrator-mediated writes for the *signup funnel itself* are preserved in v0.8 — that's the v0.4 doctrine. v0.10 introduces direct-API for *post-login self-serve* writes (plan change, top-up, etc.); the signup funnel stays orchestrator-mediated through v0.10 too. v0.11 may revisit.

#### 3.4 Empty dashboard for verified-but-unlinked identities

If a visitor verifies their email but bails before completing the signup funnel, they have an `identity` row with `customer_id = NULL`. Visiting `/` in that state shows an empty dashboard: "Welcome — you don't have any lines yet. Get started." The CTA links into `/plans`. No customer record is created lazily.

If they complete signup later, the same `identity` is linked to a freshly-created `customer_id`.

#### 3.5 Login-gated `/`

`GET /` — main dashboard:

- If no session → redirect `/auth/login`.
- If session, identity unlinked → empty-dashboard view (3.4).
- If session, identity linked → full dashboard (rendered by v0.10's pages; in v0.8 it's a placeholder showing the customer's name + "v0.10 will land lines here").

v0.8 deliberately doesn't ship the post-login self-serve UX (lines, balance, top-up, etc.) — that's v0.10. The placeholder confirms session resolution works end-to-end and gives v0.10 a stub to replace.

- Email-as-identity decision (rationale: phone is the product; SMS-OTP login is broken when the line is blocked).

## Doctrine and constraints

1. **Account-first.** Anonymous purchase is not supported. Every paid action requires a verified-email session. The signup funnel flips order: identity → KYC → customer → order. Resist any deliverable that adds a "guest checkout" path.
2. **No passwords.** Magic link and OTP only. Anyone proposing a password field starts with explaining what it's for. Same for "remember this device" — sliding session is the answer.
3. **Server-side sessions.** Cookies carry the session id. The session row is the source of truth for revocation and sliding expiration. No JWT, no localStorage, no client-side session state.
4. **Email is identity; phone is product.** Login is by email, even though the customer's lines have phone numbers. Don't blur the line — it's the same trap real telco UX has been confused about for fifteen years.
5. **Public pages are an explicit allowlist.** `/welcome`, `/plans`, `/auth/*` are the only routes accessible without a session. Everything else returns a redirect to `/auth/login`. Adding a new public route requires an entry in the allowlist file + a test.
6. **Tokens never logged in plaintext.** HMAC-SHA-256 with pepper. The `LoggingEmailAdapter` writes the OTP / magic-link **once** for the customer to see, in the dev-mailbox file; nothing else logs them. Confirm via grep: no `log.info(...otp...)` anywhere.
7. **`bss_portal_auth.current_session` is the only path to identity.** Routes don't read cookies directly. Tests don't fake sessions by setting cookie values — they call `bss_portal_auth.create_test_session(identity_id)` (a helper Track 1 ships).
8. **CSR console untouched.** v0.8 changes self-serve only. CSR auth is a v0.12 concern.
9. **No Singpass, no real SMTP, no real SMS.** Mock adapters only. v1.0 is when real integrations land.
10. **`bss_clock.now()` for everything time-related.** Token expiry, session expiry, `last_seen_at`. The v0.6 doctrine guard still passes.

## Test strategy

1. **Unit tests (`packages/bss-portal-auth/tests/`)**:
   - Token issuance: OTP and magic-link both stored hashed; codes are within format constraints; expiry stamped correctly; consumption is single-use.
   - Verify flow: correct code passes; wrong code fails with `wrong_code`; expired code fails with `expired`; consumed code fails with `already_consumed`.
   - Rate limiting: per-email and per-IP limits trip; `RateLimitExceeded` raised; audit row written.
   - Step-up: token scoped to `action_label`; using a step-up token for a different action fails; consumption is one-shot.
   - Session lifecycle: create → resolve → sliding-rotate at 24h → revoke.
   - Pepper required at startup; `validate_pepper_present()` rejects missing / short.
   - Timing-safe compare confirmed via subroutine inspection.

2. **Portal route tests (`portals/self-serve/tests/test_auth_routes.py`)**:
   - GET `/auth/login` renders.
   - POST `/auth/login` with bad email → form re-renders with error.
   - POST `/auth/login` with valid email → 302 to `/auth/check-email`; LoggingEmailAdapter writes the codes.
   - GET `/auth/verify?token=<from-mailbox>` → 302 to `/`; cookie set.
   - GET `/` without session → 302 to `/auth/login`.
   - GET `/` with session, unlinked identity → empty dashboard renders.
   - POST `/auth/logout` → 302 to `/welcome`; cookie cleared; session row marked revoked.
   - Step-up flow: start → check email → verify → token consumed on subsequent request.
   - `requires_session` dependency: 302 if no session.
   - Public allowlist: `/welcome`, `/plans` accessible without session.

3. **Funnel integration test (`portals/self-serve/tests/test_signup_funnel.py`)**:
   - Visitor verifies email → identity exists, `customer_id NULL`, `email_verified_at` set.
   - Visitor completes signup form → mock KYC + customer.create + link_to_customer + payment.method_create + order.create all run → identity now has `customer_id`, customer record exists, subscription exists.
   - Visitor bails after email verify → identity stays unlinked, no customer record.
   - Visitor completes signup later under the same email → same identity reused, linked to new customer record.

4. **Hero scenarios**:
   - All prior hero scenarios pass three runs in a row. The five v0.4 / v0.5 portal scenarios were anonymous; they need updating to drive the new auth flow. Updates are intentional and reviewed.
   - `portal_self_serve_signup` (v0.4) is rewritten to: register email → verify → choose plan → KYC → place order. Renamed to `portal_self_serve_signup_v0_8.yaml` with the v0.4 file kept for reference until v0.11 cleans up.
   - One new hero scenario: `portal_login_with_step_up.yaml` — drives login → trigger step-up flow → verify step-up. Three runs in a row green.

5. **Doctrine guards**:
   - `rg 'request\.cookies\[' portals/self-serve/` outside of the auth middleware → empty.
   - `rg 'log\.(info|debug|warning).*(otp|magic_link|token)' packages/bss-portal-auth/` → empty.
   - `rg '@app\.get\(.*\).*async def' portals/self-serve/bss_self_serve/routes/` cross-checked against the public-route allowlist.
   - `BSS_PORTAL_TOKEN_PEPPER` validated at startup; missing → portal refuses to boot.
   - `make doctrine-check` green (no `datetime.now()` regressions).

## Out of scope

- **CSR console authentication.** Stays on its v0.5 pre-baked-admin pattern. v0.12 picks this up.
- **Real SMTP / SES email.** `LoggingEmailAdapter` only. v1.0 swaps in real delivery.
- **Real SMS / WhatsApp delivery.** No SMS adapter at all in v0.8. Email is the only verification channel.
- **Social login (Google, Apple).** Out of scope. Maybe v1.x.
- **Passkeys / WebAuthn.** Worth considering for v1.x stickiness, not v0.8.
- **Account recovery for lost-email cases.** If a customer loses access to their email, recovery is manual via human CSR (open a Case). Self-serve recovery is too footgun-prone for v0.8.
- **Multi-factor beyond step-up.** The step-up OTP is the only "second factor"; there's no separate authenticator app, hardware key, or SMS code. Real MFA is a v1.x story tied to OAuth.
- **Account deletion / GDPR right-to-be-forgotten.** Soft-delete via `status='deleted'` is the data model in place; the customer-facing "delete my account" UX is v1.0.
- **Cross-device session list / "where am I logged in?" page.** Sessions exist, but there's no UI to enumerate them. v1.x.
- **Anonymous catalog browse with carted plan selection persisted across login.** v0.8 funnel is "select plan → forced through login → re-pick plan". A pre-login cart that persists is too much funnel state for v0.8.
- **Per-portal token boundary.** That's v0.9. v0.8's portal continues to use `BSS_API_TOKEN` for outbound BSS calls — same as the orchestrator and CSR. Splitting the token is the next version's job.
- **Per-customer rate limits on bss-clients calls.** v0.8 protects login/verify/step-up. Protecting downstream BSS API calls per-customer is a v0.11 chat-scoping concern.
- **Self-serve direct-API writes (top-up, plan change, etc.).** All writes in v0.8 still flow through the orchestrator (the v0.4 doctrine). v0.10 narrows that to chat-only.

## The trap

**Do not let "auth foundation" become "RBAC implementation."** Phase 12 owns OAuth2 / JWT / per-principal roles. v0.8 is one identity model (email-based customer) on one portal (self-serve). Resist the urge to design role hierarchies, tenant boundaries, or per-permission scopes "while you're in there."

**Do not introduce passwords.** Even as an "optional alternative." Even because "some customers will ask for it." Magic-link / OTP login is the right answer for a small MVNO with a tiny ops team — fewer breach vectors, fewer support tickets, simpler runbook. Stick to it.

**Do not let the signup funnel become anonymous-with-deferred-account.** The shape "fill the form → behind the scenes we create the account on submit" looks similar to account-first but loses every benefit (idempotency, identity continuity, abandoned-cart hygiene, regulatory posture). The session must exist *before* the customer commits to a plan.

**Do not store any token in plaintext.** Not in the DB, not in logs, not in step-up tokens passed to subsequent routes. HMAC-SHA-256 with pepper, timing-safe compare, server-side state for everything. The greppable assertions in the test strategy exist to catch any drift.

**Do not skip the public-route allowlist.** It's tempting to just gate `/auth/*` and call everything else session-required. The trap is `/plans` — if a logged-out visitor can't browse the catalog, your funnel becomes "log in to see plans, then choose one" instead of "browse plans → log in to buy." That's a measurable conversion difference.

**Do not let `LoggingEmailAdapter` write to stdout in production.** The dev-mailbox is *deliberately* a file (or stdout in local dev). Production-shape uses `LoggingEmailAdapter` to a structured log path or none at all (a `NoopEmailAdapter` with a "TODO: real SMTP" comment). Real production launch in v1.0 swaps to SMTP/SES — make sure that's the only swap.

**Do not change the CSR console in v0.8.** Its v0.5 pre-baked-admin pattern is fine. Adding portal auth to CSR is a v0.12 concern. If a deliverable here starts touching `portals/csr/`, stop.

**Do not roll session storage into JWT.** JWT requires either short-lived tokens with refresh dance (more state, more code) or accepting non-revocable sessions (worse security than the cookie+row approach). Server-side session rows are the simpler, safer choice for v0.8 scale. JWT can come later when a per-principal claims model demands it (Phase 12).

**Do not let the dev-mailbox file leak codes between test runs.** Tests use `NoopEmailAdapter` and inspect `bss_portal_auth.test_helpers.last_login_codes(email)` — they do not tail the file. The file is for human dev sessions; tests must be hermetic.

**Do not ship without the funnel integration test.** Account-first is the load-bearing UX claim of this version. If the test that proves "verify → bail → return → complete" works under the same identity isn't green and reviewed, the doctrine isn't actually shipped.
