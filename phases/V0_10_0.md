# v0.10.0 — Post-login self-serve pages (direct API path)

> **The thickest portal release.** v0.4 shipped the agent-driven signup demo. v0.8 put it behind a login. v0.10 finally builds out what a real customer does *after* signup: see lines and balances, top up data, manage card-on-file, redownload an eSIM, change plan, cancel, edit contact info, see charge history. Every page is a deterministic, sub-second direct API call — the LLM is not in the path. The doctrine narrows: writes-through-orchestrator was the v0.4–v0.9 rule for portals; v0.10 carves out an exception for authenticated post-login customer self-serve, scoped to the verified session. The chat surface remains orchestrator-mediated and gets its scoping treatment in v0.11. KYC and payment continue to be mocked.

## Goal

Eight customer-facing pages, all post-login, all consuming existing BSS tooling plus the v0.7 catalog/plan-change machinery:

1. **Dashboard** at `/` — lines list with state, plan, balance bars, days remaining, per-line CTAs (top up, show eSIM QR, etc.).
2. **VAS top-up** at `/top-up?subscription=...` — list active VAS offerings, purchase via mock provider, step-up auth required.
3. **Card-on-file management** at `/payment-methods` — list / add / remove with the existing simulated payment provider, step-up auth on writes.
4. **eSIM QR re-download** at `/esim/<service_id>` — read-only; ownership-checked server-side.
5. **Cancel a line** at `/subscription/<id>/cancel` — confirmation page, step-up auth, terminates via existing tool.
6. **Contact medium update** at `/profile/contact` — edit email/phone/address; email change requires re-verification of the new address.
7. **Charge history** at `/billing/history` — read-only paginated list of charges from the payment service.
8. **Plan change** at `/plan/change?subscription=...` — consumes v0.7's `subscription.schedule_plan_change` tool, lists currently-sellable offerings, schedules switch at next renewal, shows pending state and lets the customer cancel.

The doctrine carve-out: every authenticated post-login customer self-serve write goes direct via `bss-clients` from the route handler, scoped to `request.state.customer_id` from the verified session. Reads have always gone direct. The signup funnel (v0.4 / v0.8) and the chat surface (v0.4–v0.11) continue going through the orchestrator.

When v0.10 ships, a real customer can sign up, log in, see their lines, top up data, swap their card, switch plans, cancel a line, and update their email — all without touching the LLM. The chat is still there as an option (orchestrator-mediated, full scope until v0.11 narrows it).

## Deliverables

### Track 1 — Doctrine: writes-through-orchestrator carve-out

#### 1.1 CLAUDE.md update (precedes the code)

Update the v0.4 anti-pattern:

> **(v0.4–v0.9)** Don't write to bss-clients from a portal route handler. Every portal write goes through the LLM orchestrator via `agent_bridge.*` → `astream_once`. Reads can go direct.

becomes:

> **(v0.4–v0.9 / pre-login + signup + chat)** The signup funnel and the chat surface go through the orchestrator via `agent_bridge.*` → `astream_once`. Reads can go direct.
>
> **(v0.10+ / authenticated post-login customer self-serve)** Routes that require a verified linked-customer session may write directly via `bss-clients`. The customer's principal is bound from `request.state.customer_id`; policies enforce ownership server-side; step-up auth gates sensitive writes. The orchestrator path remains the only path for the chat surface and the signup funnel.

This is a real doctrine shift. It's narrower than "everything direct" — it explicitly preserves orchestrator-mediation for the surfaces where the LLM is a feature, not a tax. The change reflects the v0.4 era's purpose (showing the agent pattern works) vs. v0.10's purpose (a customer can use the product daily).

#### 1.2 Anti-patterns added

- `(v0.10+) Don't accept user-controllable customer_id / subscription_id / service_id in any post-login route handler. Read customer_id from request.state.customer_id; resolve subscription/service ownership via policies before any write.`
- `(v0.10+) Don't compose multiple post-login writes in a single route handler without a clear rollback story. Each route is one bss-clients write call (or zero); compositions belong in the orchestrator.`
- `(v0.10+) Don't bypass step-up auth on any sensitive write. The list (add/remove COF, terminate, change email, plan change) is enforced via dependency, not just convention.`

### Track 3 — VAS top-up at `/top-up`

#### 3.1 Route + template

`routes/top_up.py`:

- `GET /top-up?subscription=SUB-007` — validates the subscription belongs to `request.state.customer_id` (server-side via policy), lists VAS offerings via `vas_client.list_offerings()`, renders selection cards.
- `POST /top-up` — body: `subscription_id`, `vas_offering_id`. Requires step-up auth.
  - Calls `subscription_client.purchase_vas(subscription_id, vas_offering_id, payment_method_id=customer's_default_cof)` directly via `bss-clients`. No orchestrator.
  - On success → 302 to `/top-up/success?subscription=...&vas=...`.
  - On policy violation → re-render form with the structured error message.

#### 3.2 Pre-selection by state

If the visitor lands with `?subscription=SUB-X&context=blocked-data`, the page pre-selects the relevant data top-up. The state is passed by the dashboard's "Top up to unblock" CTA.

#### 3.3 Step-up gate

The `POST /top-up` route includes the v0.8 `requires_step_up("vas_purchase")` dependency. If no step-up token, redirect to `/auth/step-up?action=vas_purchase&next=/top-up?subscription=...`. The customer enters a fresh OTP, the token is consumed on the subsequent submit.

### Track 4 — Card-on-file at `/payment-methods` (mock provider)

#### 4.1 List + add + remove routes

`routes/payment_methods.py`:

- `GET /payment-methods` — `payment_client.list_for_customer(customer_id)`. Renders existing methods (last-4 only), default toggle, "Add new" CTA.
- `GET /payment-methods/add` — form: card number, expiry, CVV, holder name.
- `POST /payment-methods/add` — step-up required. Calls `payment_client.create_method(customer_id, card_data)` against the simulated provider. Test patterns (`4242...` etc.) succeed; others fail with a structured policy error.
- `POST /payment-methods/<id>/remove` — step-up required. Calls `payment_client.remove_method(customer_id, method_id)`. Existing payment policy refuses to remove the only method for a customer with active lines; that error is surfaced cleanly.
- `POST /payment-methods/<id>/set-default` — step-up required. Calls `payment_client.set_default_method(customer_id, method_id)`.

#### 4.2 Form fields are Stripe-shaped

The form deliberately captures the fields a real Stripe Elements integration will: card number, expiry month/year, CVV, holder name, postal code. The v0.10 backend ignores the postal code (mock provider doesn't use it); v1.0 wires it through. The UX shape doesn't change at the v1.0 cutover.

#### 4.3 Receipt of new method → the Stripe-shaped seam

After a successful add, the page renders a confirmation card and a small "About your card" panel that v1.0 will use to show real Stripe details (last-4, brand, expiry). v0.10 shows the simulated equivalents.

### Track 5 — eSIM re-download at `/esim/<service_id>`

#### 5.1 Route + template

`routes/esim.py`:

- `GET /esim/<service_id>` — server-side ownership check: load the service, assert its parent subscription's `customer_id == request.state.customer_id`. On mismatch → 403 (not 404, because the service exists; the customer just doesn't own it).
- Renders the LPA activation code as a copyable string, an inline PNG QR (reuse `qrpng.py` from v0.4), and a caption: "Scan with your phone's eSIM camera to install. Use this code if you've changed devices or factory-reset."
- ICCID / IMSI shown as last-4 only by default; `?show_full=1` reveals full values for ops debugging (admin-role only — checked via `auth_context`).

#### 5.2 Read-only

No state changes. eSIM redownload is idempotent — the LPA code is the same one minted at activation. No need for step-up.

### Track 6 — Cancel a line at `/subscription/<id>/cancel`

#### 6.1 Confirmation page

`routes/cancel.py`:

- `GET /subscription/<id>/cancel` — ownership check, then renders a confirmation page that explicitly enumerates what's lost:
  - Days remaining on current period (no proration, no refund).
  - Bundle balance (will be discarded).
  - eSIM profile (will be released back to inventory).
  - MSISDN (will be released back to inventory; not retainable).
- "I understand, cancel my line" button.

#### 6.2 Cancel submit

- `POST /subscription/<id>/cancel` — step-up required. Calls `subscription_client.terminate(subscription_id, reason="customer_requested")` directly. Existing terminate flow handles inventory release + payment-state updates + audit.
- On success → 302 to `/subscription/<id>/cancelled` showing a confirmation card.

### Track 7 — Contact medium update at `/profile/contact`

#### 7.1 List + edit forms

`routes/profile.py`:

- `GET /profile/contact` — `customer_client.list_contact_mediums(customer_id)`. Renders editable fields per medium.
- `POST /profile/contact/email/change` — step-up required. New email entered.
  - Triggers the v0.8 `start_email_login` against the **new** email — sets up email-change verification.
  - Records a pending email-change row in `portal_auth.email_change_pending(identity_id, new_email, expires_at, status)`.
  - Customer receives a verification code at the new email; clicks the magic link or enters the code at `/profile/contact/email/verify`.
  - On verification: atomically: (a) update `customer.contact_medium` for the email (CRM write), (b) update `portal_auth.identity.email`. Commit. Clear pending row. Old email is replaced; old session remains valid until logout.
- `POST /profile/contact/phone/update` — step-up required. Direct call to `customer_client.update_contact_medium(customer_id, medium_id, value)`. (Phone numbers don't have a verification flow in v0.10 — they're informational; the customer's lines are MSISDNs, separate concept.)
- `POST /profile/contact/address/update` — step-up required. Same pattern.

#### 7.2 Idempotent email-change

If the customer starts an email change but doesn't complete verification, the old email keeps working. The pending row expires after 24 hours. They can restart the flow.

If they verify before the pending row expires but later regret it, that's a CSR ticket — no self-serve "undo email change" in v0.10.

### Track 8 — Charge history at `/billing/history`

#### 8.1 Route + template

`routes/billing.py`:

- Pagination controls (prev / next).
- Link per row to a small detail panel showing the charge ID, retry count if any, the underlying subscription / VAS reference.

#### 8.2 Read-only

No exports, no PDFs, no CSV download in v0.10. Display only.

### Track 9 — Plan change at `/plan/change`

  - Renders one card per sellable offering: name, price, allowances, "Switch at next renewal on <date>" CTA.

#### 9.2 Schedule submit

## Doctrine and constraints

1. **Doctrine carve-out is narrow.** Authenticated post-login customer self-serve writes go direct via `bss-clients`. Signup funnel and chat continue going through the orchestrator. If a deliverable creates a third category, stop and ask.
2. **`customer_id` is bound from `request.state`, never from a request parameter.** Greppable via `rg 'customer_id\s*=\s*(form|body|query|path)' portals/self-serve/`.
3. **Server-side ownership checks for every cross-resource access.** A subscription_id, service_id, payment_method_id passed by the customer must be checked against the verified principal before any read/write. Policies own this; route handlers don't reimplement it.
4. **Step-up auth gates every sensitive write.** The greppable catalogue in `security.py` is the source of truth. Adding a new sensitive route requires adding to the catalogue.
5. **No composition.** One route, one BSS write call (or zero). If a flow needs two writes, it's a workflow concern that belongs in the orchestrator or in a service-side composite operation, not in a route handler.
6. **Reads are read; writes are write.** No "read-then-write" route handlers that race. Concurrency safety lives in the policy + repository layer (where it already does).
7. **No proration on plan change.** Confirmation page says so explicitly. The v0.7 machinery enforces it.
8. **No real Stripe, no real Singpass.** Mock provider for COF; mock attestation for KYC. v1.0 swaps both behind unchanged UX shapes.
9. **Email is identity; phone is product.** Email change requires re-verification; phone update is a contact-medium write (no flow). MSISDNs (the actual phone numbers attached to lines) are inventory, not contact mediums.
10. **`bss_clock.now()` everywhere; `bss_portal_auth` for all session reads.**
11. **CSR console untouched.** v0.12 still owns CSR auth + post-login changes there.
12. **Chat surface untouched in v0.10.** v0.11 owns the scoping + escalation. The `/chat` route in self-serve continues to call `astream_once` with the v0.9 service identity but no tool filter yet.

## Test strategy

1. **Unit tests per route**:
   - Each route's ownership policy: known-good owner passes; cross-customer forbidden.
   - Step-up dependency: missing token → redirect; valid token → consumed.
   - Form validation: bad inputs return form re-render with errors.

2. **Integration tests** (require running stack + frozen clock):
   - Top-up flow: balance before < balance after; `usage.recorded` doesn't fire (top-up is a balance grant); audit rows match.
   - COF add → set as default → remove → policy refuses to remove last with active lines.
   - eSIM redownload: known good service → 200; cross-customer → 403 (not 404).
   - Cancel → state transitions to terminated; eSIM + MSISDN released back to inventory; subsequent reads of the subscription show terminated.
   - Email change two-step: starts pending → new email receives token → verify → CRM email updated AND portal_auth.identity.email updated atomically (assertion: rollback the verify and confirm both stay un-updated).
   - Charge history: paginates correctly; only the requesting customer's charges appear.
   - Plan change schedule → renewal trigger → balance + plan flip happens at the renewal boundary, not before.
   - Plan change cancel: pending fields cleared; renewal applies the original offering.

3. **Cross-customer attempt suite**:
   - For every sensitive route, attempt to act on customer B's resource while logged in as customer A. Every such attempt returns 403 with a structured error and writes a `portal_action` audit row with `success=false, error_rule=...`. Greppable test that all sensitive routes have this test variant.

4. **Hero scenarios**:
   - All prior hero scenarios pass three runs in a row.
   - New `portal_post_login_self_serve.yaml` passes three runs in a row.
   - The v0.7 `catalog_versioning_and_plan_change.yaml` continues to pass — v0.10 doesn't change the underlying machinery.

5. **Doctrine guards**:
   - `rg 'customer_id\s*=\s*(form|body|query|path|request\.json)' portals/self-serve/bss_self_serve/routes/` → empty.
   - `rg 'astream_once' portals/self-serve/bss_self_serve/routes/` matches only the chat route + signup funnel; NOT the post-login routes.
   - `rg 'requires_step_up\(' portals/self-serve/bss_self_serve/routes/` cross-checked against `SENSITIVE_ACTION_LABELS` — every label appears at least once and every call site uses a label from the set.
   - `make doctrine-check` green.

## Out of scope

- **Real Stripe / Singpass.** Mocks remain. v1.0 swap.
- **CSR console post-login features.** v0.10 is self-serve only. CSR's v0.5 pattern stays untouched.
- **Chat surface scoping.** v0.11. The chat box in v0.10 is functional but full-tool-access (the v0.4 / v0.5 baseline).
- **PDF invoices, CSV exports, export to mail.** Charge history is on-screen only.
- **Plan change that takes effect immediately.** Doctrine. Next-renewal only.
- **Adding a second line via portal post-login.** v0.10 dashboard shows lines. Adding a new line goes through `/plans` → signup funnel (orchestrator-mediated). A future "add another line" flow could go direct, but that's a v0.12+ decision.
- **Voicemail / SMS / data add-ons that aren't in the existing VAS catalogue.** The VAS list comes from catalog; no new VAS types in v0.10.
- **Payment retry ladder / dunning.** Doctrine: failed charge = blocked. No retry.
- **Customer can change their own NRIC / KYC details.** No. Identity changes are CSR-mediated tickets.
- **Account closure / "delete me" / GDPR right-to-be-forgotten UX.** v1.x.
- **Mobile-optimised layout.** v0.10 is desktop-shaped. Mobile responsive comes with the v1.0 polish.
- **Localisation / multi-language.** English only.
- **Dark mode toggle.** v0.6 renderers were dark-themed; portals are light-themed. Toggle is a v1.x cosmetic.
- **Notifications / push / SMS reminders.** No outbound communication in v0.10 except email-change verification.
- **A/B tests on UI variants.** Not v0.10.
- **WebSocket / push beyond email-change SSE.** No new realtime channels.
- **Programmatic API for partners.** Customer portal only. Partner APIs are v1.x.
- **Adding new lines to existing customers via direct API in the portal.** That goes through signup funnel (orchestrator-mediated) — single path for "new line" continues.

## The trap

**Do not let the doctrine carve-out spread.** The v0.10 narrowing is precise: authenticated post-login customer self-serve. Not signup. Not chat. Not CSR. Not partner. If a future deliverable says "make signup direct because it'd be faster," that's a different doctrine conversation — it doesn't ride on the v0.10 carve-out.

**Do not accept `customer_id` as a route parameter.** Even "for backward compat with the v0.4 demo URLs." Even "to make tests easier." The customer principal comes from the verified session, full stop. Greppable guard exists for a reason; treat its findings as P0.

**Do not skip step-up for "low-risk" writes.** Phone update feels low-risk until a fraudster combines a session takeover with a phone update to misroute SMS-based 2FA somewhere else. The catalogue is comprehensive deliberately. Don't trim it.

**Do not compose multiple writes in a route handler.** Top-up = one purchase_vas call. Plan change = one schedule_plan_change. If a workflow demands "do A, then if A succeeds do B, else compensate" — that's an orchestrator concern. Route handlers that compose are how data drift sneaks in (the kind that's only visible when one of the writes fails between the others).

**Do not silently downgrade the email-change flow's atomicity.** The CRM email update and the `portal_auth.identity.email` update must commit together or roll back together. If you implement them as two sequential commits, you'll occasionally end up with mismatched states and no easy way to detect or recover. A small two-phase / saga pattern (or a database transaction spanning both schemas — they're in the same Postgres) is the right answer. Don't shortcut.

**Do not let the dashboard's "balance bars" be flat blocks.** The v0.6 renderer polish established that proportional rendering matters for the legibility story. Carry it through to HTML — CSS gradients with the same proportional widths. A `<div style="width: 17%">` that just renders a small block reads worse than the v0.6 ASCII version did. Match the bar.

**Do not let the eSIM admin-only `?show_full=1` flag slip into customer-accessible code paths.** It's there for ops debugging only. The auth_context check must reject non-admin principals; the route must not have a fallback that "shows full if no admin context found." Test the negative path.

**Do not bypass the cross-customer attempt suite.** Every sensitive route gets an explicit "try as the wrong customer" test. It's tedious; it's load-bearing. The day a policy bug ships and customer A acts on customer B's resource, this suite is what saves you.

**Do not let "the chat is still there" turn into "the chat is somehow still our recommendation for top-up."** The chat box stays for general questions and exploratory flows. The dashboard's "Top up" CTA goes direct, fast, deterministic. Conversational top-ups via the chat work too (existing tools), but the dashboard is the primary path. Make sure the UX leads with the direct flow.

**Do not let plan change forget pending state.** When a customer schedules a switch, the dashboard must surface the pending state every time they look at the line — not just once on the confirmation page. The `pending_renewal_with_pending_plan_change` CTA branch in the dashboard spec exists because customers genuinely forget; make sure they see it on every visit.

**Do not consider the email-change flow done until the rollback test passes.** Plant a synthetic failure at the moment between the CRM commit and the portal_auth commit; assert the system is in a recoverable state (not a half-committed mess). If you skip this test, the doctrine "atomic" claim is just words.
