# v0.4.0 — Self-serve customer portal

> **Demo-grade web surface that amplifies the LLM story.** A small FastAPI + Jinja + HTMX portal that walks a prospect through `browse → signup → activate` and visibly routes every write through the LangGraph orchestrator. The hero artifact is not the signup form — it's the "Agent Activity" log widget on the side, streaming the LLM's tool calls in real time so the viewer understands the portal is a thin skin over the agent, not a parallel backend path.

## Goal

Prove two things with one small portal:

1. **The CLI-native agent stack is web-addressable.** A customer can, from a browser, complete the same signup flow that `bss scenario run customer_signup_and_exhaust.yaml` drives — three plans, customer creation, card on file, order, activation, eSIM QR. Desktop-first, no auth, no styling pageantry.
2. **Every write in the portal goes through the LLM orchestrator.** The portal's backend does not call `CustomerClient.create` directly. It constructs a natural-language instruction (`"create a customer named Ck with email ... on PLAN_M and add card 4242..."`) and passes it to `bss_orchestrator.session.astream_once()`. The agent plans the tool chain, fires the calls, and streams events back. The portal renders the resulting state change into the HTML response AND streams the agent's tool-call sequence into a visible side-panel log widget.

The log widget is the v0.4 demo artifact — the screenshot equivalent to the v0.2 swimlane. If you strip it away, the portal looks like any other CRUD web app. With it, the viewer sees the LLM doing the work, tool call by tool call, in the same page they're submitting a form in. **That's the story.** Do not hide the widget, do not relegate it to a debug panel, do not make it opt-in.

Non-goals: a production customer portal, mobile layout, real auth, payment gateway integration, email/SMS verification, account management post-signup. v0.4 is a demo. v0.5 reuses the stack for the CSR console.

### 3. The agent bridge — `bss_self_serve/agent_bridge.py`

Single-purpose module: construct a natural-language prompt from structured portal input, pass it to the orchestrator's streaming entry point, and yield the stream of events.

```python
# bss_self_serve/agent_bridge.py

from bss_orchestrator.session import astream_once  # existing or new (see §3a)

async def drive_signup(name: str, email: str, plan: str, card_pan: str, phone: str):
    prompt = (
        f"Create a new customer named '{name}' with email '{email}' and phone '{phone}'. "
        f"Attest their KYC using the pre-verified Myinfo attestation (channel=portal-self-serve). "
        f"Add card {card_pan} as their payment method on file. "
        f"Create an order for offering '{plan}'. "
        f"Wait for the order to complete and return the resulting subscription ID and eSIM activation code."
    )
    async for event in astream_once(prompt, allow_destructive=False, channel="portal-self-serve"):
        yield event
```

Everything the portal does to the business domain goes through this function (or a sibling function per flow). **No direct `bss-clients` writes.** Reads (fetching plan details for the landing page, fetching a completed subscription for the confirmation page) can go through `bss-clients` directly — those are non-mutating and the agent layer adds no value.

Write surfaces that call the agent:
- `POST /signup` → `drive_signup(...)` — the full flow
- (v0.4 has no other write surfaces — no account management, no plan changes, no VAS top-ups on the self-serve portal)

Read surfaces that go direct:
- `catalog.list_offerings()` on the landing page
- `subscription.get()` on the confirmation page (polling for activation)
- `order.get()` on the activation progress page

#### 3a. Orchestrator streaming entry point

v0.1 shipped `bss_orchestrator.session.ask_once(prompt)` which returns the final answer. v0.4 needs **streaming** so the log widget can show tool calls as they happen. If `astream_once` doesn't already exist as a public API (reconnaissance: the REPL consumes the graph's astream directly), v0.4 adds it in `orchestrator/bss_orchestrator/session.py`:

```python
async def astream_once(prompt: str, *, allow_destructive: bool = False, channel: str = "cli") -> AsyncIterator[AgentEvent]:
    """Streaming variant of ask_once. Yields typed AgentEvent objects:
       AgentEventPromptReceived, AgentEventToolCallStarted,
       AgentEventToolCallCompleted, AgentEventFinalMessage."""
```

Same contract the REPL already uses internally, lifted to a public function. The CLI's `bss ask --stream` (if it exists) uses the same function — single source of truth.

### 4. The agent log widget — `templates/partials/agent_log.html`

**The v0.4 hero artifact.** A side-panel on every page of the portal, always visible (not a collapsible drawer, not a debug overlay). Initial state shows an explanatory line so viewers who haven't submitted anything yet understand what they're looking at.

Visual target:

```
┌─ Agent Activity ─────────────────────────────────┐
│                                                   │
│  The portal doesn't call the backend directly.    │
│  It sends natural language to an LLM agent that   │
│  plans and executes tool calls. Submit the form   │
│  to watch it work.                                 │
│                                                   │
│  ─────────────────────────────────────────────    │
│                                                   │
│  → prompt: "Create a new customer named Ck       │
│            with email ck@example.com on PLAN_M..." │
│                                                   │
│  ↳ customer.create(name="Ck", email="ck@...")     │
│    ← CUST-042                                     │
│                                                   │
│  ↳ customer.attest_kyc(customer_id="CUST-042")    │
│    ← ok                                           │
│                                                   │
│  ↳ payment.add_card(customer="CUST-042",          │
│                     card_last4="4242")            │
│    ← PM-018                                       │
│                                                   │
│  ↳ order.create(customer="CUST-042",              │
│                 offering="PLAN_M")                │
│    ← ORD-014 (acknowledged)                       │
│                                                   │
│  [ streaming... ]                                 │
│                                                   │
└───────────────────────────────────────────────────┘
```

Rendering rules:

- Monospace font. Terminal-adjacent palette. Not the body font of the rest of the page.
- Each event is one row (or a small group: `tool X → result Y`)
- Icons: `→` for received prompt, `↳` for tool call started, `←` for tool result, `✓` for final success, `⚠` for error
- Tool args are truncated to 80 chars; `hover to reveal full` via a simple CSS tooltip, no JS framework
- Widget shows maximum 50 most-recent events; older events scroll out (the full stream is accessible via `bss trace` from v0.2 if anyone cares)
- Connection status indicator in the header: `● live` (green) while SSE is open, `○ done` when the stream closes, `⚠ disconnected` if SSE errors

Implementation:

- `GET /agent/events/{session_id}` returns `text/event-stream`. The server-side handler consumes `astream_once(...)` and yields `data: <json>\n\n` frames per event.
- The `signup.html` page includes `<div hx-ext="sse" hx-sse="connect:/agent/events/{{ session_id }}"><div hx-sse="swap:message" hx-target="#agent-log" hx-swap="beforeend"></div></div>` — pure HTMX, no bespoke JavaScript.
- Each event frame is a pre-rendered HTML partial (`agent_event.html`), not raw JSON. Server-side rendering keeps the frontend dumb — no client-side JSON parsing, no template engine in the browser.

### 5. The flow — 4 pages

#### 5a. Landing — `GET /`

Three plan cards rendered from `catalog.list_offerings()`. Each card has:
- Plan name (S / M / L)
- Monthly price (SGD inclusive)
- Data allowance (in GB)
- Voice allowance (in minutes)
- "Pick this plan" button → navigates to `/signup/{plan_id}`

The agent log widget is visible but inactive ("Submit the form to watch the agent work"). No signup has happened yet.

#### 5b. Signup — `GET /signup/{plan}`, `POST /signup`

One form, three fields:
- Name (text, required)
- Email (email, required)
- Phone (tel, required, defaults to +65 prefix)
- Card number (text, required, the mock tokenizer accepts any Luhn-valid PAN — `.env.example` documents the `4242 4242 4242 4242` test card)
- **KYC** section: a **pre-verified badge** (not a file upload):
  ```
  ✓ Identity verified via Myinfo (simulated)
    Attestation ID: KYC-PREBAKED-001 · Status: valid · Expires: 2027-04-20
  ```
  Clicking "Edit" does nothing in v0.4 — it's a static badge. The backend wires `KYC-PREBAKED-001` as a known-good attestation signature that `customer.attest_kyc` accepts.

On submit:
1. Generate a `session_id` (UUID) and stash the form input in `session.py`'s in-memory store
2. Redirect to `/signup/{plan}/progress?session=<session_id>`
3. That page opens the SSE stream via HTMX; the server-side handler fires `agent_bridge.drive_signup(...)` and relays events
4. When the agent emits `AgentEventFinalMessage`, the progress page swaps itself for a `/activation/{order_id}` redirect

No database writes on the server-side of the portal. The agent is the only path.

#### 5c. Activation — `GET /activation/{order_id}`

Progress page. Shows:
- Order state (acknowledged → in_progress → completed) as a simple stepper
- Decomposition summary: "Reserving MSISDN... Provisioning HLR... Preparing eSIM profile... Activating subscription..."
- A live "still running" spinner until `order.wait_until(state=completed)` resolves
- The agent log widget continues to stream (the agent is still chained through the rest of the order→subscription lookup flow)

Polling strategy: HTMX `hx-trigger="every 1s"` re-fetches a tiny status partial. Under the hood it calls `order.get(order_id)` directly (read path) until state is `completed`, then swaps in the confirmation page.

#### 5d. Confirmation — `GET /confirmation/{subscription_id}`

Final page. Shows:
- Activation success message
- **eSIM QR code** rendered as a real PNG via the `qrcode` Python lib (not ASCII — this is the one concession to "being a web page"). Served inline as `data:image/png;base64,...` to keep the deploy-size small.
- The LPA activation code as a copy-to-clipboard string: `LPA:1$<SMDP+>$<activation-code>`
- Plan summary (name, allowance, next renewal)
- "Create another" link back to landing
- Agent log widget shows the full completed transcript

No account management. No "update plan" button. No "cancel subscription" button. The portal is one-shot signup, by design.

### 8. Telemetry + auth wiring

- `configure_telemetry(service_name="portal-self-serve")` in lifespan (v0.2 pattern)
- `BSSApiTokenMiddleware` NOT on the portal itself — the portal is a public-facing surface, unauthenticated by design (it's a signup page)
- But the portal's **outbound** calls to services must carry `BSS_API_TOKEN` (v0.3 `TokenAuthProvider` on its bss-clients instance) — same wiring as the CLI
- `X-BSS-Channel: portal-self-serve` on every outbound call — gives CRM's interaction auto-log the right channel label

The portal's own HTTP surface is open. Anything it does to BSS services is token-authenticated on the outbound side.

### 9. Hero scenario — `scenarios/portal_self_serve_signup.yaml`

Fifth hero scenario. Tagged `hero, portal`. Does not use the YAML action language — drives the portal via HTTP and then asserts state via the CLI's existing tools.

```yaml
name: portal_self_serve_signup
tags: [hero, portal, regression]

setup:
  reset_operational_data: true
  freeze_clock_at: "2026-05-05T09:00:00+08:00"

steps:
  # Portal-driven signup — the scenario runner grows a minimal HTTP step type
  - name: visit landing page
    http: GET http://portal-self-serve:8000/
    expect:
      status: 200
      body_contains: ["PLAN_S", "PLAN_M", "PLAN_L"]

  - name: submit signup form
    http: POST http://portal-self-serve:8000/signup
    form:
      plan: PLAN_M
      name: "Portal Demo Customer"
      email: "portal-demo-{{ run_id }}@bss-cli.local"
      phone: "+6590009999"
      card_pan: "4242424242424242"
    expect:
      status: 303        # redirect to progress page
    capture:
      session_id: "$.headers.Location | regex:session=([^&]+)"

  - name: wait for activation
    http: GET http://portal-self-serve:8000/activation/wait?session={{ session_id }}
    poll:
      interval_ms: 1000
      timeout_seconds: 30
      until: body_contains("PLAN_M ACTIVE")
    capture:
      subscription_id: "$.headers.X-Subscription-Id"

  # Backend-side verification — not via the portal
  - action: subscription.get
    args: { subscription_id: "{{ subscription_id }}" }
    expect:
      state: active
      offering_id: PLAN_M
      channel_created: portal-self-serve

  - action: interaction.list
    args: { customer_id: "{{ customer_id }}" }
    expect:
      any_match:
        channel: portal-self-serve
        type: order.create
```

Three passes in a row before tagging `v0.4.0`. Promoted into `make scenarios-hero`.

Scenario runner needs a tiny extension: HTTP step type (`http: GET|POST <url>`, with `form:`, `expect:`, `poll:`, `capture:` sub-fields). That extension is part of v0.4 deliverables, in `cli/bss_cli/scenarios/actions.py`.

## Doctrine and constraints

1. **Every write goes through the orchestrator.** No `CustomerClient.create()`, no `OrderClient.create()`, no policy-gated write calls from portal route handlers. The only exception is health/read endpoints, which go direct. Grep guard enforces it (see §verification).
2. **Reads go direct via bss-clients.** LLM-mediating a `catalog.list_offerings` call for the landing page would be silly — 500 ms of LLM latency to return a 3-row table that never changes. Reads are pass-through.
3. **No auth on the portal.** It's a demo. Anyone hitting the URL can create a fake customer. In production this would sit behind customer-facing auth (OAuth with Google/Apple Sign-In, SMS OTP, whatever) — all channel-layer concerns, all out of scope.
4. **Desktop-first.** No media queries for mobile. If someone opens the portal on a phone, it'll render awkwardly, and that's fine.
5. **KYC is pre-baked.** No file upload. No liveness check. No document OCR. The portal fires `customer.attest_kyc` with a hardcoded "simulated Myinfo" signature that the CRM accepts in dev mode. The attestation ID `KYC-PREBAKED-001` is documented in `services/crm/bss_seed/kyc_prebaked.py`.
6. **Payment is the mock tokenizer.** Same tokenizer the CLI uses. `4242 4242 4242 4242` always approves. Any other Luhn-valid PAN also approves (it's a sandbox). Real payment gateway is out of scope.
7. **One-shot signup.** No account management after confirmation. No persistent customer session. No "log back in to see my plan". If they refresh after confirmation, it's gone. That's a v0.5+ concern (CSR console adds customer-360 view, not the self-serve portal).
8. **Pure server-rendered HTML + HTMX.** No React. No Vue. No Svelte. No SPA. No bundler. No node toolchain. Single vendored `htmx.min.js`, one CSS file, that's it.
9. **SSE over WebSockets.** Simpler, one-way (server → client), fits behind any HTTP proxy, doesn't need extra middleware. The only thing the client needs to stream is the agent log.
10. **Portal runs in the same budget.** Adding the portal is ~150 MB RAM (uvicorn + FastAPI + Jinja, same footprint as a BSS service). Total bundled-infra stack with Jaeger + portal: ~3.2 GB. Still under 4 GB.
11. **Campaign OS untouched.** Verify by grep.
12. **Graceful when LLM is slow.** MiMo v2 Flash completes a signup chain in 10-20s; a worse model could take 60s. The activation page's spinner + SSE live stream should make the wait feel intentional, not broken. No timeout-and-retry on the portal side; `astream_once` either completes or fails, the portal shows the failure in the log widget, and the user hits back.

## Security model

v0.4 adds a public-shaped web surface. Before tagging, be honest about what it protects and what it doesn't — because the portal is a demo, not a production customer portal.

**What the agent can do for any reaching prompt.** `agent_bridge.drive_signup` invokes `astream_once` which binds every tool in the registry minus the destructive ones (v0.1's `--allow-destructive=false` gate blocks `subscription.terminate`, `customer.close`, `ticket.cancel`, `payment.remove_method`, `order.cancel`). Every remaining tool carries `BSS_API_TOKEN` on outbound calls. **Effectively: any prompt reaching the agent via the portal can do anything a non-destructive admin can do** — create customers, add cards, create orders, trigger activations, purchase VAS. The `BSS_API_TOKEN` (v0.3) protects services from *direct* access; it does NOT protect the agent from being *invoked* via the portal.

**Who can invoke the agent through the portal.** Anyone who can POST to `/signup` on port 9001. The self-serve portal has no auth at its front door — by design. The signup form IS the feature.

**What actually keeps the demo safe.** Network-level exposure control. Nothing else. If port 9001 is published only on `localhost` or a Tailscale interface, only someone on that network can reach it. If it ever lands on a public IP without mitigations, expect signup spam (filling the DB), LLM-budget drain (exhausting OpenRouter credits), and prompt-injection attempts against the agent.

**What's deferred to Phase 12 and future hardening.** Rate limiting per source IP, captcha / bot protection, prompt-input content filters, per-session LLM-spend caps, customer-facing OAuth (Apple/Google Sign-In, SMS OTP), CSRF tokens on form submission. None of this exists in v0.4. All of it is channel-layer concern territory.

**What you must NOT do before Phase 12:**

- Do not publish port 9001 on a public IP.
- Do not run the demo on a shared host where untrusted users have access to `localhost:9001`.
- Do not remove the `allow_destructive=False` pin on `astream_once` in `agent_bridge.py`. It's the one in-band guardrail still standing.
- Do not assume the mock tokenizer makes abuse harmless — spammed DB rows and an exhausted LLM budget are real damage regardless.

**The right mental model:** v0.4's portal is a demo running on a trusted network. Treat it like an unsecured `jupyter notebook` — fine on your LAN/Tailscale, never on the open internet.

## Test strategy

1. **Unit — `portals/self-serve/tests/test_routes_unit.py`.** Each route tested with a mocked `agent_bridge.drive_signup` that yields a canned event stream. Asserts templates render, HTMX targets exist, SSE headers set.
2. **Integration — `portals/self-serve/tests/test_signup_integration.py`.** Full stack via `httpx.AsyncClient` with `ASGITransport` against the portal app. Mocks `astream_once` at the orchestrator entry point (no real LLM, no real services) — this proves the portal wiring, not the orchestrator's model-dependent behavior.
3. **SSE test — `portals/self-serve/tests/test_agent_log_widget.py`.** Asserts the event stream yields well-formed `text/event-stream` frames with HTML partials (not JSON).
4. **Hero scenario — `scenarios/portal_self_serve_signup.yaml`.** End-to-end against the real compose stack with a real LLM. Covers: HTTP step type in scenario runner, full signup flow, backend state verification, interaction-log channel attribution.
5. **v0.1-v0.3 hero scenarios still pass unchanged.** The portal is additive; nothing else changes.

## Out of scope

- **Analytics pixels, GTM, tracking scripts.** None.

## The trap

**Do not let the portal become a production customer portal.** The moment you're designing "forgot password" flows, mobile breakpoints, or loyalty program banners, you've lost the thread. v0.4 is a demo of the LLM-agent channel pattern. Complexity in the frontend dilutes the story that the LLM is doing the work.

**Do not call `CustomerClient.create`, `OrderClient.create`, or any policy-gated write method from a route handler.** If you do, you have two parallel write paths: the LLM agent (via the CLI) and the portal (via bss-clients directly). They'll drift in behavior, silently, and the DECISIONS.md claim "every write goes through the policy layer via the agent or CLI" becomes a lie. The whole point of the agent bridge is that the portal is never a bypass.

**Do not reach for React/Vue/Svelte to "make the progress page feel snappy".** HTMX + SSE handles the entire streaming-tool-call-log use case with less code, no bundler, and no npm dependency tree. If you're tempted because HTMX feels old-fashioned, read `htmx.org/essays/` for fifteen minutes — the pattern fits this exact use case.

**Do not build a file upload for KYC.** Real eKYC needs document capture + liveness + vendor integration — out-of-scope by DECISIONS 2026-04-11 (Phase 0). The pre-baked attestation badge is the honest representation of what BSS-CLI handles: the attestation itself. Adding a fake file upload just to "look real" mis-teaches what the channel/BSS boundary is.

**Do not hide the agent log widget behind a "Show debug info" button.** It's the headline feature. It goes on every page, always visible, top-right or right sidebar. The viewer is supposed to watch the LLM work.

**Do not make the SSE stream send JSON.** Send HTML partials. The browser swaps them into the DOM via HTMX with zero JavaScript. Sending JSON means the browser needs a template engine, which means JS, which means a bundler, which means you've broken rule #8.

**Do not forget the `X-BSS-Channel: portal-self-serve` header.** Without it, interactions logged against the customer look like CLI actions. The hero scenario asserts the channel value — if the header is missing, the scenario fails, which is the right failure.

**Do not skip the hero scenario three-in-a-row discipline.** The portal is the widest surface v0.x has ever introduced; LLM flakiness will show up here first. If one run passes and two don't, that's the model or the stack, not a reason to tag.
