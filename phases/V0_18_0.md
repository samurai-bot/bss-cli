# v0.18.0 — Automated Subscription Renewal Worker

> **The release that makes renewal happen on its own.** v0.17 was the first release where renewals could already be triggered (CLI, cockpit tool, scenario action) but nothing fired them. v0.18 ships the in-process tick loop that polls the database for due rows and dispatches `renew()` automatically. One feature, one schema column, one new module, one new env var. Everything that already happens on a manual renewal happens on an automatic one for free — the worker calls the existing `subscription_service.renew()` and never duplicates a single line of its logic.

## Goal

After v0.17 closed the three real-telco hygiene gaps (MNP, MSISDN replenishment, roaming), v0.18 closes the most obvious *non*-hygiene gap: the platform's central UX promise — "your bundle renews, you keep using your phone, you never think about it" — was load-bearing on a function nobody was calling. Manual `subscription.renew_now` was the operator escape hatch *only*; there was no production path.

v0.18 makes the production path automatic via an in-process tick loop attached to the subscription service's lifespan. Multi-replica safe by construction (FOR UPDATE SKIP LOCKED) even though we run single-replica today, because adding it later is a redesign and adding it now is one keyword.

## Schema delta — single migration `0020`

`packages/bss-models/alembic/versions/0020_v0180_renewal_attempted_at.py` (down_revision="0019"):

- `subscription.subscription` add column `last_renewal_attempted_at TIMESTAMPTZ` (nullable). Pre-v0.18 rows are valid without backfill.
- New partial index `ix_subscription_due_for_renewal` on `(state, next_renewal_at) WHERE state IN ('active','blocked')` backs both sweep queries; without it the worker scans the whole subscription table on every tick.

## What ships

### Renewal worker (the feature)

**New module** — `services/subscription/app/workers/renewal.py`:

- `_renewal_tick_loop(app, interval_seconds)` — `while True: try: sweep_due + sweep_skipped; except CancelledError: raise; except Exception: log; sleep(interval)`. Wall-clock `asyncio.sleep` for cadence (operator contract = `BSS_RENEWAL_TICK_SECONDS`).
- `_sweep_due(app)` — two transactions: (1) SELECT FOR UPDATE SKIP LOCKED + bulk UPDATE `last_renewal_attempted_at` + commit (releases row locks with the dedup column already visible), (2) per-id session: push worker auth, build `SubscriptionService`, call `svc.renew(id)`, swallow per-id failures.
- `_sweep_skipped(app)` — single txn: emit `subscription.renewal_skipped` for each blocked-overdue row + bulk UPDATE the same `last_renewal_attempted_at` column + commit.
- Worker actor: `system:renewal_worker`, channel: `system`, service_identity: `default`.

**Lifespan wiring** — `services/subscription/app/dependencies.py`:

```python
tick = int(os.environ.get("BSS_RENEWAL_TICK_SECONDS", "60"))
if tick > 0:
    app.state.renewal_task = asyncio.create_task(_renewal_tick_loop(app, tick))
```

Cancel + suppress `CancelledError` immediately after `yield`, before MQ + clients close (so an in-flight `renew()` doesn't hit a closed `payment_client`).

### Auth context push/pop helper

**Modified** — `services/subscription/app/auth_context.py`:

- New `push(*, actor, channel, tenant, service_identity) -> Token` — sets the ContextVar and returns the reset Token.
- New `pop(token)` — resets the ContextVar to its prior value.

`set_for_request()` (HTTP path) stays unchanged — it's safe to discard the Token there because Starlette runs each request in its own asyncio Task. The renewal worker is one long-lived Task, so values would leak across iterations without the explicit Token-pattern.

### Repository methods

**Modified** — `services/subscription/app/repositories/subscription_repo.py`:

- `due_for_renewal(*, now, limit, tenant_id)` — `WHERE state='active' AND next_renewal_at <= :now AND (last_renewal_attempted_at IS NULL OR last_renewal_attempted_at < next_renewal_at) FOR UPDATE SKIP LOCKED ORDER BY next_renewal_at LIMIT :limit`.
- `overdue_blocked(*, now, limit, tenant_id)` — same shape, `state='blocked'`. Reuses the same dedup column so a blocked sub doesn't re-emit on every tick.
- `mark_renewal_attempted(*, ids, at)` — bulk UPDATE for both sweeps inside the SELECT-txn.

### Admin tick endpoint (deterministic scenario driver)

**New module** — `services/subscription/app/api/renewal_admin.py`:

- `POST /admin-api/v1/renewal/tick-now` — invokes `_sweep_due` + `_sweep_skipped` once. Gated by `BSS_ALLOW_ADMIN_RESET` (the same flag that gates `reset-operational-data` and `clock/freeze`). Returns `{"status": "ok"}`.

**SubscriptionClient method** — `packages/bss-clients/bss_clients/subscription.py`:

- `tick_renewals_now()` — wraps the admin endpoint.

**Cockpit tool** — `orchestrator/bss_orchestrator/tools/subscription.py`:

- `subscription.tick_renewals_now` — registered in `operator_cockpit` profile only. Used by the v0.18 hero scenario after `clock.advance` to drive a single tick deterministically.

### Env var + config

- `BSS_RENEWAL_TICK_SECONDS=60` in `.env.example`. `0` disables the worker entirely (useful for tests and for deployments running multiple subscription replicas behind an external orchestrator).
- `Settings.renewal_tick_seconds: int = 60` in `services/subscription/app/config.py` (documents the surface; lifespan still reads via `os.environ` to keep a single `int(...)` site at startup).

## Doctrine amendments

Two new v0.18+ anti-pattern entries in `CLAUDE.md`:

- **Don't trigger renewal from anywhere except the subscription lifespan tick loop.** The renewal worker lives in `services/subscription/app/workers/renewal.py` and is started in `services/subscription/app/dependencies.py:lifespan`. No sibling scheduler container, no cron entry, no Celery beat, no external poller. The whole point of `FOR UPDATE SKIP LOCKED` is "in-process tick in every replica" multi-replica safe by construction; running the trigger elsewhere defeats the design and creates double-billing risk. Manual `subscription.renew_now` (CLI / cockpit / scenario action) stays as the operator escape hatch only — the automatic dispatch is the sole production path.
- **Don't duplicate `subscription_service.renew()` logic in the worker.** The worker calls `service.renew(sub_id)` and nothing else. No re-implementation of charge, no parallel state machine, no inline pivot of `pending_offering_id`. If renewal grows a new step (notification, additional audit emission, side effect), it goes into `renew()` once and the worker picks it up for free. The worker's job is *triggering*, not *renewing*.

## Known limits (intentional)

- **No per-renewal jitter.** A 60-second tick + bounded batch (100) is fine at thousand-customer scale; tens-of-thousands would benefit from spreading the dispatch across the tick interval to avoid charge spikes against the payment provider. Future v0.x can add `asyncio.sleep(jitter)` between dispatches.
- **No DLQ for repeatedly-failing renewals.** A renewal that fails (declined COF) sets `last_renewal_attempted_at < next_renewal_at` once per tick; `renew()` itself transitions the subscription to `blocked`, so the active-due query stops matching it on the next tick. Blocked-overdue is then tracked separately by `_sweep_skipped`. A future v0.x can add a `renewal_attempt_count` column + max-attempts policy if real customers see the issue.
- ~~**No upcoming-renewal notification.**~~ Shipped after the initial v0.18 cut. New `_sweep_upcoming_renewal_reminder` runs alongside the renewal sweep on every tick. Selects active subs whose `next_renewal_at` falls inside `BSS_RENEWAL_REMINDER_LOOKAHEAD_SECONDS` (default 86400 = 24h), sends one reminder email per period via the existing portal `EmailAdapter` (logging in dev, Resend in prod), deduped via a new `subscription.renewal_reminder_sent_at` column (migration `0021`). The email reuses the OTP visual vibe (dark phosphor-green theme, monospaced highlight block) but carries a non-secret highlight (the renewal amount) instead of an OTP. Set `BSS_RENEWAL_REMINDER_LOOKAHEAD_SECONDS=0` to disable the reminder sweep without changing the email provider; the renewal sweep keeps running.
- **Single-bucket per-tenant filter.** The sweep filters `tenant_id='DEFAULT'` (matches the rest of the v0.x service surface). Real multi-tenancy is a Phase 12+ concern.

## Rollout

1. Pull the v0.18 release.
2. `make migrate` — applies `0020`. Backwards compat is the column nullability; pre-v0.18 rows become immediately due once `next_renewal_at <= clock_now()`, regardless of the column being NULL — that's correct semantics for the first sweep after deploy.
3. Recreate the subscription container so it picks up the new code + worker:
   ```
   docker compose up -d --force-recreate subscription
   ```
4. Verify the worker started:
   ```
   docker compose logs subscription | grep "renewal.worker.started"
   ```
5. Run `make scenarios` against the live stack to confirm.

## Operator note

`BSS_ALLOW_ADMIN_RESET=true` on the subscription service enables `POST /admin-api/v1/renewal/tick-now`. Production deployments must keep this flag false. The worker runs naturally on its `BSS_RENEWAL_TICK_SECONDS` interval; the admin endpoint exists ONLY for scenario determinism. The same env flag already gates the destructive `reset-operational-data` and `clock/freeze` endpoints — re-using it means production sites already have the right posture without a new flag.
